01

The model

1.1 — CLIENT

Local SQLite, everywhere

Every client carries its own database. Reads are local, writes are local; the network is an optimization, not a requirement.

1.2 — SERVER

One authoritative log

All writes converge on a single server-side commit log. Ordering is total; history is replayable; truth is drawn to a single point.

1.3 — BOUNDARY

Scopes as code

Authorization is declared next to your schema. Each client syncs exactly the rows its scopes admit — nothing escapes validation.

Table 1 — the three-body arrangement. One written protocol governs all interactions.

02

Occlusion

Going offline is drifting out of contact — not an error state. Writes accumulate in a durable outbox and replay in order when the client re-enters range.

When histories diverge, syncular reports honest conflicts: surfaced to your code, never silently merged. Columns that should merge automatically can be declared as CRDTs; everything else stays deterministic.

  • ▸ durable offline outbox, ordered replay
  • ▸ conflicts surfaced, never invented
  • ▸ CRDT columns where merge is the right answer
