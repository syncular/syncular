# Golden vectors

Committed, CI-blocking test vectors for the v2 wire protocol. Every
implementation in this tree (TS web core, TS server, later the Rust native
core) must pass all of them; a wire-format change requires updated vectors
in the same commit as the spec change (SPEC.md §9).

Status: **generated.** The vector *set* is enumerated in SPEC.md
Appendix A; binaries are generated by the reference codec via
`packages/core/scripts/generate-vectors.ts` (`bun run generate:vectors`
in `packages/core`), never hand-hexed, and are deterministic — a
regeneration must be byte-identical.

## Layout

Message kinds live directly under this directory:

```
spec/vectors/
  request/           SSP2 request messages
  response/          SSP2 response messages
  segment/           standalone SSG2 rows segments
  realtime/          JSON control messages (no binary form)
    <case>.bin       canonical encoded bytes (binary kinds)
    <case>.json      canonical JSON debug rendering (SPEC.md §11 —
                     non-contractual for wire, contractual for vectors)
    manifest.json    per-kind: list of cases + expectations (incl. the
                     expected error code per invalid case)
    invalid/         negative cases; descriptive slug names (several
                     share an error code), error declared in manifest.
                     Binary kinds: .bin files that must fail decoding.
                     realtime: .json files that must fail control-message
                     parsing (malformed *known* events — unknown event
                     names are tolerated per SPEC.md §8.1 and are never
                     invalid cases)
```

## Requirements per case

1. **Decode:** `<case>.bin` must decode without error; the decoded value's
   canonical JSON rendering must deep-equal `<case>.json`.
2. **Round-trip:** re-encoding the decoded value must reproduce `<case>.bin`
   byte-for-byte (the encoding is canonical — one valid encoding per value).
   This includes unknown frame types: they are preserved, not dropped,
   by the skip rule (SPEC.md §1.2 rule 2).
3. **Negative cases:** files under `<kind>/invalid/` must fail to decode
   with the error named in `manifest.json`. For the realtime kind the
   invalid file is JSON text that must be rejected by the control-message
   parser with the named error (SPEC.md §8.1's known-event malformed-data
   rule).

## Coverage

The authoritative initial vector list (17 cases + invalid set) is
SPEC.md Appendix A. Summary of what it covers:

- requests: minimal pull, bootstrap pull (accept bits, resume token),
  multi-commit push (payload edge cases), combined push+pull
- responses: empty pull with cursor advance, incremental commits with
  every row-codec column type (null, bytes, non-BMP unicode), bootstrap
  segments (ref + signed URL + inline), push applied/conflict/cached,
  subscription revoked, cursor reset (horizon), mid-stream ERROR frame,
  unknown-frame skip (forward compat), schema floor
- segment: SSG2 with two row blocks + end marker, nullable columns
- realtime: `wake` and `hello` JSON control vectors
- invalid: truncated envelope (missing END), bad magic, unsupported
  wireVersion, non-zero flags, overlong frame length, bad bool byte,
  null bit on non-nullable column, rows segment without end marker,
  json column value that does not parse, wake with `requiresPull` not
  the literal `true`, fractional realtime numeric field
