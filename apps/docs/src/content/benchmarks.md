# Benchmarks

The measured performance record for the syncular core: bootstrap wall time, realtime
propagation, client bundle size, and server memory — with the caveats stated
up front, because the lane these numbers come from flatters some of them. The
curated source of truth is
[bench/RESULTS.md](https://github.com/syncular/syncular/blob/main/bench/RESULTS.md).

## The lane, honestly

All current numbers come from a **bun:sqlite loopback** lane: the TypeScript
client core and the real server library exchange in-process SSP2 bytes —
transport, segment download, and realtime are direct function calls. No
sockets, no HTTP stack, no browser. The 0.1.3 reference numbers came from an
external harness (Dockerized server, real HTTP, WASM client in a browser-like
lane), so they include network and serialization costs this lane does not —
and this lane's client runs on bun:sqlite, not sqlite-wasm. **Read every 0.1.3
comparison as order-of-magnitude, not apples-to-apples.**

Numbers below were generated by `bun run bench` on 2026-07-03 (Bun 1.3.14,
darwin/arm64), with deterministic seeded row data; timings vary run to run.

## Bootstrap wall time

Fresh client to fully-applied local rows, median over the recorded runs:

| Case | Measured (median) | Rows/sec | 0.1.3 reference |
|---|---|---|---|
| 1k rows (rows lane) | 4.7 ms | 213,552 | — |
| 100k rows (rows lane) | 362.6 ms | 275,774 | 204 ms (artifact lane) / 801 ms (TS client) / 227 ms (Rust client) |
| 100k rows (**sqlite image**) | 30.4 ms | 3,293,979 | 204 ms (artifact lane — the direct ancestor) |

The image lane's median is the warm case: the server builds the image once per
(scopes, pin) and every later client downloads and imports at file-copy speed.
The **cold** first bootstrap — which also builds and stores the image
server-side — took 288.5 ms. The rows lane uses no precomputed artifact at
all: segments are scanned, encoded, and hash-verified per bootstrap.

## Realtime propagation

RealtimeHub, 2 clients, 200 iterations — measured from `mutate()` on client A
to client B's post-apply ack (B applies the binary delta to its local
database, then acks):

| Metric | Measured | 0.1.3 reference |
|---|---|---|
| Propagation p50 | 0.1 ms | — |
| Propagation p95 | 0.2 ms | 22.8 ms p95 |
| Write-ack p50 (A's push+pull round) | 0.1 ms | 11.2 ms |
| Write-ack p95 | 0.2 ms | — |

Caveat from the results file: 0.2 ms p95 is loopback-fast and says nothing
comparable about 0.1.3's socketed 22.8 ms beyond "no regression is visible
in-process".

## Client bundle size

`bun build --minify --target=browser` of a minimal entry importing
`SyncClient` plus the sqlite-wasm database backend. The `sqlite3.wasm` binary
is an external asset fetched at runtime, measured as shipped by
`@sqlite.org/sqlite-wasm`:

| Artifact | Raw | Gzip | Whose bytes |
|---|---|---|---|
| syncular client code (core + codec) | 65.4 KB | 19.6 KB | ours |
| sqlite-wasm JS glue | 214.3 KB | 65.2 KB | vendor (SQLite) |
| sqlite3.wasm (external asset) | 844.5 KB | 396.1 KB | vendor (SQLite) |
| **Total** | **1124.1 KB** | **480.9 KB** | |

The size gate covers **our** bytes only: every wasm-SQLite product ships the
stock SQLite distribution, so vendor engine bytes don't gate. syncular's own
client JS (65.4 KB raw) is a ~3.3× reduction vs 0.1.3's 217.7 KB client JS; the
total including vendor bytes is still ~3.0× under 0.1.3's ~3.5 MB.

## Server memory

Peak RSS delta during the 100k-row bootstrap: **20.0 MB** (0.1.3 reference:
295–400 MB average RSS in the external harness). Caveat: server and client
share one process in this lane, so the delta includes the client's local
SQLite writes too — it is an **upper bound on the server share**. Sampled at
~2 ms.

## Reproduce it

```sh
git clone https://github.com/syncular/syncular
cd syncular
bun install
bun run bench
```

The driver lives in
[bench/](https://github.com/syncular/syncular/tree/main/bench); it prints the
same tables and regenerates the numbers on your machine. Expect different
absolute values — the recorded run is darwin/arm64.
