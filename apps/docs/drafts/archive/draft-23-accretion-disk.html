Orbital features

Built for convergence at every layer

From the outermost client edge to the innermost server core, every layer is designed to pull data toward consistency.

// OUTER ORBIT — CLIENT

outer

Offline-First Storage

Reads and writes hit the local database instantly. Changes queue in an outbox, spiraling toward the server when connectivity returns.

outer

Conflict Resolution

When two clients edit the same row offline, last-write-wins or custom merge functions resolve the collision automatically.

outer

End-to-End Encryption

Optional E2EE plugin encrypts data before it leaves the client. The server never sees plaintext -- only encrypted commits spiral inward.

// MID ORBIT — TRANSPORT

mid

HTTP + WebSocket

Pull over HTTP for reliable batch sync. WebSocket for realtime push notifications that wake sleeping clients instantly.

mid

Snapshot Chunks

New clients bootstrap from chunked snapshots instead of replaying the entire commit history. Fast cold starts, no matter how deep the log.

mid

Edge Relay

Deploy relay nodes at the edge to reduce latency. Commits pass through the relay on their spiral toward the origin server.

// INNER ORBIT — SERVER CORE

core

Commit-Log Ordering

The server maintains a strict commit log. Every mutation from every client is assigned a monotonic sequence number -- the gravitational order.

core

Scope Subscriptions

Clients subscribe to scoped subsets of data. Only relevant commits spiral down to each client, keeping bandwidth and storage minimal.

core

Prune & Compact

Old commits compact into snapshots. The log stays lean while the source of truth remains complete and consistent at the singularity.

Convergence

Data spirals inward.
Truth emerges at the center.

Each client generates mutations independently -- offline, concurrent, uncoordinated. When they sync, commits enter the accretion disk: ordered by the server's commit log, resolved through deterministic conflict handling, and compacted into the single authoritative state at the core.

3

Minimal surface area

A few lines to orbit

Define your schema, create a client, and start syncing. The commit log and convergence happen behind the scenes.

client.ts 
import { createClient } from '@syncular/client'
import { httpTransport } from '@syncular/transport-http'

const client = createClient({
  transport: httpTransport({
    baseUrl: 'https://api.example.com'
  }),
  tables: { todos, projects }
})
server.ts 
import { createServer } from '@syncular/server'
import { honoAdapter } from '@syncular/server-hono'

const server = createServer({
  db: database,
  shapes: { todos, projects }
})

export default honoAdapter(server)

Start converging

Install the client and server packages. Your data begins its spiral toward truth.
