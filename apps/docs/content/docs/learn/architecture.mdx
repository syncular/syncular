# How Sync Works

Syncular is built on a **commit-log architecture**. Every data change is recorded as an atomic commit with a monotonically increasing sequence number. Both server and client maintain ordered logs, enabling reliable synchronization even through intermittent connectivity, device restarts, and multi-device concurrency.

Data always flows through the server's commit log. Clients never talk to each other directly. The server is the single source of truth; clients hold local replicas filtered by [scopes](/learn/scopes).

## Sync Lifecycle Overview

Every client goes through the same lifecycle. The four phases happen in a
continuous loop managed by the Rust client runtime:

  chart={`

  Incremental --> Push["3. Push outbox"]
  Push --> Pull["4. Pull commits"]
  Pull --> Incremental
  Bootstrap --> Ready["Client is synced and enters the steady-state loop"]
`}
/>

## Phase 1: Bootstrap (First Sync)

When a client subscribes for the first time (or has fallen too far behind), it needs a full snapshot of the current data.

### How it works

1. Client sends a pull request with `cursor: -1` for the new subscription
2. Server calls `resolveScopes` to determine what data this actor can access
3. Server builds a point-in-time snapshot from the table handler
4. Server returns snapshot pages (chunked for large datasets) with an `asOfCommitSeq`
5. Client verifies and applies the snapshot through the Rust runtime to local SQLite
6. Client sets its cursor to the snapshot's `asOfCommitSeq`

<Mermaid
  chart={`
sequenceDiagram
  participant Client
  participant Server

  Client->>Server: POST /sync pull with cursor = -1
  Server->>Server: resolveScopes()
  Server->>Server: Build snapshot as of commit 42
  Server-->>Client: Return bootstrap chunks or artifacts
  Client->>Client: Verify and apply snapshot to SQLite
  Client->>Client: Set cursor = 42
`}
/>

For large datasets, snapshots are returned as **binary-table snapshot chunks**
or scoped snapshot artifacts. The client fetches each chunk separately via
`GET /sync/snapshot-chunks/:chunkId`, verifies integrity, decodes the typed
row payload, and applies it to SQLite.

For caching, streaming, and tuning knobs, see [Bootstrap Deep Dive](/learn/first-sync).

## Phase 2: Incremental Pull

Once bootstrapped, subsequent syncs only fetch new commits since the client's cursor. This is the steady-state mode.

<Mermaid
  chart={`
sequenceDiagram
  participant Client
  participant Server

  Client->>Server: POST /sync pull with cursor = 42
  Server->>Server: Fetch commits greater than 42 that match scopes
  Server-->>Client: Return commits 43..47 and nextCursor 47
  Client->>Client: Runtime verifies and applies changes
  Client->>Client: Advance cursor to 47
`}
/>

The server only returns commits whose scopes match the client's subscription. Scope filtering is covered in detail on the [Scopes](/learn/scopes) page.

## Phase 3: Push (Client to Server)

When the user makes a local change through a generated mutation, it is applied
optimistically and queued for push:

<Mermaid
  chart={`
sequenceDiagram
  actor User
  participant Client
  participant Server

  User->>Client: Call generated insert/update/delete mutation
  Client->>Client: Write through Rust runtime to local SQLite
  Client->>Client: Add operation to the outbox
  Client->>Server: POST /sync push with operations
  Server->>Server: Validate auth and apply in a transaction
  Server->>Server: Extract scopes and write the commit log
  Server-->>Client: status = applied, commitSeq = 48
  Client->>Client: Mark outbox commit acked
`}
/>

The push pipeline on the server:

1. **Validate** -- Check auth, verify the actor has write access
2. **Apply** -- Execute operations inside a database transaction
3. **Extract scopes** -- Automatically tag each change with scope values from row data
4. **Commit** -- Write to `sync_commits` and `sync_changes`, assign `commit_seq`
5. **Respond** -- Return the result with per-operation status

### Conflict detection

If another client changed the same row since this client last read it, the server detects a version mismatch and returns a conflict:

```typescript
// Push operation with base_version
{
  table: 'tasks',
  row_id: 'task-1',
  base_version: 5,
  payload: { title: 'Updated title' }
}

// Conflict response (server is at version 6)
{
  opIndex: 0,
  status: 'conflict',
  server_version: 6,
  server_row: { ... }
}
```

See [Conflict Resolution](/learn/conflict-resolution) for how to handle conflicts.

## Phase 4: Pull (Server to Client)

After pushing, the client pulls to receive changes from other clients. This uses the same incremental pull mechanism from Phase 2.

The full push-then-pull cycle looks like this:

<Mermaid
  chart={`
flowchart LR
  Push["Push outbox"] --> Pull["Pull commits"]
  Pull --> Apply["Runtime verifies and applies to SQLite"]
  Apply --> Wait["Wait for the next trigger (poll or wake-up)"]
  Wait --> Pull
`}
/>

## Realtime Deltas And Recovery Wake-ups

When WebSocket transport is enabled, realtime is an event-driven fast path over
the same server-authoritative sync model. A websocket frame may carry a bounded,
verified sync-pack delta that the client can apply immediately. If the frame is
too large, the client is behind, scopes changed, or recovery is required, the
server sends an explicit pull-required wake-up and the client uses the normal
HTTP pull path.

<Mermaid
  chart={`
sequenceDiagram
  participant Server
  participant Client

  Server-->>Client: WebSocket binary sync-pack delta
  Client-->>Client: Verify and apply delta
  Server-->>Client: Pull-required wake-up when recovery is needed
  Client->>Server: HTTP pull for checkpoint/recovery
  Server-->>Client: Return commits, snapshots, or artifacts
`}
/>

WebSocket is not a second sync model. It is the steady-state delta path when
the message can be verified and applied safely. HTTP pull remains the recovery
and checkpoint path for reconnects, overflow, auth refresh, schema mismatch,
large snapshots, artifact downloads, and any case where the client should not
trust an inline realtime frame.

## The Commit Log

The server commit log is the backbone of the sync protocol. It consists of two tables:

| `sync_commits` | `sync_changes` |
|---|---|
| `1 · device-A · user-1` | `1 · tasks · t-1 · write · { user: "1" }` |
| `2 · device-B · user-2` | `2 · tasks · t-2 · write · { user: "2" }` |
| `3 · device-A · user-1` | `3 · tasks · t-1 · write · { user: "1" }` |
| `4 · device-B · user-2` | `4 · tasks · t-2 · delete · { user: "2" }` |

Each commit:
- Has a **monotonically increasing `commit_seq`** assigned by the server
- Contains one or more changes across one or more tables
- Each change is tagged with **scopes** (extracted automatically from row data)
- Stores a cached push response for **idempotent retries**

Clients track their position with a **cursor** (the last `commit_seq` they have seen). On each pull, the server returns commits where `commit_seq > cursor` and scopes match.

Here is the complete flow when Client A makes a change that Client B receives:

  B->>B: Apply to SQLite and advance cursor when verified
`}

## Data Integrity Guarantees

Syncular provides three guarantees that make the sync protocol safe for production use:

**Idempotency** -- Every push is keyed by `(client_id, client_commit_id)`. Retrying the same push returns the cached result. No duplicates, even with unreliable networks.

**Atomicity** -- All operations in a commit are applied inside a single database transaction. Either every operation succeeds or none do. Partial application is impossible.

**Ordering** -- Commits have a strict global order via `commit_seq`. Clients always see changes in order. Cursors ensure exactly-once delivery with no gaps.
