# Glossary

Quick reference for the terms used throughout Syncular's documentation and API.

---

## Sync Flow

### Bootstrap

The initial full snapshot sent when a client first subscribes to a [Table](#table) (cursor is `-1`) or when its cursor is too old. The server sends all current rows for the subscription's effective [Scopes](#scope), delivered as compressed [Snapshot Chunks](#snapshot-chunk). After bootstrap completes, the client switches to [Incremental Sync](#incremental-sync).

```typescript
{ cursor: -1 }  // Triggers bootstrap
```

See [Commits -- Pull Lifecycle](/learn/commits#pull-lifecycle-server-to-client).

### Commit

Atomic unit of change in the sync protocol. Every commit has a global `commitSeq` (assigned by the server), an `actorId`, a timestamp, and an array of [Changes](#change). Commits are the entries in Syncular's append-only log.

```typescript
{ commitSeq: 42, actorId: 'user-1', changes: [{ table: 'tasks', ... }] }
```

See [Commits](/learn/commits).

### Change

A single row-level modification inside a [Commit](#commit). Each change records
the table, row id, operation, committed row data when present, row version, and
[Scopes](#scope). App code should normally see this as `insert`, `update`, or
`delete` through runtime row-delta events; server storage may use lower-level
row-write records internally.

```typescript
{ table: 'tasks', rowId: 'task-1', operation: 'update', scopes: { user_id: '123' } }
```

### Cursor

A client's position in the commit log, represented by a `commitSeq` number. On each pull, the client sends its cursor and receives only commits after that position. A cursor of `-1` means "start from scratch" (triggers [Bootstrap](#bootstrap)).

```typescript
{ cursor: 100 }  // Give me everything after commit_seq 100
```

### Incremental Sync

After [Bootstrap](#bootstrap), the client receives only [Commits](#commit) since its last [Cursor](#cursor) position. Much more efficient than re-fetching everything.

### Snapshot Chunk

A compressed (`binary-table-v1` + gzip) bundle of one or more snapshot pages sent during [Bootstrap](#bootstrap). Chunks are cached on the server with a TTL and can be stored in external storage (S3, etc.) for large datasets.

## Data Model

### Scope

Authorization boundary stored as key-value pairs on each [Change](#change). Scopes define data partitions -- clients only receive changes for scopes they are authorized to access. See [Scopes](/learn/scopes).

```typescript
{ user_id: '123' }                 // User's private data
{ project_id: '456' }              // Project data
{ user_id: '123', org_id: 'A' }   // User within organization
```

### Table

A server-defined sync unit. Clients request [Subscriptions](#subscription) by `table` and [Scopes](#scope). Each table maps to a server-side [Handler](#handler) that controls access, snapshots, and mutations.

```typescript
// Client subscribes to the "tasks" table
{ table: 'tasks', scopes: { user_id: '123' } }
```

See [Subscriptions](/learn/subscriptions).

### Subscription

A client's active request to receive data for a specific [Table](#table) and [Scopes](#scope). Each subscription has an `id`, a `cursor`, and resolved scope values. The server tracks subscriptions per-pull and returns per-subscription responses.

```typescript
{
  id: 'sub-tasks',
  table: 'tasks',
  scopes: { user_id: '123' },
  cursor: 42,
}
```

See [Subscriptions](/learn/subscriptions).

### SyncOperation

Client-to-server operation in a push. Generated app mutations create these
records from `insert`, `update`, and `delete` calls. Each operation contains
the table, row id, payload when present, and optional base version for
[Conflict](#conflict) detection. One or more operations make up a [Commit](#commit).

```typescript
{ table: 'tasks', rowId: 'task-1', payload: { title: 'New' }, baseVersion: 5 }
```

## Client Concepts

### Outbox

Client-side queue of pending commits waiting to be pushed to the server. Mutations write to the local database and the outbox immediately. When the device is online, the outbox drains in order. See [Commits -- Push Lifecycle](/learn/commits#push-lifecycle-client-to-server).

### Conflict

Occurs when a client's [SyncOperation](#syncoperation) has a `base_version` that does not match the server's current `server_version`. The server rejects the commit and returns the conflict details. Conflicts are stored in the client's `sync_conflicts` table for resolution. See [Conflict Resolution](/learn/conflict-resolution).

### Fingerprinting

An optimization used by `@syncular/client` and `@syncular/react` to avoid expensive deep-equality checks on query results. A fingerprint is a string combining row count, row IDs, and mutation timestamps. React hooks compare fingerprints to decide whether to re-render.

```typescript
// Fingerprint format: "count:id1@ts1,id2@ts2,..."
"3:abc@1706123456789,def@1706123456790,ghi@0"
```

### Runtime extension

An explicit capability layered into the Rust runtime or generated app contract,
such as field encryption, CRDT/Yjs fields, blob handling, command history, or
app-layer editor adapters.

```typescript
import { defineSyncularClient, encryptedField, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  tables: {
    notes: syncedTable({
      table: 'notes',
      scopes: [scope('user_id', { source: 'actorId', required: true })],
      serverVersion: 'server_version',
      encryptedFields: [encryptedField('body', { scope: 'notes' })],
    }),
  },
});
```

Prefer generated contract features or documented adapter packages over
low-level lifecycle hooks.

## Architecture

### Actor ID

Authenticated user identifier. Used for authorization checks and tracking who made each [Commit](#commit). Passed to [Handlers](#handler) and stored on every commit.

### Client ID

Unique identifier for each device or browser instance. Paired with `clientCommitId` to form the idempotency key that prevents duplicate commit processing.

### Handler

Server-side (`ServerTableHandler`) implementation that defines how a table
syncs. Client-side table behavior is generated from the app contract and
executed by the Rust runtime, not hand-authored as a separate handler layer.

**Server handler** responsibilities: resolve scopes, build snapshots, apply operations, extract scopes from rows.

```typescript
// Server handler key methods
{
  table: 'tasks',
  resolveScopes: async (ctx) => ({ user_id: [ctx.actorId] }),
  extractScopes: (row) => ({ user_id: row.user_id }),
  snapshot: async (ctx) => ({ rows: [...], nextCursor: null }),
  applyOperation: async (ctx, op, opIndex) => ({ result: ..., emittedChanges: [...] }),
}
```

See [Table Handlers](/server/table-handlers).

### Transport

The network layer that moves data between client and server. The current
client runtime uses HTTP for push/pull and snapshot/artifact downloads, with
WebSocket realtime as a verified delta or recovery wake-up path. Most apps
configure `baseUrl` on the generated client instead of constructing a transport
directly.

### Presence

Realtime awareness of other connected users (cursor positions, online status, custom state). Delivered over WebSocket. Not part of the commit log -- presence is ephemeral.

See [Realtime And Presence](/clients/javascript/realtime-presence).

### resolveScopes

Function on server [Handlers](#handler) that determines what [Scopes](#scope) the authenticated user can access. Called during pull to compute effective scopes for each [Subscription](#subscription).

```typescript
async resolveScopes(ctx) {
  return { user_id: [ctx.actorId] };
}
```
