# JavaScript Testing

Use `@syncular/testkit` instead of mocking Syncular internals in TypeScript
tests. It gives JavaScript apps real local SQLite behavior, real sync
semantics, deterministic fixtures, HTTP/websocket helpers, and app-owned domain
assertions.

Use `@syncular/testkit` to test TypeScript/server/browser behavior across:

- fast in-process tests
- real HTTP integration tests
- runtime-oriented tests (node, deno, cloudflare)

The TypeScript API is test-runner agnostic. You can use Bun, Vitest, Jest, or
other runners without changing testkit code. For Rust/native generated clients,
use [Rust Testkit](/docs/rust/testing).

If you are still evaluating whether Syncular is trustworthy enough to adopt, start with [Testing & Confidence](/docs/start/testing-and-confidence).

If you are still wiring the system itself, use [JavaScript](/docs/javascript),
[Rust](/docs/rust), [Server](/docs/server), and [Features](/docs/features) for
implementation guides first.

## API groups

These groups refer to the TypeScript `@syncular/testkit` package:

| Group | Main exports |
|---|---|
| Fixtures | `createSyncFixture`, `createTestServer`, `createTestClient`, `createEngineTestClient` |
| HTTP fixtures | `createHttpServerFixture`, `createHttpClientFixture`, `createNodeHonoServer` |
| Protocol + parsing | `createSyncUpsertOperation`, `createSyncSubscription`, `createSyncCombinedRequest`, `parseSyncCombinedResponse` |
| Lifecycle resources | `withSyncFixture`, `createSyncFixtureResource`, `withTestServer`, `openRealtimeWsResource` |
| Scenario + deterministic | `createScenarioFlow`, `runPushPullCycle`, `createIdFactory`, `createFakeClock` |
| Runtime realtime | `createRealtimeWsUrl`, `openRealtimeWs`, `waitForWsJsonMessage`, `closeWsSafe` |
| Assertions | `assertOutboxEmpty`, `assertConflictCount`, `assertRowExists`, `waitForOutboxEmpty` |
| Faults | `withFaults`, `withRecording`, `createMockTransport` |

Need a rollout checklist rather than APIs? See [Testing Strategy](/docs/testing/strategy).
