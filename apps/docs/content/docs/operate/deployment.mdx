# Deployment

The Syncular server is a set of Hono routes, so it deploys anywhere Hono runs:
Node.js, Bun, Cloudflare Workers, Deno, or a container on any host. This page
covers the deployment shapes and the production settings that matter.

## Use this page when

- your local or staging server is ready to become a real service
- you need a concrete shape for Docker, Cloudflare, Fly.io, or a VPS
- you are building a go-live checklist around health checks, CORS, limits, and
  backups
- you need repeatable restore, blob-storage, rate-limit, or credential
  rotation drills

## Prerequisites

- A working server ([Server Getting Started](/server/getting-started))
- PostgreSQL for production, or SQLite (WAL mode) for single-node deployments

## The server entry point

Every target below serves the same app. A minimal production entry looks like
this (see [Server Getting Started](/server/getting-started) for the full
walkthrough of handlers and auth):

```typescript
import { Hono } from 'hono';
import { createSyncServer } from '@syncular/server/hono';

const { syncRoutes, consoleRoutes } = createSyncServer({
  db,
  dialect,
  sync: { handlers, authenticate },
  routes: {
    cors: process.env.CORS_ORIGINS?.split(',') ?? [],
  },
  upgradeWebSocket,
  console: { token: process.env.SYNC_CONSOLE_TOKEN },
});

const app = new Hono()
  .route('/api/sync', syncRoutes)
  .route('/api/console', consoleRoutes!);

export default app;
```

Clients then point `config.baseUrl` at the sync mount itself
(`https://your-host/api/sync`).

### Environment variables

Syncular reads only one environment variable itself (`SYNC_CONSOLE_TOKEN`, as
the fallback for `console.token`). Everything else is your app's wiring — a
typical set:

```bash
DATABASE_URL=postgres://user:pass@host:5432/mydb   # your db setup reads this
AUTH_SECRET=...                                    # your authenticate() reads this
SYNC_CONSOLE_TOKEN=a-strong-random-token           # console bearer token
CORS_ORIGINS=https://app.example.com               # passed to routes.cors
PORT=3000
```

## Health checks

Sync routes ship a built-in health endpoint at `GET <mount>/health` (for the
example above: `GET /api/sync/health`):

```json
{ "status": "healthy", "timestamp": "2026-06-12T00:00:00.000Z" }
```

It confirms the process is up and routing works, but does not touch the
database. If your platform's health check should also verify DB connectivity,
add a small route of your own:

```typescript
app.get('/health', async (c) => {
  try {
    await db.selectFrom('sync_commits').select('commit_seq').limit(1).execute();
    return c.json({ status: 'ok' });
  } catch {
    return c.json({ status: 'error', message: 'database unavailable' }, 503);
  }
});
```

## Docker

### Bun

```dockerfile
FROM oven/bun:1
WORKDIR /app

COPY package.json bun.lock ./
RUN bun install --frozen-lockfile --production

COPY . .

EXPOSE 3000
CMD ["bun", "src/index.ts"]
```

### Node.js

```dockerfile
FROM node:22-slim
WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci --omit=dev

COPY . .
RUN npm run build

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

### Docker Compose with PostgreSQL

```yaml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://syncular:syncular@db:5432/syncular
      SYNC_CONSOLE_TOKEN: ${SYNC_CONSOLE_TOKEN}
      CORS_ORIGINS: ${CORS_ORIGINS:-http://localhost:5173}
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://localhost:3000/api/sync/health || exit 1"]
      interval: 10s
      timeout: 3s
      retries: 5

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: syncular
      POSTGRES_PASSWORD: syncular
      POSTGRES_DB: syncular
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U syncular"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  pgdata:
```

Run `ensureSyncSchema(db, dialect)` plus your app migrations as a release or
operator step before serving traffic. Keep normal request startup focused on
opening already-prepared infrastructure, especially on edge runtimes.

## Cloudflare Workers

Cloudflare deployment uses the dedicated adapter package
`@syncular/server/cloudflare` (Workers + Durable Objects + R2 helpers) instead
of hand-rolling the Hono wiring. The full setup — including the
D1-vs-Neon/Hyperdrive database choice and WebSocket realtime via Durable
Objects — is on the [Cloudflare](/server/cloudflare) page.

The short version of the platform constraints:

- **Database:** `@syncular/server/d1` + the SQLite server dialect, or
  Neon/Postgres via `@syncular/server/neon` + `createNeonServerDialect()`
  (Hyperdrive in front of regular Postgres also works).
- **WebSockets:** Workers need Durable Objects for realtime connections; the
  adapter provides this. Clients recover via HTTP pull either way.
- **Blobs:** no filesystem — use R2 for blob and external snapshot-chunk
  storage.
- **CPU limits:** large bootstraps are fine because snapshots stream as
  chunked downloads, but keep `maxPullLimitSnapshotRows` reasonable so a
  single pull stays within Worker CPU budgets.

## Fly.io

A standard Dockerfile deploy:

```toml
# fly.toml
app = "my-syncular-app"
primary_region = "iad"

[build]
  dockerfile = "Dockerfile"

[env]
  PORT = "3000"

[http_service]
  internal_port = 3000
  force_https = true
  auto_stop_machines = false   # keep WebSocket connections alive

  [[http_service.checks]]
    interval = "10s"
    timeout = "2s"
    method = "GET"
    path = "/api/sync/health"
```

```bash
fly launch
fly secrets set DATABASE_URL=postgres://... SYNC_CONSOLE_TOKEN=...
fly deploy
```

Fly's proxy supports WebSockets natively. If you run **more than one
machine**, configure a realtime broadcaster so wake-up events reach clients
connected to other instances (see
[Multi-instance](#multi-instance-deployments) below). Avoid
`auto_stop_machines` for realtime-heavy apps — stopping the machine drops
every WebSocket.

## Self-hosted / VPS

A plain Linux box works well for single-tenant deployments:

1. Run the app under a process manager (`systemd`, or a container via
   Docker/Podman).
2. Put a reverse proxy (Caddy, nginx) in front for TLS. Make sure WebSocket
   upgrade headers are forwarded for the realtime route
   (`/api/sync/realtime`):

```nginx
location /api/ {
  proxy_pass http://127.0.0.1:3000;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
  proxy_read_timeout 120s;   # > websocket heartbeat interval (default 30s)
}
```

3. Point your platform/uptime monitoring at `GET /api/sync/health`.

SQLite is a legitimate production choice here (one node, WAL mode, file on
local disk); Postgres becomes necessary once you need multiple app instances
or managed backups.

## CORS

Sync routes own their CORS handling — configure it through the route config
rather than generic Hono middleware so the protocol headers
(`x-syncular-schema-version`, snapshot scope headers, `traceparent`, etc.) are
allowed automatically:

```typescript
const { syncRoutes } = createSyncServer({
  // ...
  routes: {
    cors: ['https://app.example.com', 'http://localhost:5173'],
  },
});
```

The same origins are used as the default WebSocket origin allowlist. To
diverge, set `routes.websocket.allowedOrigins` explicitly — browser CORS does
not protect WebSocket upgrades, so this guard matters in production.

## Rate limiting

Sync routes rate-limit by default: 120 pulls and 60 pushes per minute per
client. Treat these as starting points, not product policy. Tune or disable via
`routes.rateLimit`:

```typescript
routes: {
  rateLimit: {
    pull: { maxRequests: 120, windowMs: 60_000, includeHeaders: true },
    push: { maxRequests: 60, windowMs: 60_000, includeHeaders: true },
  },
  // or, if a gateway in front already rate-limits:
  // rateLimit: false,
}
```

Use this tuning loop before launch and after any sync cadence change:

1. Run the staging app with realistic tab counts, offline bursts, and blob
   uploads.
2. Record normal pull/push rates per `clientId` and actor, plus websocket
   reconnect/message rates.
3. Keep enough headroom for reconnect storms after deploys and restores.
4. Alert on `sync.rate_limited`, `sync.websocket_connection_limit`, and
   repeated `retryLater` recommended actions.
5. If an edge gateway owns the public rate limit, keep Syncular's inner limit
   enabled unless you have equivalent per-client visibility at the gateway.

HTTP pull/push limits and websocket message limits are separate knobs. Raising
one does not raise the other.

## WebSocket connection limits

```typescript
routes: {
  websocket: {
    heartbeatIntervalMs: 30_000,   // default 30s
    maxConnectionsTotal: 5000,     // per process, default 5000
    maxConnectionsPerClient: 3,    // per clientId, default 3
  },
}
```

## Pruning and compaction

Enable automatic maintenance in production so the commit log does not grow
without bound. Auto-maintenance piggybacks on sync traffic and is gated by
`minIntervalMs`:

```typescript
routes: {
  prune: {
    minIntervalMs: 5 * 60 * 1000,           // default: check every 5 minutes
    options: {
      activeWindowMs: 14 * 24 * 60 * 60 * 1000,   // default 14 days
      fallbackMaxAgeMs: 30 * 24 * 60 * 60 * 1000, // default 30 days
      keepNewestCommits: 1000,                     // default 1000
    },
  },
  compact: {
    minIntervalMs: 30 * 60 * 1000,          // default: every 30 minutes
    options: { fullHistoryHours: 48 },      // required — compact is disabled without options
  },
}
```

What these mean and how to size them for your offline window is covered in
[Performance & Scaling](/operate/performance). The Console can also run both
on demand with previews ([Console Operations](/operate/console/operations)).

## Database backups

The sync state lives entirely in the database: the commit log
(`sync_commits`, `sync_changes`), client cursors, snapshot caches, and your
app tables — all created by `ensureSyncSchema` plus your migrations.

- **Postgres:** regular `pg_dump`/WAL archiving covers everything. Restore is
  safe: clients whose cursor is ahead of a restored (older) commit log will
  re-bootstrap rather than diverge.
- **SQLite:** snapshot the database file with `sqlite3 ... ".backup"` or run
  Litestream for continuous replication. Back up the WAL alongside the main
  file.
- Snapshot chunk/artifact caches are derived data — losing them costs a
  regeneration, not correctness. If you use external chunk storage (R2/S3) it
  does not need point-in-time backup.
- After restoring to an older state, expect a wave of client re-bootstraps;
  consider raising `keepNewestCommits` temporarily.

Run a restore drill before launch and on a recurring cadence:

1. Restore the latest production backup into an isolated staging database.
2. Run `ensureSyncSchema(db, dialect)` and your app migrations against the
   restored database; they should be idempotent.
3. Run `getSyncularServerSchemaReadiness(...)` and
   `npx syncular schema check --json` against the same version set you plan to
   deploy.
4. Start the server on the restored database, call `GET <mount>/health`, and
   perform a pull as a seeded test actor.
5. Compare row counts for `sync_commits`, `sync_changes`,
   `sync_blob_uploads` if blobs are enabled, and your largest app tables.
6. Record the restore time, expected client re-bootstrap load, and the
   rollback decision you would take if the drill were production.

## Blob storage consistency

The database is authoritative for blob metadata. External storage such as S3,
R2, or the filesystem is authoritative for bytes. If you use the database blob
adapter, the `sync_blobs` body table is covered by database backup and restore;
if you use external storage, run a separate consistency check.

A practical scheduled check:

1. Sample recent and old `sync_blob_uploads` rows where `status = 'complete'`
   for each active partition.
2. For each sample, call the storage adapter's `exists(...)` or
   `getMetadata(...)` path, or mint a signed URL through the normal blob route
   with a service actor that should have access.
3. Sample app rows in every configured `blobColumns` field and assert the
   referenced hash has a complete `sync_blob_uploads` row in the same
   partition.
4. Alert separately for `blob_record_missing`, `storage_object_missing`, and
   `signed_url_failed`; they point at metadata loss, byte loss, and credential
   or signing failures respectively.
5. Never delete an object by content hash until no configured scoped metadata
   row can reference it and your product retention policy says the body is
   disposable.

When you enable `createBlobCleanupScheduler`, start with
`deleteFromStorage: false` in staging and review the deleted candidates before
allowing storage deletion in production.

## Credential rotation

Syncular does not own your app auth secrets; your `authenticate` function does.
Plan rotations by credential class:

- **App auth signing keys/session secrets:** deploy code that accepts both old
  and new keys, issue new sessions, wait for old sessions to expire, then
  remove the old key. Expect websocket clients to reconnect after refresh.
- **Console token (`SYNC_CONSOLE_TOKEN` or `console.token`):** set a new token,
  restart or redeploy the console route, have operators reconnect, and use
  `Disconnect` with credential clearing on shared workstations.
- **Console API keys:** use staged rotation from the Console Config view:
  create the replacement key, update automation, then revoke the old key.
- **Storage credentials:** rotate S3/R2/filesystem signer credentials with a
  canary upload and download. Watch for `blob.signing_failed` and
  `storage_object_missing` during and after the rotation.

## Operations readiness gate

Keep a small production evidence file next to your deployment config and fail
deploys when the runbook facts are missing or stale:

```bash
npx syncular ops check --config ./ops/production.syncular.ops.json --json
```

The check validates the facts from this page: schema readiness, latest restore
drill, blob consistency policy, credential rotation ownership/cadence,
rate-limit review status, Console log/event retention, and offline
support-window sizing. It does not replace live probes such as
`GET <mount>/health`; it makes the operator runbook evidence machine-readable
so CI can block an unsafe deploy before traffic moves.

See [Ops Check Command](/reference/cli/ops-check) for the JSON shape and
freshness flags.

## Multi-instance deployments

Syncular servers are stateless between requests except for WebSocket
connections, so horizontal scaling needs no sticky sessions. The one thing to
add is a realtime broadcaster, so a push handled by instance A wakes clients
connected to instance B:

```typescript
routes: {
  realtime: {
    broadcaster: myRedisBroadcaster, // or NATS, or any pub/sub fanout
  },
}
```

Without a broadcaster, cross-instance clients still converge — they just wait
for their next HTTP pull instead of waking immediately.

## Production checklist

- **Database**
  - [ ] PostgreSQL (or single-node SQLite with WAL mode)
  - [ ] Connection pooling (PgBouncer / Hyperdrive) sized for your instance count
  - [ ] `ensureSyncSchema` + app migrations run on deploy, before traffic
  - [ ] `syncular schema check --json` or `getSyncularServerSchemaReadiness(...)`
        runs in CI/deploy readiness
  - [ ] `syncular ops check --json` validates the production evidence file
  - [ ] Upgrade and rollback order reviewed in the
        [Upgrade Guide](/reference/upgrade-guide)
  - [ ] Backups configured and restore tested
  - [ ] Last restore drill recorded with expected client re-bootstrap load
  - [ ] Offline support window reviewed; `routes.prune.activeWindowMs` covers it
  - [ ] Compaction `fullHistoryHours` reviewed for support/debug expectations
- **Authentication**
  - [ ] `authenticate` validates real tokens/sessions; secrets from env, not source
  - [ ] Auth, console, API-key, and storage credential rotation owners/cadence set
- **Networking**
  - [ ] HTTPS enforced (proxy or platform TLS)
  - [ ] `routes.cors` restricted to your app origins
  - [ ] WebSocket origin allowlist covers exactly your app origins
  - [ ] Rate limits tuned from staging traffic and left on (or consciously moved to a gateway)
- **Maintenance**
  - [ ] `routes.prune` enabled with an `activeWindowMs` ≥ your users' realistic offline window
  - [ ] `routes.compact` enabled with `options.fullHistoryHours` set
  - [ ] Console request-event and operation-event retention windows reviewed
  - [ ] Request-payload snapshot policy recorded as disabled or redacted/bounded
  - [ ] Blob consistency sampling configured when using external blob storage
  - [ ] Health endpoint wired into platform checks and alerting
- **Console** (if enabled)
  - [ ] `SYNC_CONSOLE_TOKEN` is strong and rotated like any operator credential
  - [ ] Console CORS restricted ([Operations Setup](/operate/operations-setup))
- **Multi-instance** (if applicable)
  - [ ] Realtime broadcaster configured
