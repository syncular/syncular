# Performance & Scaling

Syncular handles datasets from small personal apps to catalogs with hundreds of
thousands of rows. This page is the tuning story: where time goes, which knobs
exist, and how to prove a configuration under load before launch.

## Use this page when

- bootstrap size or pull latency is starting to matter
- you are sizing pruning and compaction before launch
- you want to load-test a deployment with realistic sync traffic

## Bootstrap at scale

A first sync (or a re-bootstrap after pruning) delivers a **snapshot** of all
subscribed data. Three mechanisms keep that fast; all three are documented in
depth on [Bootstrap & Snapshots](/server/snapshot-pull) — this section covers
the tuning angle.

### Snapshot chunks and caching

Large snapshots are encoded as `binary-table-v1` chunks (gzip-compressed),
cached server-side, and downloaded separately via
`GET <mount>/snapshot-chunks/:chunkId`. Concurrent first-syncs of the same
scope hit the cache instead of re-querying your tables — a bootstrap storm
costs one snapshot query per scope, not one per client.

- Chunk bodies live in `sync_snapshot_chunks` by default. For very large
  deployments, move bodies to external storage (S3/R2) with
  `createDbMetadataChunkStorage`:

```typescript
import { createDbMetadataChunkStorage } from '@syncular/server';

const { syncRoutes } = createSyncServer({
  db,
  dialect,
  sync: { handlers, authenticate },
  chunkStorage: createDbMetadataChunkStorage({ db, blobAdapter: r2Adapter }),
});
```

- The built-in client transport authenticates chunk downloads with the
  subscription's scope values (`X-Syncular-Snapshot-Scopes`); chunk IDs are
  not bearer tokens.

### Page and row limits

The server enforces upper bounds; clients request within them:

```typescript
// Server caps (createSyncServer `routes` / createSyncRoutes `sync`)
routes: {
  maxPullLimitSnapshotRows: 50_000, // default 50000 rows per snapshot page
  maxPullMaxSnapshotPages: 50,      // default 50 pages per subscription per pull
}
```

```typescript
// Client requests (config.pull — see Client Configuration for all options)
config: {
  pull: {
    limitSnapshotRows: 5_000, // rows per snapshot page
    maxSnapshotPages: 10,     // pages per pull cycle
    limitCommits: 100,        // commits per incremental pull
  },
}
```

Client requests above the server caps are clamped, and paginated bootstraps
continue automatically across pull cycles, so these are throughput knobs, not
correctness knobs. Bigger pages mean fewer round-trips but larger memory spikes
on both ends. The full client-side surface (including bootstrap phases and
push batching) is on
[Client Configuration](/clients/javascript/client-configuration).

### Binary columns and snapshot artifacts

Two server-side levers matter once tables get wide or bootstraps get huge:

- **`snapshotBinaryColumns`** — generated server modules wire this
  automatically; hand-rolled handlers should pass it so the server can skip
  per-page column inference when encoding chunks.
- **Scoped snapshot artifacts** — prebuilt SQLite snapshot files per scope,
  downloaded via `GET <mount>/snapshot-artifacts/:artifactId` and attached
  directly by the client instead of row-by-row application. Configure
  `snapshotArtifactStorage` on `createSyncServer` for external bodies.

Both are documented with config details on
[Bootstrap & Snapshots](/server/snapshot-pull).

## Pruning

The commit log (`sync_commits` + `sync_changes`) grows indefinitely without
maintenance. Pruning deletes commits that all active clients have consumed.

### How it works

1. Each client's pull cursor is tracked in `sync_client_cursors`.
2. A client is *active* if it pulled within `activeWindowMs`.
3. The watermark is the minimum cursor across active clients; commits at or
   below it are deleted.
4. `fallbackMaxAgeMs` prunes past stuck cursors; `keepNewestCommits` always
   preserves the newest N commits.

Clients that fall behind pruned history are forced to re-bootstrap — so
`activeWindowMs` must be at least your users' realistic offline window.

### Configuration

```typescript
routes: {
  prune: {
    minIntervalMs: 5 * 60 * 1000, // gate between auto-prune runs (default 5 min)
    options: {
      activeWindowMs: 14 * 24 * 60 * 60 * 1000,   // default 14 days
      fallbackMaxAgeMs: 30 * 24 * 60 * 60 * 1000, // default 30 days
      keepNewestCommits: 1000,                     // default 1000
    },
  },
}
```

Auto-maintenance piggybacks on sync traffic (no separate scheduler needed);
`minIntervalMs` gates how often it actually runs.

| Scenario | Recommendation |
|----------|---------------|
| Clients connect daily | `activeWindowMs: 3 days`, `keepNewestCommits: 500` |
| Clients may be offline for weeks | `activeWindowMs: 30 days`, `keepNewestCommits: 5000` |
| High write volume | shorter `fallbackMaxAgeMs`, watch `sync.server.pull.bootstrap_subscriptions` for forced re-bootstraps |
| Storage is cheap, re-bootstraps are expensive | longer `activeWindowMs`, higher `keepNewestCommits` |

### Manual pruning

```typescript
import { computePruneWatermarkCommitSeq, pruneSync } from '@syncular/server';

const watermark = await computePruneWatermarkCommitSeq(db, {
  activeWindowMs: 14 * 24 * 60 * 60 * 1000,
});

const deleted = await pruneSync(db, {
  watermarkCommitSeq: watermark,
  keepNewestCommits: 1000,
});
```

The Console exposes the same operation with a dry-run preview
([Console Operations](/operate/console/operations)).

## Compaction

Compaction is complementary: instead of deleting whole commits, it removes
**intermediate history**, keeping only the latest change per (table, row,
scope) for data older than `fullHistoryHours`. Clients can still pull
incrementally through compacted ranges — no re-bootstrap.

```typescript
routes: {
  compact: {
    minIntervalMs: 30 * 60 * 1000,        // default 30 min
    options: { fullHistoryHours: 48 },    // required; compact is off without options
  },
}
```

For most applications, enable both: pruning bounds total size, compaction
keeps the retained window dense. Manual entry points are `compactChanges` /
`maybeCompactChanges` from `@syncular/server`.

## Indexing

### Server (PostgreSQL)

The sync infrastructure tables ship with the indexes they need. For your app
tables, index:

1. **Scope columns** used by subscriptions and `extractScopes`:

```sql
CREATE INDEX idx_notes_user_id ON notes (user_id);
```

2. **Snapshot query columns** — whatever your handler's `snapshot` filters and
   orders by:

```sql
CREATE INDEX idx_notes_user_id_id ON notes (user_id, id);
```

### Client (SQLite)

Add indexes in your client migrations so the generator carries them into every
generated client:

```sql
create index idx_notes_user_created
  on notes (user_id, created_at);
```

Keep client indexes minimal — each one costs write time on every sync-applied
change.

## Query and subscription hygiene

- **Subscribe narrowly.** Scoped subscriptions
  (`scopes: { project_id: ... }`) bound both bootstrap size and ongoing pull
  fan-out. Model the default scopes in `syncular.app.ts` so generated
  subscription helpers do this for you.
- **The server caps subscriptions per pull** (default 200,
  `maxSubscriptionsPerPull`) — apps that generate one subscription per entity
  hit this; prefer one subscription per scope.
- **Operations per push** are capped too (default 200,
  `maxOperationsPerPush`). For bulk imports, split into multiple commits.
- **In React, `useSyncQuery`** uses fingerprint diffing + structural sharing
  to avoid re-renders; keep an `id` column in single-table queries (or set
  `keyField`) for the fast row-timestamp path.

## Connection management

```typescript
routes: {
  websocket: {
    heartbeatIntervalMs: 30_000,  // default 30s
    maxConnectionsTotal: 5000,    // per process (default 5000)
    maxConnectionsPerClient: 3,   // per clientId (default 3)
  },
  realtime: {
    broadcaster: myRedisBroadcaster, // required for multi-instance wake-ups
  },
}
```

Realtime is the canonical wake-up path; HTTP pull remains the recovery path,
so realtime capacity problems degrade latency, not correctness. Interval
polling is opt-in on the client (`lifecycle.pollIntervalMs`) for apps that
want a background cadence besides realtime.

## Load testing

The repo ships a k6 load suite under `tests/load` that speaks the real
protocol — combined `POST /sync` requests with **binary SSP1
(`binary-sync-pack-v1`) responses**, realtime WebSocket wake-ups, and snapshot
chunk downloads. It includes a dependency-free SSP1 reader for k6, so the
scenarios track row-level convergence (push-to-visibility lag), not just HTTP
status codes.

```bash
brew install k6        # or apt/winget

# 1) start the standalone load server (sqlite or pglite, seeded)
bun run test:load:server

# 2) run a scenario from the repo root
bun run test:load:push-pull
```

Scenarios and what they answer:

| Scenario | Question it answers |
|---|---|
| `test:load:push-pull` | push/pull throughput and push-to-visibility lag (`sync_lag_ms`) under steady write load |
| `test:load:bootstrap` | bootstrap throughput incl. chunk downloads (rows/second SLO) |
| `test:load:bootstrap-storm` | many first-time clients bootstrapping concurrently — chunk-cache effectiveness |
| `test:load:websocket` | realtime wake-up latency and post-wake convergence |
| `test:load:reconnect-storm` / `:reconnect-soak` | reconnect churn: WS reconnect + pull catch-up under concurrent writes |
| `test:load:maintenance-churn` | does prune/compact running mid-traffic hurt latency? |
| `test:load:mixed` / `:mixed-soak` | 80/20 reader/writer workload with realtime listeners |
| `test:load:nightly` | the whole macro suite with JSON artifacts + trend stats |

Server profiles: `test:load:server:sqlite:large` seeds ~1M rows across 1200
users for catalog-shaped tests; `SEED_ROWS`, `SEED_USERS`, and
`LOAD_DB_DIALECT=pglite` tune it further. Thresholds (p95 latencies, error
rates, sync-lag SLO) are encoded in the scripts — see `tests/load/README.md`
for every variable.

To test **your** deployment rather than the bundled server, the scenarios are
plain k6 scripts: point `BASE_URL` at your staging stack and provide
credentials your `authenticate` accepts.

## Monitoring what you tuned

After tuning, watch the signals that tell you whether it worked —
`sync.server.pull.duration_ms`, `sync.server.pull.bootstrap_subscriptions`
(forced re-bootstraps), `sync.conflicts.detected`, outbox sizes in the Console
Fleet view. The full list lives in [Observability](/operate/observability).
