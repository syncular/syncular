# Observability

This page covers what is actually observable in a Syncular deployment today:
which events and metrics the framework emits, how distributed tracing flows
through sync requests, how client-side diagnostics reach the Console, and how
to route all of it into Sentry or your own backend.

## Use this page when

- you are moving from local development to a real deployment
- you need sync logs, traces, or metrics in an existing observability stack
- you want to debug sync incidents without guessing from user reports

## The three observability surfaces

1. **Server telemetry** — structured logs, spans, and metrics emitted through
   `@syncular/core`'s pluggable telemetry hooks. Default backend: console
   JSON. Wire Sentry or anything else.
2. **Console request events** — when console routes are enabled, every push
   and pull is recorded as a request event (actor, client, status, duration,
   affected scopes) queryable via the Console UI and API.
3. **Client diagnostics** — the client runtime emits structured diagnostic
   events and can publish periodic snapshots (connection state, outbox stats,
   recent sync timings) to the Console's `client-diagnostics` endpoint.

## What to log

### Server side

Out of the box the server emits structured telemetry events (`sync.server.push`,
`sync.server.pull`, maintenance events like `sync.prune_auto` /
`sync.compact_auto` and their `_failed` variants) through the configured
telemetry backend — console JSON unless you replace it. In production, make
sure these land in your log aggregation; the `_failed` maintenance events in
particular are easy to miss and mean your commit log is growing.

If console routes are enabled, request events give you a per-request audit
trail without any extra wiring: `GET <console>/events` filters by
`eventType`, `actorId`, `clientId`, and `outcome`, and each event carries
status code, duration, operation/row counts, and affected scopes. See
[Operations Setup](/operate/operations-setup) for the endpoints and retention
settings.

### Client side

Pass a `diagnostics` sink when creating the client to receive every diagnostic
event (level, source, code, message, plus sync attempt/trace IDs and
table/row context where relevant):

```ts
const database = await createSyncularAppDatabase({
  config: { /* ... */ },
  diagnostics: (event) => {
    // event: { at, level, source, code, message, syncAttemptId?, traceId?, ... }
    myLogger.log(event.level, `[syncular] ${event.code}: ${event.message}`, event);
  },
});
```

For operator visibility without app plumbing, enable Console diagnostics
publishing — the client debounces and ships compact snapshots (connection
state, subscriptions, recent diagnostics and sync timings, outbox/conflict/
blob-queue stats) that show up per client in the Console's
[Fleet view](/operate/console/fleet):

```ts
const database = await createSyncularAppDatabase({
  config: { /* ... */ },
  consoleDiagnostics: {
    baseUrl: 'https://api.example.com/api/console',
    token: consoleToken, // or getHeaders
  },
  // or `consoleDiagnostics: true` when the console is mounted next to baseUrl
});
```

The same snapshot is available programmatically for support tooling or bug
reports:

```ts
const snapshot = await database.client.diagnosticSnapshot();
// snapshot.connection, snapshot.subscriptions, snapshot.recentDiagnostics,
// snapshot.recentSyncTimings, snapshot.outboxStats, snapshot.conflictStats, ...
```

## Metrics worth watching

These are the metrics Syncular emits today (verified against the server
implementation), plus where to find the values that are not emitted as
metrics:

| Signal | Where it comes from | Why it matters |
| --- | --- | --- |
| `sync.server.pull.duration_ms` (distribution, `status` attr) | server pull | pull latency — your realtime feel |
| `sync.server.push.duration_ms` (distribution, `status` attr) | server push | push latency |
| `sync.server.push.requests` / `sync.server.pull.requests` (counts) | server | traffic + error rate via the `status` attribute |
| `sync.conflicts.detected` (count) | server push | conflict rate — a climbing rate means stale `base_version`s or hot rows |
| `sync.server.pull.bootstrap_subscriptions` (distribution) | server pull | how often clients re-bootstrap; spikes mean pruning outran your offline window |
| `sync.server.pull.snapshot_pages`, `sync.server.pull.commits`, `sync.server.pull.changes` (distributions) | server pull | bootstrap and pull payload sizes |
| `sync.server.push.operations`, `sync.server.push.emitted_changes` (count/distribution) | server push | write volume and fan-out |
| `sync.sessions.started` / `sync.sessions.ended`, `sync.transport.reconnects` (counts) | realtime routes | WebSocket churn — reconnect storms show up here |
| Outbox size | client `diagnosticSnapshot().outboxStats` and Console Fleet | growing pending counts mean pushes are failing or blocked on a conflict |
| Bootstrap duration | Console request events + `sync.server.pull.duration_ms` on bootstrap pulls; client `recentSyncTimings` | first-load experience at scale |
| Commit backlog / client lag | Console `GET /stats` (`maxCommitSeq` vs. client cursors) and the Fleet view | clients falling behind, prune watermark health |

The Console also computes operational aggregates server-side — request/error
time series (`GET /stats/timeseries`) and push/pull latency percentiles
(`GET /stats/latency`) — useful for dashboards without any metrics backend.

There are currently **no client-emitted metrics** through the telemetry hooks;
client-side health is observed via diagnostics (above), not metrics. If you
need client metrics in your backend, forward them from a `diagnostics` sink.

## Tracing

Sync routes participate in distributed tracing:

- Incoming `traceparent` (W3C) and `sentry-trace` headers are parsed and
  attached to the request's spans and console request events; both headers are
  in the default CORS allowlist, so browser clients can send them.
- The client runtime stamps each sync attempt with a `syncAttemptId` and a
  generated `traceparent` (`SyncularSyncAttempt`), which also appears on
  client diagnostic events — letting you correlate one sync attempt across
  client log, server span, and console request event.
- Server work runs inside spans named `sync.server.push` (op `sync.push`) and
  `sync.server.pull` (op `sync.pull`) with status, duration, and payload-size
  attributes.

## The telemetry contract

`@syncular/core` owns the hooks; all framework telemetry routes through them:

```ts
import {
  configureSyncTelemetry, // install a backend (call once at startup)
  logSyncEvent,
  startSyncSpan,
  countSyncMetric,
  gaugeSyncMetric,
  distributionSyncMetric,
  captureSyncException,
} from '@syncular/core';
```

A backend implements the `SyncTelemetry` interface:

```ts
import type { SyncTelemetry } from '@syncular/core';

interface SyncTelemetry {
  log(event: SyncTelemetryEvent): void;        // structured log events
  tracer: SyncTracer;                          // startSpan(options, callback)
  metrics: SyncMetrics;                        // count / gauge / distribution
  captureException(error: unknown, context?: Record<string, unknown>): void;
}
```

You can emit your own app telemetry through the same helpers so it lands in
whatever backend is configured:

```ts
logSyncEvent({ event: 'checkout.completed', level: 'info', orderId: 'ord_456' });
countSyncMetric('checkout.orders', 1, { attributes: { plan: 'pro' } });
captureSyncException(new Error('Payment failed'), { orderId: 'ord_456' });
```

## Sentry backend

`@syncular/observability-sentry` adapts the hooks to Sentry.

```bash
bun add @syncular/observability-sentry
```

### Browser app

```ts
import { initAndConfigureBrowserSentry } from '@syncular/observability-sentry';

initAndConfigureBrowserSentry({
  dsn: import.meta.env.VITE_SENTRY_DSN,
  environment: 'production',
  release: appVersion,
  enableLogs: true,
  tracesSampleRate: 0.2,
});
```

### Cloudflare Worker

```ts
import {
  configureCloudflareSentryTelemetry,
  withCloudflareSentry,
} from '@syncular/observability-sentry';

configureCloudflareSentryTelemetry();

export default withCloudflareSentry(
  (env) => ({
    dsn: env.SENTRY_DSN,
    environment: env.SENTRY_ENVIRONMENT,
    release: env.SENTRY_RELEASE,
    enableLogs: true,
    tracesSampleRate: 0.2,
  }),
  {
    async fetch(request, env, ctx) {
      return app.fetch(request, env, ctx);
    },
  }
);
```

Sentry-specific log/message helpers (`logBrowserSentryMessage`,
`captureBrowserSentryMessage`, and the `Cloudflare` equivalents) are available
for app events that should go straight to Sentry rather than through the
generic hooks. Use one consistent `release` value across client and server
builds so traces and sourcemaps line up.

## Custom backend (OpenTelemetry, Datadog, …)

Implement `SyncTelemetry` and install it before creating any client or server:

```ts
import { configureSyncTelemetry, createDefaultSyncTelemetry } from '@syncular/core';

// Partial implementations are fine — start from the default and override:
const telemetry = createDefaultSyncTelemetry();
telemetry.log = (event) => myLogger.log(event.level ?? 'info', event.event, event);
telemetry.metrics = {
  count: (name, value, opts) => myMetrics.increment(name, value ?? 1, opts?.attributes),
  gauge: (name, value, opts) => myMetrics.gauge(name, value, opts?.attributes),
  distribution: (name, value, opts) => myMetrics.histogram(name, value, opts?.attributes),
};
configureSyncTelemetry(telemetry);
```

Implementation notes:

- Attribute values are always primitives (`string | number | boolean`).
- `tracer.startSpan` must invoke its callback synchronously and pass through
  the callback's return value (which may be a `Promise`).
- For OpenTelemetry, bridge `startSpan` to `tracer.startActiveSpan`,
  `distribution` to histograms, and `log` to span events or your log bridge.
  Syncular does not manage the OTel SDK lifecycle — initialize it first.
- Use `resetSyncTelemetry()` in tests to restore the console backend.

## If you don't see data

1. Confirm `configureSyncTelemetry(...)` (or the Sentry init helper) runs
   before any Syncular client/server is created.
2. For Sentry: DSN present at runtime, `enableLogs: true`, tracing enabled
   (`tracesSampleRate`), and check project/environment filters.
3. For console request events: console routes must be enabled
   (`createSyncServer({ console: { token } })` or an explicit
   `createConsoleRoutes` wired via `consoleLiveEmitter`) — without them, no
   request events are recorded.
4. For client diagnostics in Fleet: check the `consoleDiagnostics` endpoint
   URL and token; the publisher drops payloads silently rather than breaking
   sync.
