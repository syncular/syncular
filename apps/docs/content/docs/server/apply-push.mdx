# Push & Conflicts

Push sends local client operations to the server. The server validates the
request, applies each operation through the table handler, records accepted
changes in the commit log, and returns per-operation results.

## Push flow

1. The client queues generated mutations locally.
2. The runtime pushes one or more commits — each with a unique
   `clientCommitId` (the idempotency key), the client `schemaVersion`, and
   `operations[]`.
3. The server authenticates the request.
4. The handler authorizes and applies each operation inside a transaction.
5. Applied operations emit committed changes with row versions and scopes.
6. The server stores the commit and wakes affected realtime subscribers.

## Request and response shape

Pushes travel in the combined `POST /` sync body as `push.commits[]`. Each
commit gets its own response entry:

```json
{
  "ok": true,
  "push": {
    "commits": [
      {
        "clientCommitId": "commit-0001",
        "ok": true,
        "status": "applied",
        "commitSeq": 42,
        "results": [{ "opIndex": 0, "status": "applied" }]
      }
    ]
  }
}
```

Commit-level `status`:

- `applied` — all operations committed; `commitSeq` is the new commit log
  position.
- `cached` — this `clientCommitId` was already applied earlier; the stored
  results are replayed and nothing is written twice. This is what makes
  client retries safe.
- `rejected` — at least one operation failed; the commit's writes are rolled
  back as a unit, and the rejection outcome itself is recorded so a retry of
  the same `clientCommitId` replays the same results.

Operation-level results are `applied`, `conflict`, or `error` — the shapes and
error codes are documented on
[Table Handlers](/server/table-handlers#applyoperation-results-and-error-shapes).

## Conflict detection

Upsert operations may carry a `base_version` — the `server_version` the client
last saw for that row. The default handler turns it into an optimistic
concurrency check:

- `base_version` matches the current row → the write applies and the version
  increments.
- `base_version: 0` and the row does not exist → insert.
- Mismatch → the operation returns status `conflict` with the current
  `server_version` and the winning `server_row`, and nothing is written:

```json
{
  "opIndex": 0,
  "status": "conflict",
  "code": "sync.version_conflict",
  "message": "Version conflict: server=7, base=5",
  "server_version": 7,
  "server_row": { "id": "task-1", "title": "Renamed elsewhere", "server_version": 7 }
}
```

Operations without `base_version` are last-write-wins upserts. How clients
resolve returned conflicts (merge, retry, accept server) is covered in
[Conflict Resolution](/learn/conflict-resolution).

## Apply a custom operation

The default `applyOperation` covers same-shape tables. Implement your own when
the server stores a different shape or writes through a repository layer:

```typescript
async applyOperation(ctx, op, opIndex) {
  if (op.op === 'delete') {
    const deleted = await repo.deleteTask({
      rowId: op.row_id,
      actorId: ctx.actorId,
    });

    return {
      result: { opIndex, status: 'applied' },
      emittedChanges: [
        {
          table: 'tasks',
          row_id: deleted.rowId,
          op: 'delete',
          row_json: null,
          row_version: null,
          scopes: { user_id: ctx.actorId },
        },
      ],
    };
  }

  const stored = await repo.saveTask({
    rowId: op.row_id,
    actorId: ctx.actorId,
    payload: op.payload,
    schemaVersion: ctx.schemaVersion,
  });

  const row = documentToTaskRow(stored);
  return {
    result: { opIndex, status: 'applied' },
    emittedChanges: [
      {
        table: 'tasks',
        row_id: row.id,
        // Internal emitted-change storage uses the row-write operation marker.
        // Generated app mutations still expose insert/update/delete.
        op: 'upsert',
        row_json: row,
        row_version: row.server_version,
        scopes: { user_id: row.user_id },
      },
    ],
  };
}
```

`ctx` gives you the transaction (`ctx.trx`), `ctx.actorId`, the full
`ctx.auth` object, `ctx.clientId`, `ctx.commitId`, and `ctx.schemaVersion`.

## Emitting changes

`emittedChanges` is how an applied write becomes visible to other clients: the
entries are stored in the commit log, matched against subscriptions by their
`scopes`, returned by later pulls, and used to wake realtime subscribers.

Rules that keep sync consistent:

- The emitted `row_json` must be the generated client row shape for that
  table — not necessarily the raw server row.
- `row_version` must reflect the committed version so client conflict checks
  keep working.
- `scopes` must contain the row's stored scope values (e.g.
  `{ user_id: 'U1' }`); a change with wrong scopes is invisible to the
  clients that need it — or visible to ones that should not see it.
- Only emit for `applied` results. Never emit for conflicts or errors.
- For writes that happen outside the push path entirely (cron jobs, admin
  scripts), use [External Changes](/server/external-changes).

## Version-aware pushes

Use `ctx.schemaVersion` when older clients can still push valid operations.
Generated server helpers can validate whether the version is supported and can
provide older mutation payload types. If a version is outside your support
window, set `routes.requiredSchemaVersion` so the server rejects the commit
with an update-required response instead of attempting a partial write.

## Limits

`routes.maxOperationsPerPush` (default 200) caps operations per commit;
`routes.maxSyncRequestJsonBytes` (default 4 MiB) caps the request body. Large
payloads belong in [Blobs](/server/blobs), not in row operations.
