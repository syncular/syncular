# Blobs

Blob sync keeps large binary bodies out of row commits. Rows store metadata and
blob references; the server owns authenticated upload, download, and state
tracking for the binary payloads.

## Server responsibilities

The server should own:

- blob metadata tables and lifecycle state;
- authenticated upload and download routes;
- storage adapter wiring for local disk, S3, R2, or another object store;
- row-level authorization checks through table handlers and scopes;
- sync wakeups when blob state changes affect subscribed rows.

## Route shape

Blob routes are mounted next to the sync routes. They should call the same
authentication function as push and pull.

```typescript
import {
  createBlobManager,
  createScopedBlobAccessDecisionChecker,
} from '@syncular/server';
import { createBlobRoutes } from '@syncular/server/hono';

const blobManager = createBlobManager({
  db,
  adapter: blobStorageAdapter,
});

const canAccessBlob = createScopedBlobAccessDecisionChecker({
  db,
  handlers: [photosHandler],
  references: [{ table: 'photos', blobColumns: ['original', 'thumbnail'] }],
});

const blobRoutes = createBlobRoutes({
  db,
  blobManager,
  authenticate: async (c) => authenticate(c.req.raw),
  maxUploadSize: 25 * 1024 * 1024,
  canAccessBlob,
});

app.route('/api/sync', syncRoutes);
app.route('/api/sync', blobRoutes);
```

Use platform storage in production. For Cloudflare, wire blob bodies to R2 and
keep Syncular's metadata in the server database.

## Access diagnostics

`canAccessBlob` may return a boolean, but returning a scoped access decision is
better for app tests and operator logs. `createScopedBlobAccessDecisionChecker`
keeps the route fail-closed and lets `GET /blobs/:hash/url` explain which
authority boundary failed:

- `blob.forbidden` with `details.accessReason: "missing_reference"` means the
  hash exists or was requested, but no configured scoped row currently grants
  access in this partition.
- `blob.forbidden` with `details.accessReason: "scope_denied"` means a row
  references the hash, but the actor's resolved scopes do not include that
  row. The details include `referenceTable` and `referenceColumn`, not tokens
  or signed URLs.
- `blob.not_found` with `details.failureKind: "blob_record_missing"` means the
  completed upload metadata is not present in the requested partition.
- `blob.signing_failed` with `details.failureKind: "signed_url_failed"` means
  the storage adapter could not create a download URL.
- `blob.not_found` with `details.failureKind: "storage_object_missing"` means
  the direct download token was valid, but the underlying bytes were missing
  from storage.

For global/base data, use scoped metadata rows over shared bytes by default.
Store the body once by content hash, then create a package/file/base-asset row
in each campaign/project partition that should fetch it. If the reference table
stores rows from multiple route partitions, configure
`partitionColumn: "partition_id"` so a global row cannot authorize a
campaign/project request. Fetch through a global partition only when the client
is intentionally subscribed to that global partition; use shared partitions
only as an explicit auth policy, not as an implicit hash fallback.

## Client interaction

Client bindings expose blob operations as part of the generated client/runtime
surface. Apps should store blob references in normal synced rows, then let the
runtime upload, download, retry, and surface progress events.

Keep UI-specific concepts such as image resizing, preview generation, and file
pickers in the app layer.
