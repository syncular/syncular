# Authorization

Syncular clients request subscriptions, but the server decides what those
requests mean for the current actor. Authentication produces an actor context;
`resolveScopes` turns that context into the scope values the actor can access.

<Callout title="Concept lives in Learn">
This page covers the server implementation: `authenticate()`, `resolveScopes`,
and write-path authorization. For what scopes *are*, the intersection model,
revocation behavior, and the standard patterns (user-owned, project
membership, team-shared), read the canonical [Scopes](/learn/scopes) page
first.
</Callout>

## Authenticate requests

`createSyncServer` expects a `sync.authenticate` function. It receives the raw
`Request`; return `null` to reject (the client gets a 401), or return an auth
object with at least `actorId`.

```typescript
import type { SyncServerAuth } from '@syncular/server';

interface AppAuth extends SyncServerAuth {
  workspaceIds: string[];
  role: 'member' | 'admin';
}

export async function authenticate(request: Request): Promise<AppAuth | null> {
  const token = request.headers.get('authorization')?.replace('Bearer ', '');
  if (!token) return null;

  const session = await verifyToken(token);
  if (!session) return null;

  return {
    actorId: session.userId,
    workspaceIds: session.workspaceIds,
    role: session.role,
  };
}
```

That object is available as `ctx.auth` in table handlers — load everything
your handlers need for authorization decisions here (roles, membership ids)
so `resolveScopes` stays cheap.

## Resolve allowed scopes

`resolveScopes` must return the scope values the actor is allowed to access at
request time. The effective subscription is the intersection of
client-requested scopes and these allowed scopes — the
[intersection model](/learn/scopes#effective-scopes-the-intersection-model)
is explained on the canonical page. Query your membership tables here; do not
derive access from anything the client sent.

```typescript
const tasksHandler = createServerHandler<ServerDb, ClientDb, 'tasks', AppAuth>({
  table: 'tasks',
  scopes: ['user:{user_id}', 'project:{project_id}'],

  resolveScopes: async (ctx) => ({
    user_id: [ctx.actorId],
    project_id: ctx.auth.workspaceIds,
  }),
});
```

Two server-side details beyond the conceptual model:

- **Wildcard:** returning `'*'` (or including it in an array) for a scope key
  allows any value the client requests for that key. Reserve it for genuinely
  global access such as admin tooling.
- **Caching:** `resolveScopes` runs per request per table. Results are
  memoized within a request; pass a `scopeCache` backend to `createSyncServer`
  to share resolved scopes across requests when membership queries get
  expensive.

## Validation is fail-loud

Both sides of the intersection are validated against the scope keys declared
in the handler's `scopes` patterns:

- A **requested** scope key the handler does not declare rejects the request
  with a descriptive `InvalidSubscriptionScopeError` naming the subscription,
  table, offending key, and the expected keys.
- A **resolved** scope key that the handler does not declare throws the same
  way — a typo in `resolveScopes` surfaces as a server error instead of
  silently never matching.
- Malformed scope **values** from `resolveScopes` (anything other than
  strings, string arrays, or `null`/`undefined` for "no access") throw with
  the table, key, and offending entry type.

## When subscriptions become revoked

A pull response marks a subscription `status: 'revoked'` instead of `active`
when any of these hold:

- `resolveScopes` threw (the error is logged server-side; the client is not
  told why).
- The intersection of requested and allowed scopes is empty.
- Any single requested scope key ends up with no allowed values — partial
  access does not degrade into a narrower subscription; it revokes, to avoid
  partially-scoped leaks.

The client runtime reacts by removing matching local rows and stopping that
subscription. If users "lose their data" unexpectedly, check
[Troubleshooting](/server/troubleshooting#subscription-unexpectedly-revoked)
first.

## Keep authorization server-owned

Do not trust client-provided scope values, row fields, or mutation payloads as
authorization. Use them as requests. The handler should still check current
server state and membership before returning rows or applying writes.

On the write path the default handler already enforces scope columns
automatically ([Table Handlers](/server/table-handlers#authorization-hooks-and-write-path-checks)).
For checks beyond scope columns, use the `authorize` hook or reject inside a
custom `applyOperation`:

```typescript
async applyOperation(ctx, op, opIndex) {
  const projectId = readProjectIdFromPayload(op.payload);

  if (!ctx.auth.workspaceIds.includes(projectId)) {
    return {
      result: {
        opIndex,
        status: 'error',
        error: 'Actor cannot write this project.',
        code: 'sync.forbidden',
        retriable: false,
      },
      emittedChanges: [],
    };
  }

  // Apply the write and emit the committed change.
}
```

Auth leases can help a native client queue offline work with recorded auth
provenance, but the server still evaluates the current actor and current
membership before accepting the push.

## Scope shape guidelines

- Use stable IDs, not display names.
- Return arrays for multi-membership scopes.
- Keep scopes coarse enough for efficient pull planning, but specific enough to
  avoid over-syncing.

Key choice and cardinality guidance lives on the canonical
[Scopes](/learn/scopes#scope-guidelines) page.
