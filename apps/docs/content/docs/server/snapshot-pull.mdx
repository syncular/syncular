# Bootstrap & Snapshots

Pull is how a client catches up. The server resolves the actor's effective
subscriptions, returns newer commits when possible, and serves a snapshot when
a client needs a table bootstrap (first sync, or after falling behind pruned
history).

## Pull flow

1. The client sends its `schemaVersion`, commit limit, and subscriptions —
   each with a `cursor` (last seen commit sequence) and requested scopes.
2. The server authenticates the request.
3. Each table handler resolves allowed scopes; the server intersects requested
   and allowed scopes ([Authorization](/server/scopes-and-auth)).
4. Per subscription, the server returns either committed changes after the
   cursor, or — when the subscription has no usable cursor — a bootstrap
   snapshot plus the cursor to continue from.
5. Large snapshots are delivered as binary snapshot chunks downloaded
   separately.

Realtime does not change this flow. WebSocket messages only wake the client so
it pulls again.

## Default snapshots

For same-shape tables, `createServerHandler` builds a keyset-paginated
snapshot from the table, primary key, version column, and scope columns:

```typescript
const tasksHandler = createServerHandler({
  table: 'tasks',
  scopes: ['user:{user_id}'],
  resolveScopes: async (ctx) => ({ user_id: [ctx.actorId] }),
});
```

Use the default when the server row is already the generated client row.
Handler `codecs` are applied to outbound rows automatically.

## Custom snapshots

Use a custom `snapshot` when the server stores a different shape, needs joins,
or must project the row to the requesting client's schema version.

```typescript
const tasksHandler = createSyncularAppServerHandler<ServerDb, AppAuth>({
  table: syncularGeneratedApp.tables.tasks,
  resolveScopes,

  async snapshot(ctx) {
    const documents = await ctx.db
      .selectFrom('documents')
      .select([
        'id',
        'content',
        'done',
        'owner_id',
        'workspace_id',
        'revision',
      ])
      .where('owner_id', 'in', ctx.scopeValues.user_id)
      .orderBy('id')
      .limit(ctx.limit)
      .execute();

    return {
      rows: documents.map((row) =>
        syncularGeneratedApp.projectClientRowForVersion(
          'tasks',
          documentToTaskRow(row),
          ctx.schemaVersion
        )
      ),
      nextCursor:
        documents.length === ctx.limit
          ? documents[documents.length - 1]!.id
          : null,
    };
  },
});
```

Honor `ctx.limit` and `ctx.cursor` for large tables. Return `nextCursor` until
the snapshot is complete. The client persists `bootstrapState` between pulls,
so a multi-page bootstrap survives interruption and resumes at the right table
and row cursor.

## Chunking and caching

Bootstrap snapshot pages are not inlined into the pull response. The server
encodes them as **binary table chunks** (`binary-table-v1`, gzip-compressed),
stores them, and returns chunk references; the client downloads bodies from
`GET /snapshot-chunks/:chunkId`. This keeps pull responses small and lets
chunk downloads stream and retry independently.

Chunks are cached and reused:

- The cache key covers partition, table, effective scope values, the
  `asOfCommitSeq` the bootstrap is pinned to, and the page cursor/limits — two
  clients of the same user bootstrapping at the same commit reuse the same
  chunks.
- Lifetime defaults to 24 hours per handler; tune with `snapshotChunkTtlMs`.
- Chunk bodies live in the database (`sync_snapshot_chunks`) by default. Pass
  `chunkStorage` to `createSyncServer` to keep bodies in external storage
  (S3, R2, …) instead — recommended once snapshots get large.
- `routes.snapshotChunkGzipLevel` (default 1) trades server CPU against chunk
  size; `routes.maxSnapshotChunkResponseBytes` caps a single chunk response
  (default 64 MiB).

## Binary columns: `snapshotBinaryColumns`

The binary encoding needs stable column metadata (names and types). Where it
comes from matters for performance:

- **Typegen-generated handlers supply it automatically** — the generated
  server module wires `snapshotBinaryColumns`, `snapshotBinaryEncoder`, and
  their version-aware variants for every table.
- **Hand-rolled handlers should pass `snapshotBinaryColumns`.** Without it,
  the server infers column types by scanning every row of every page before
  encoding — correct, but it costs encode time on large bootstraps and types
  ride on whatever the first rows contain. Supplying the metadata skips
  inference and pins types.

```typescript
const tasksHandler = createServerHandler({
  table: 'tasks',
  scopes: ['user:{user_id}'],
  resolveScopes: async (ctx) => ({ user_id: [ctx.actorId] }),
  snapshotBinaryColumns: [
    { name: 'id', type: 'string' },
    { name: 'title', type: 'string' },
    { name: 'completed', type: 'integer' },
    { name: 'user_id', type: 'string' },
    { name: 'created_at', type: 'integer' },
    { name: 'server_version', type: 'integer' },
  ],
});
```

Column types are `'string' | 'integer' | 'float' | 'boolean' | 'json' |
'bytes'`; add `nullable: true` for optional columns.

If you support multiple client schema versions with different column sets,
implement `snapshotBinaryColumnsForVersion(schemaVersion)` (and the matching
encoder variant) instead; returning `null` from it intentionally disables the
fallback to the current metadata. If the server cannot produce a supported
client schema version, fail clearly so the app can show an update-required
screen.

## Scoped snapshot artifacts

For very large bootstraps, clients can opt into **snapshot artifacts**: the
server materializes a whole subscription page set as a prebuilt SQLite
database (`sqlite-snapshot-v1`) the client downloads from
`GET /snapshot-artifacts/:artifactId` and attaches directly. Configure
`snapshotArtifactStorage` on `createSyncServer` to enable external body
storage; `routes.maxSnapshotArtifactResponseBytes` caps the response (default
256 MiB). Artifacts and chunked snapshots are mutually exclusive per
response — the client requests what it supports.
