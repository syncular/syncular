# Quick Start

> **Alpha.** APIs, wire formats, and storage layouts will change between releases. Pin versions and expect manual migrations between versions.

## Run the canonical client

The current client product path is the Rust-owned package published as
`@syncular/client`. Clone the repo and run the browser binding gates:

```bash
git clone https://github.com/syncular/syncular.git
cd syncular
bun install
bun run rust:browser:test
bun run rust:browser:tsgo
```

## Build your own app (canonical path)

Syncular packages are published on npm under the `@syncular` scope. The recommended implementation path is:

1. [Server Setup](/docs/build/server-setup)
2. [Rust Client](/docs/rust-client)
3. [Browser Client](/docs/rust-client/browser)
4. Learn the concepts: [Scopes](/docs/concepts/scopes) and [Subscriptions](/docs/concepts/subscriptions)
5. Then add features as needed: blobs, encryption, migrations, observability, console, relay

If you’re not sure where to start, see [Adoption Paths](/docs/introduction/adoption-paths).

## What happens on a write (high level)

Here's the full lifecycle of adding a task:

<Mermaid
  chart={`
sequenceDiagram
  autonumber
  actor User
  participant Client as Local client
  participant Server
  participant Others as Other clients

  User->>Client: Click "Add Task"
  Client->>Client: Write row to SQLite
  Client->>Client: Queue outbox commit
  Client->>Server: POST /api/sync (push)
  Server->>Server: Validate handler and apply row
  Server->>Server: Write sync_commits + sync_changes
  Server-->>Client: Ack with commitSeq
  Server-->>Others: Optional wake-up signal
  Others->>Server: POST /api/sync (pull)
  Server-->>Others: Return commits since cursor
  Others->>Others: Apply rows to local SQLite
`}
  caption="Local writes happen first. Syncular persists the change, pushes it when possible, and other clients pick it up on their next pull."
/>

**Bootstrap:** The first time a subscription syncs, the server returns a snapshot, then switches to incremental commits. For scale/performance details see [Bootstrap Deep Dive](/docs/concepts/first-sync).

**Conflicts:** Version conflicts are detected on push; your app chooses resolution. See [Conflict Resolution](/docs/concepts/conflict-resolution).
