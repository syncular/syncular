# Quick Start

> **Alpha.** APIs, wire formats, and storage layouts will change between releases. Pin versions and expect manual migrations between versions.

## Run the local checks

If you want to verify the framework repo itself, clone it and run the canonical
Rust browser client checks:

```bash
git clone https://github.com/syncular/syncular.git
cd syncular
bun install
bun run client:test
bun run client:tsgo
bun run javascript-bindings:build:wasm
```

## Build your own app

The implementation path is:

1. [Server Setup](/server/setup-with-hono)
2. Choose [JavaScript Client](/clients/javascript) or
   [Rust Client](/clients/rust)
3. Define the app schema with migrations and `syncular.app.ts`
4. Generate the app client for your language
5. Learn the model: [Scopes](/learn/scopes) and
   [Subscriptions](/learn/subscriptions)
6. Then add features as needed: blobs, encryption, migrations, observability,
   console, relay

If you’re not sure where to start, see [Pick Your Path](/start/pick-your-path),
or build the smallest end-to-end app first with [Hello World](/start/hello-world).

## JavaScript first

Use this path for browser, React, React Native/Expo, Tauri, or Electron apps
where app code is TypeScript/JavaScript. This is still the Rust-owned client
runtime; JavaScript owns the Kysely/read, mutation-helper, React, and host
bridge surface:

```bash
npm install @syncular/client @syncular/react kysely
npm install -D syncular @syncular/typegen
npx syncular codegen install
```

Then continue with [JavaScript Quick Start](/clients/javascript/quick-start).

## Rust first

Use this path for Rust app code or native shells embedding Syncular directly:

```bash
cargo add syncular syncular-client
cargo add --dev syncular-testkit
cargo install syncular-codegen --locked --version 0.1.0
npm install -D @syncular/typegen
```

Then continue with [Rust Quick Start](/clients/rust/quick-start).

## What happens on a write (high level)

Here's the full lifecycle of adding a task:

<Mermaid
  chart={`
sequenceDiagram
  autonumber
  actor User
  participant Client as Local client
  participant Server
  participant Others as Other clients

  User->>Client: Click "Add Task"
  Client->>Client: Runtime writes row to SQLite
  Client->>Client: Queue outbox commit
  Client->>Server: POST /api/sync (push)
  Server->>Server: Validate handler and apply row
  Server->>Server: Write sync_commits + sync_changes
  Server-->>Client: Ack with commitSeq
  Server-->>Others: Realtime delta or pull-required wake-up
  Others->>Server: POST /api/sync (pull)
  Server-->>Others: Return commits since cursor
  Others->>Others: Runtime applies rows to local SQLite
`}
  caption="Local writes happen first. Syncular persists the change, pushes it when possible, and other clients receive it through realtime deltas or the next pull."
/>

**Bootstrap:** The first time a subscription syncs, the server returns a snapshot, then switches to incremental commits. For scale/performance details see [Bootstrap Deep Dive](/learn/first-sync).

**Conflicts:** Version conflicts are detected on push; your app chooses resolution. See [Conflict Resolution](/learn/conflict-resolution).
