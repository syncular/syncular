# Generated Client

JavaScript apps should not hand-write table metadata, mutation namespaces, or
runtime feature contracts. Generate them from your SQLite migrations and
`syncular.app.ts`, commit the output, and regenerate whenever either input
changes.

## The app contract: `syncular.app.ts`

The generator reads two inputs:

- `migrations/**/up.sql` — the local SQLite replica shape, in plain SQL.
- `syncular.app.ts` — the app contract: Syncular semantics that SQL cannot
  express, such as subscription ids, scopes, server-version columns, blob
  columns, CRDT fields, encrypted fields, and output paths.

```ts
import { defineSyncularClient, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  typescriptOutputPath: 'src/generated/syncular.generated.ts',
  typescriptServerOutputPath: 'src/generated/syncular.server.generated.ts',
  tables: {
    tasks: syncedTable({
      table: 'tasks',
      subscriptionId: 'sub-tasks',
      scopes: [scope('user_id', { source: 'actorId', required: true })],
      serverVersion: 'server_version',
    }),
  },
});
```

This is the same contract shape used by `apps/demo` and the
`create-syncular-app` template.

## The generate workflow

```bash
npx syncular generate
```

One command runs the whole pipeline (`packages/syncular`):

1. `syncular.app.ts` is evaluated and serialized to
   `generated/syncular.codegen.json` — a language-neutral handoff.
2. The Rust `syncular-codegen` binary combines that handoff with your
   migrations and emits the language clients. The binary is auto-installed
   into Syncular's tool cache on first run (or install it explicitly with
   `npx syncular codegen install`; `SYNCULAR_CODEGEN_BIN` points at a custom
   binary).

Useful flags: `--manifest-dir <path>` (where `syncular.app.ts` and
`generated/` live, default `.`), `--migrations-dir <path>`, `--app <path>`,
and `--check`. Rust-only projects can omit `syncular.app.ts` entirely;
`syncular generate` then initializes the codegen config from migrations.

Keep the check form in CI so generated files never drift from the contract:

```bash
npx syncular generate --check
```

## What gets generated

For a TypeScript app the interesting outputs are:

- **`src/generated/syncular.generated.ts`** — the client module:
  - a typed Kysely `SyncularAppDb` interface plus per-table row types
    (`TaskRow`, `NewTask`, `TaskPatch`, payload builders);
  - `createSyncularAppDatabase(options)` — the open helper that wires
    schema version, embedded migrations, codecs, table config, and required
    runtime features into [`CreateSyncularDatabaseOptions`](/clients/javascript/client-configuration),
    installs the local schema, sets default subscriptions, wraps
    mutations with command history (undo/redo), and starts the sync
    lifecycle (initial sync plus realtime) before resolving;
  - `database.schemaReadiness()` — an app-shaped readiness helper that fills
    in the generated schema version, so UI code and tests can check
    generated/runtime/local/server schema compatibility without passing
    version constants around;
  - table-scoped local-visibility helpers such as
    `database.awaitTaskVisibility(...)` — generated wrappers over the raw
    local-visibility primitive that inject the table list for the matching
    app table;
  - mutation namespaces (`mutations.tasks.insert/update/...`) and
    subscription helpers (`taskSubscription({ actorId })`);
  - changed-row helpers for row/field deltas;
  - `assertSyncularAppRuntime(...)` runtime/feature assertions.
- **`src/generated/syncular.server.generated.ts`** — server-side table
  handler scaffolding and codecs for `@syncular/server`.
- **`generated/syncular.codegen.json`** — the contract handoff (input to the
  Rust binary; also what Rust-only apps edit).
- **`syncular.schema.json`** — a readable summary of the app schema:
  `appSchemaVersion`, `clientSchemaSupport`, migrations, and table metadata.

The same run can emit Rust (Diesel), Swift, and Kotlin clients for native
hosts — the `create-syncular-app` template keeps them under `generated/`.

Generated files are app code: commit them so TypeScript, React, and bridge
code can see concrete table types, and so the app builds without the codegen
toolchain installed.

## When to regenerate

Regenerate after changing:

- any migration SQL (new table, new column, index with semantic meaning);
- anything in `syncular.app.ts` (scopes, subscription ids, blob/CRDT/encrypted
  fields, output paths);
- the Syncular package versions (the generated header pins runtime
  expectations).

Do not edit generated files by hand — the next `generate` run overwrites
them, and `--check` in CI will flag the drift.

## Version handling

Every generated client carries a schema version
(`syncularGeneratedSchemaVersion`, derived from your migrations) and a
support window (`syncularGeneratedClientSchemaSupport` with
`current`/`minSupported`/`supported`). Three layers enforce it:

- **Open time:** `createSyncularAppDatabase` stamps `config.schemaVersion`
  and embeds migrations; `assertSyncularAppRuntime` fails fast if the loaded
  WASM runtime's package version, worker protocol version, or feature set
  does not match what the client was generated against, or if the local
  database was written by a *newer* schema than the generated code.
- **Local migrations:** older local databases are migrated forward using the
  embedded migration SQL.
- **Sync time:** the server rejects pushes from incompatible clients with
  `sync.schema_mismatch` (regenerate and ship) or
  `sync.client_schema_unsupported` (the client is older than the server's
  `minSupported` — users must upgrade).

See [Troubleshooting](/clients/javascript/troubleshooting#schema-version-mismatch)
for the recovery paths, and [Migrations](/features/migrations) for how to
evolve the schema safely.

Generated app databases also expose `schemaReadiness()` with the generated
version already supplied:

```ts
const readiness = await database.schemaReadiness();

if (!readiness.ready) {
  showSchemaIssue(readiness.issues[0]);
}
```

Use the lower-level `@syncular/client` readiness helper only when you are
building an ungenerated or tooling-facing client.

## Local visibility after commands

Reads stay plain Kysely, but app code often needs to wait until a committed
server command, realtime wakeup, or bootstrap update is visible in the local
read model. Generated databases expose one helper per app table:

```ts
const rows = await database.awaitTaskVisibility(
  (db) =>
    db
      .selectFrom('tasks')
      .select(['id', 'title'])
      .where('project_id', '=', 'project-1')
      .execute(),
  {
    timeoutMs: 1_000,
    predicate: (rows) => rows.length > 0,
  }
);
```

The generated helper injects `tables: ['tasks']` for you. Use the raw
`awaitLocalVisibility(...)` helper only for advanced multi-table waits or
ungenerated clients, and avoid using manual `sync()` calls as the normal answer
to stale reads.
