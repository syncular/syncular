# Testkit Primitives

Use these helpers to remove test boilerplate without binding your tests to a
specific runner.

## Protocol builders

Build sync payloads with typed helpers instead of hand-writing JSON objects:

- `createSyncUpsertOperation(...)`
- `createSyncDeleteOperation(...)`
- `createSyncSubscription(...)`
- `createSyncPushRequest(...)` — wraps one commit into a push batch
- `createSyncPullRequest(...)`
- `createSyncCombinedRequest(...)` — combined push + pull body

```ts
import {
  createSyncCombinedRequest,
  createSyncSubscription,
  createSyncUpsertOperation,
} from '@syncular/testkit';

const request = createSyncCombinedRequest({
  clientId: 'client-a',
  push: {
    commits: [
      {
        clientCommitId: 'commit-1',
        schemaVersion: 1,
        operations: [
          createSyncUpsertOperation({
            table: 'tasks',
            rowId: 't1',
            payload: { title: 'Task', completed: 0, project_id: 'p0' },
          }),
        ],
      },
    ],
  },
  pull: {
    schemaVersion: 1,
    limitCommits: 50,
    subscriptions: [
      createSyncSubscription({
        id: 'tasks-p0',
        table: 'tasks',
        scopes: { user_id: 'u1', project_id: 'p0' },
      }),
    ],
  },
});
```

`push` takes a `commits` array — the same `SyncPushBatchRequest` shape a real
client sends — so multi-commit batches are just more array entries.
`createSyncUpsertOperation` accepts an optional `baseVersion` for conflict
tests (see
[Testing Strategy — layer 4](/testing/strategy#layer-4--conflict-resolution)).

## HTTP protocol calls

Post protocol payloads to a fixture (or any running server) with actor headers
and schema-validated responses in one call:

- `postSyncCombinedRequest({ fetch, url, actorId, body })`
- `postSyncPushRequest(...)` / `postSyncPullRequest(...)`
- `postJsonWithActor(...)` — generic JSON POST with actor headers
- `createJsonActorHeaders(...)` — headers only (defaults to `x-actor-id`,
  override with `actorHeader`)

Each returns `{ response, json }` where `json` has already passed the protocol
parser.

## Response parsing

When you make raw `fetch` calls yourself, validate responses at runtime with
the schema-backed parsers:

- `parseSyncCombinedResponse(value)` / `parseSyncPushResponse(value)` /
  `parseSyncPullResponse(value)` — for already-parsed JSON
- `readSyncCombinedResponse(response)` / `readSyncPushResponse(response)` /
  `readSyncPullResponse(response)` — read + parse a `Response`

## Inspecting pull results

Pull responses nest changes inside `subscriptions[].commits[].changes`. These
helpers flatten the digging:

```ts
import {
  findSubscriptionChange,
  requireRevokedSubscription,
  subscriptionChangeRow,
  subscriptionChanges,
} from '@syncular/testkit';

// All changes delivered for one subscription id:
const changes = subscriptionChanges(json.pull?.subscriptions, 'tasks-p0');

// One row's change and its row payload:
const change = findSubscriptionChange(json.pull?.subscriptions, 'tasks-p0', 't1');
const row = subscriptionChangeRow(change); // Record<string, unknown> | undefined

// Negative-path tests should assert the stable protocol state:
requireRevokedSubscription(json.pull?.subscriptions, 'tasks-p0');
```

For push results and negative-path assertions, use:

- `findPushCommit(...)` / `requirePushCommit(...)`
- `findPushOperationResult(...)` / `requirePushOperationResult(...)`
- `requirePushErrorCode(...)`
- `findPullSubscription(...)` / `requirePullSubscription(...)`
- `requireRevokedSubscription(...)`

These helpers fail with the observed commit ids, operation indexes, statuses,
or subscription ids, so app tests can assert `sync.forbidden`, conflicts, and
revoked subscriptions without parsing human error messages.

## Command proof assertions

`database.commandTimeline(...)` exposes `summary.proof` so E2E tests can
assert the ordered command evidence chain without interpreting timeline event
arrays. It also exposes `summary.evidence` with the concrete subscription ids,
request id, sync attempt id, realtime cursor, pull reason, local-apply link,
and local visibility state/source for debugging failed proofs. Use
`requireCompleteCommandProof(...)` when the test should prove the whole chain,
or `requireCommandProofEvidence(...)` for narrower browser or support-flow
checks:

```ts
import {
  requireCommandProofEvidence,
  requireCompleteCommandProof,
} from '@syncular/testkit';

const timeline = await database.commandTimeline({
  command: mutation.commit,
  localVisibility,
});

requireCompleteCommandProof(timeline.summary.proof);
requireCommandProofEvidence(timeline.summary.proof, [
  'requestCorrelated',
  'realtimeCursorObserved',
  'pullReasonObserved',
]);
```

The assertion failure names the missing evidence keys, such as
`realtimeCursorObserved` or `localVisibilityObserved`, so a failed test points
at the broken lifecycle phase instead of a generic timeout.

## Failure artifact assertions

Use the failure artifact helpers when a smoke writes
`browser-preview-failure.json` or `cloudflare-runtime-failure.json`. They
assert the redacted, bounded artifact shape and fail with the field path that
is malformed or unsafe:

```ts
import {
  assertFailureArtifactRedacted,
  requireBrowserPreviewFailureArtifact,
  requireCloudflareRuntimeFailureArtifact,
} from '@syncular/testkit';

const browserArtifact = requireBrowserPreviewFailureArtifact(
  JSON.parse(await Bun.file('browser-preview-failure.json').text()),
  { forbiddenSubstrings: ['raw-test-token'] }
);

requireCloudflareRuntimeFailureArtifact(
  JSON.parse(await Bun.file('cloudflare-runtime-failure.json').text())
);

assertFailureArtifactRedacted(browserArtifact);
```

The browser assertion checks marker metrics, support-policy counts,
support-bundle redaction, lifecycle resume/pause evidence, starter timing
fields, and bounded page text excerpts. The Cloudflare assertion checks route,
exit, bounded output, and safe blob timing/byte metrics.

## Reference tasks schema

The testkit bundles a complete project-scoped `tasks` table so tests do not
need their own schema: `PROJECT_SCOPED_TASKS_DDL`,
`ensureProjectScopedTasksTable(db)`, `createProjectScopedTasksHandler()` (a
full `ServerTableHandler` with scope checks, snapshot paging, version
conflicts), plus shorthand builders `createProjectScopedTasksSubscription` and
`createProjectScopedTaskUpsertOperation`. See the
[Quick Start](/clients/javascript/testing/quick-start) for them in action.

Pass explicit project membership when you need deterministic denied-scope
tests:

```ts
import {
  createProjectMembership,
  createProjectScopedActorHeaders,
  createProjectScopedTestActor,
} from '@syncular/testkit';
import { createProjectScopedTasksHandler } from '@syncular/testkit/server';

const actor = createProjectScopedTestActor({
  actorId: 'user-a',
  projectId: 'campaign-a',
});
const deniedProject = createProjectScopedTestActor({
  actorId: 'user-b',
  projectId: 'campaign-b',
});

const handler = createProjectScopedTasksHandler({
  projectsByActor: createProjectMembership([actor, deniedProject]),
});

const headers = createProjectScopedActorHeaders(actor);
```

After you send a denied push or pull, assert stable protocol fields instead
of matching messages:

```ts
import {
  requirePushErrorCode,
  requireRevokedSubscription,
} from '@syncular/testkit';

requirePushErrorCode(deniedWrite.json.push, {
  clientCommitId: 'commit-denied',
  code: 'sync.forbidden',
});
requireRevokedSubscription(
  deniedPull.json.pull?.subscriptions,
  'tasks-campaign-b'
);
```

With explicit membership, writes outside the actor's projects return
`sync.forbidden`, and subscriptions outside membership are revoked. Assert
those outcomes with `requirePushErrorCode(...)` and
`requireRevokedSubscription(...)` so campaign/project join, denied
membership, and auth-context replacement tests stay tied to stable protocol
fields instead of response prose.

## Diagnostic markers

Use stable marker helpers for app diagnostics, server logs, and E2E scans
instead of matching human messages:

- `SYNCULAR_DX_MARKER_CODES` — named constants for common markers such as
  `sync.scope_revoked`, `realtime.hello`, `realtime.sync_wakeup`,
  `realtime.pull_required`, `blob.forbidden`, `blob.signing_failed`, and
  `sync.local_visibility_timeout`
- `findDiagnosticMarker(events, code, predicate?)`
- `hasDiagnosticMarker(events, code, predicate?)`
- `requireDiagnosticMarker(events, code, predicate?)`

```ts
import {
  requireDiagnosticMarker,
  SYNCULAR_DX_MARKER_CODES,
} from '@syncular/testkit';

const denied = requireDiagnosticMarker(
  diagnostics,
  SYNCULAR_DX_MARKER_CODES.blobAccessDenied,
  (event) => event.details?.accessReason === 'missing_reference'
);

expect(denied.code).toBe('blob.forbidden');
```

## Deterministic IDs and clock

Use deterministic helpers instead of ad-hoc counters and `Date.now()` coupling:

- `createIdFactory(...)` — `next()`, `peek()`, `reset()`
- `createCommitIdFactory(...)` — id factory with a `commit` prefix
- `createFakeClock(...)` — `now()`, `iso()`, `set()`, `advance()`, `tick()`

```ts
import { createCommitIdFactory, createFakeClock } from '@syncular/testkit';

const commits = createCommitIdFactory({ prefix: 'commit', startAt: 1 });
const clock = createFakeClock({ startMs: 1_700_000_000_000, tickMs: 1000 });

const id1 = commits.next(); // 'commit-1'
const t1 = clock.tick(); // 1_700_000_001_000
```

## Realtime WebSocket helpers

For suites that hit a realtime-enabled server
(see [Realtime WebSocket](/server/realtime-websocket)):

- `createRealtimeWsUrl(...)` — builds the `ws(s)://.../sync/realtime` URL with
  `clientId` and actor query params
- `openRealtimeWs(...)` — opens a socket from the same options
- `waitForWsOpen(ws)` / `waitForWsMessage(ws, { predicate })` /
  `waitForWsJsonMessage(ws, { predicate })`
- `parseWsJsonMessage(event)`
- `closeWsSafe(ws)`

See the
[multi-client realtime example](/clients/javascript/testing/examples-multi-client-realtime)
for a full flow.

## Bridge harness (Tauri / React Native shells)

`createClientBridgeHarness()` provides an in-process `SyncularBridge` backed by
real SQLite, with recorded invocations, mutation batches, fake Tauri
`invoke`/`listen`, and a React Native native-module stand-in — for testing
host-shell glue without a device.

## Audit assertions

For server audit/redaction tests: `assertAuditChangeRedacted`,
`assertAuditJsonExcludes`, `assertAuditDebugExportRedacted`, plus
`readJsonResponse`/`readTextResponse` response readers.

## Resource lifecycle helpers

If you want to remove cleanup boilerplate, use
[Resource Lifecycle (`using`)](/clients/javascript/testing/lifecycle):

- callback wrappers: `withAsyncDisposableFactory`, `withNodeHonoServer`,
  `withRealtimeWs`, `withChildProcess`
- resource wrappers: `createAsyncDisposableResource`,
  `createNodeHonoServerResource`, `openRealtimeWsResource`,
  `createChildProcessResource`
