# Runtime Patterns

Runtime tests usually duplicate the same three pieces:

- app table DDL
- server table handler logic
- Hono → runtime server bridge

`@syncular/testkit` provides reusable runtime-safe primitives so runtime
suites stay aligned:

- `createProjectScopedTasksHandler(...)` / `ensureProjectScopedTasksTable(...)`
- `createNodeHonoServer(...)` / `closeNodeServer(...)`
- `waitForJsonPortFromStdout(...)` / `stopChildProcess(...)`
- `createRealtimeWsUrl(...)`, `openRealtimeWs(...)`, `waitForWsOpen(...)`,
  `waitForWsJsonMessage(...)`, `closeWsSafe(...)`

## Node / Deno server pattern

`createNodeHonoServer` bridges a Hono app onto `node:http` (with CORS
handling built in). Wire the shared DDL and handler into real sync routes:

```ts
import { createDatabase } from '@syncular/core';
import { createBunSqliteDialect } from '@syncular/dialects/bun-sqlite';
import { ensureSyncSchema } from '@syncular/server';
import { createSqliteServerDialect } from '@syncular/server-dialect-sqlite';
import { createSyncRoutes } from '@syncular/server-hono';
import {
  createNodeHonoServer,
  createProjectScopedTasksHandler,
  ensureProjectScopedTasksTable,
} from '@syncular/testkit';
import { Hono } from 'hono';

const db = createDatabase({
  dialect: createBunSqliteDialect({ path: ':memory:' }),
  family: 'sqlite',
});
const dialect = createSqliteServerDialect();

await ensureSyncSchema(db, dialect);
await ensureProjectScopedTasksTable(db);

const app = new Hono();
app.route(
  '/sync',
  createSyncRoutes({
    db,
    dialect,
    handlers: [createProjectScopedTasksHandler()],
    authenticate: async (c) => {
      const actorId = c.req.header('x-actor-id');
      return actorId ? { actorId } : null;
    },
  })
);

const server = createNodeHonoServer(app, { cors: false });
await new Promise<void>((resolve) => server.listen(0, resolve));
```

If these exact steps are all you need, `createHttpServerFixture` performs them
in one call and returns `{ db, dialect, app, httpServer, baseUrl, destroy }` —
see the [Quick Start](/clients/javascript/testing/quick-start). Use the manual
wiring above when a runtime needs its own database adapter or route layout.

## Spawned-process runtimes

For runtimes you start as a child process (`deno run`, `wrangler dev`,
release binaries), the testkit covers the process plumbing:

```ts
import { spawn } from 'node:child_process';
import { waitForJsonPortFromStdout, withChildProcess } from '@syncular/testkit';

const child = spawn('deno', ['run', '-A', 'server.ts'], {
  stdio: ['ignore', 'pipe', 'pipe'],
});

await withChildProcess(child, async () => {
  // Server prints a JSON line such as {"port":4173} on stdout.
  const port = await waitForJsonPortFromStdout(child, {
    processName: 'deno server',
  });

  // ...run the shared protocol flow against http://localhost:${port}
});
```

`withChildProcess` stops the process (SIGTERM, then SIGKILL on timeout) even
when assertions throw; `stopChildProcess` and `createChildProcessResource` are
the lower-level pieces.

## Cloudflare worker pattern

Use the same tasks helper in your Durable Object setup, then attach sync +
blob routes. This keeps scope/auth/snapshot/apply behavior identical across
runtime suites; only the database binding (D1) and process bootstrap differ.
See [Cloudflare](/server/cloudflare) for the production wiring.

## When to keep runtime-specific code

Keep runtime-specific code only for:

- process lifecycle (`spawn`, `SIGTERM`, `wrangler dev`)
- runtime database adapters (`node:sqlite`, D1 bindings)
- transport environment constraints (headers/query auth differences)

Everything else should be shared through testkit primitives. The
[runtime release smoke example](/clients/javascript/testing/examples-runtime-release-smoke)
shows one protocol flow reused across all runtime suites.
