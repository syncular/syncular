# Fault Injection

`withFaults(transport, options)` wraps any `SyncTransport` (the
`@syncular/core` interface with `sync` and `fetchSnapshotChunk`) and injects
deterministic failures. Two base transports are common in tests:

- `createMockTransport()` from the testkit — an in-memory transport that acks
  every push and returns empty pulls; ideal for unit-testing code that
  consumes a transport
- `createHttpTransport({ baseUrl })` from `@syncular/core/http` — the
  real HTTP transport, pointed at a
  [`createHttpServerFixture`](/clients/javascript/testing/quick-start) for
  end-to-end fault drills

The wrapper returns `{ transport, getState, reset, setOptions }`.

## Deterministic failure after N requests

```ts
import { createMockTransport, withFaults } from '@syncular/testkit';

const wrapped = withFaults(createMockTransport(), {
  failAfter: 1, // first request passes, then every request fails
  failOnPush: true,
  failWith: new Error('simulated push failure'),
});

await wrapped.transport.sync(pushRequest); // succeeds
await expect(wrapped.transport.sync(pushRequest)).rejects.toThrow(
  'simulated push failure'
);

expect(wrapped.getState()).toEqual({
  pushCount: 1,
  pullCount: 0,
  fetchCount: 0,
  failureCount: 1,
});
```

`failOnPush` / `failOnPull` / `failOnFetch` scope which operations are
targeted; with none set, all operations are. The wrapper classifies a
`sync(request)` call as a `push` when `request.push` is present, otherwise as
a `pull`.

## Scripted plans and ack loss

Use a `plan` for exact pass/fail sequences. `phase: 'after'` fails the caller
*after* the base transport succeeded — the server applied the commit but the
client never saw the ack. Push delivery is idempotent per `clientCommitId`,
so the retry must not double-apply; this test proves it end to end:

```ts
import { createHttpTransport } from '@syncular/core/http';
import {
  createHttpServerFixture,
  createProjectScopedTasksHandler,
  createProjectScopedTaskUpsertOperation,
  createSyncCombinedRequest,
  ensureProjectScopedTasksTable,
  withFaults,
} from '@syncular/testkit';

const server = await createHttpServerFixture({
  serverDialect: 'sqlite',
  createTables: ensureProjectScopedTasksTable,
  handlers: [createProjectScopedTasksHandler()],
  authenticate: async (c) => {
    const actorId = c.req.header('x-actor-id');
    return actorId ? { actorId } : null;
  },
});

try {
  const wrapped = withFaults(
    createHttpTransport({
      baseUrl: server.baseUrl, // the transport posts to `${baseUrl}/sync`
      getHeaders: () => ({ 'x-actor-id': 'u1' }),
    }),
    {
      plan: [
        {
          operation: 'push',
          phase: 'after', // server applies, then the ack is "lost"
          failWith: new Error('SIMULATED_ACK_LOSS'),
        },
      ],
    }
  );

  const request = createSyncCombinedRequest({
    clientId: 'client-a',
    push: {
      commits: [
        {
          clientCommitId: 'commit-ack-loss',
          schemaVersion: 1,
          operations: [
            createProjectScopedTaskUpsertOperation({
              taskId: 'task-1',
              title: 'Survives ack loss',
            }),
          ],
        },
      ],
    },
  });

  await expect(wrapped.transport.sync(request)).rejects.toThrow(
    'SIMULATED_ACK_LOSS'
  );

  // Retry the same clientCommitId: replayed from the idempotency cache,
  // not applied twice.
  const retry = await wrapped.transport.sync(request);
  expect(retry.push?.commits[0]?.status).toBe('applied');

  const rows = await server.db.selectFrom('tasks').select('id').execute();
  expect(rows).toHaveLength(1);
} finally {
  await server.destroy();
}
```

Plan steps support `action: 'pass' | 'fail'`, `repeat`, per-step `latencyMs`,
and `operation: 'push' | 'pull' | 'fetch' | 'any'`; steps are consumed in
order per matching operation.

## Flaky and slow networks

```ts
const wrapped = withFaults(baseTransport, {
  flaky: 0.3, // 30% failure probability per request
  latencyMs: 150, // fixed added latency per request
});
```

Prefer scripted plans in CI — `flaky` is random and best kept for local soak
testing.

## Toggling faults mid-test (offline → reconnect)

`setOptions` updates the wrapper in place, and `reset` clears counters and
re-arms the original plan:

```ts
const wrapped = withFaults(baseTransport, { failOnPush: true, failAfter: 0 });

await expect(wrapped.transport.sync(pushRequest)).rejects.toThrow(); // offline

wrapped.setOptions({ failAfter: undefined, failOnPush: false }); // back online

await wrapped.transport.sync(pushRequest); // succeeds
```

See the
[offline reconnect example](/clients/javascript/testing/examples-offline-reconnect)
for the full worked flow.

## Request recording

Use `withRecording` to assert exact sync request payloads:

```ts
import { withRecording } from '@syncular/testkit';

const recorded = withRecording(baseTransport);

await recorded.transport.sync(request);

expect(recorded.syncRequests).toHaveLength(1);
expect(recorded.syncRequests[0]?.push?.commits[0]?.clientCommitId).toBe(
  'commit-ack-loss'
);
recorded.clear();
```

`recorded.fetchRequests` captures snapshot-chunk fetches the same way. Wrappers
compose: `withRecording(withFaults(base, options).transport)` records exactly
what survived the fault layer.

## Asserting client-side outbox behavior

Fault wrappers operate at the transport/protocol level. To assert that an app
client queues offline writes and drains its outbox after reconnect, use the
real client's `diagnosticSnapshot()` (`outboxStats.pending`) — the pattern is
in [Testing Strategy — layer 3](/testing/strategy#layer-3--offline-queue-and-reconnect).
