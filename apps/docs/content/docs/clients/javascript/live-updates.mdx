# Live Updates

JavaScript clients refresh UI from runtime events. The runtime emits row,
field, query, outbox, conflict, blob, CRDT, and presence signals; app code maps
those signals to views.

## Row deltas

Generated helpers turn generic `changedRows` events into table-specific
decisions:

```ts
import { syncularChangedRows } from './generated/syncular.generated';

const unsubscribe = syncular.on('rowsChanged', (event) => {
  for (const task of syncularChangedRows.tasks(event)) {
    if (task.isDelete) removeTask(task.rowId);
    if (task.changed.title || task.changed.completed) refreshTask(task.rowId);
    if (task.crdt.title_yjs_state) refreshActiveEditor(task.rowId);
  }
});
```

Use row and field metadata instead of reloading an entire screen for every
table-level change.

## React query refresh

`useSyncQuery` refreshes when relevant row or query events arrive:

```tsx
const { data } = useSyncQuery(
  ({ selectFrom }) =>
    selectFrom('tasks')
      .selectAll()
      .where('user_id', '=', actorId),
  { watchTables: ['tasks'] }
);
```

## Operational state

Use runtime events or React hooks for app chrome:

```ts
syncular.on('outboxChanged', (stats) => {
  setPendingSyncCount(stats.pending + stats.sending);
});

syncular.on('conflictsChanged', (stats) => {
  setConflictCount(stats.unresolved);
});
```

React apps can use `useOutboxStats()`, `useConflictStats()`, and
`useSyncConnection()` from [React](/clients/javascript/react).

## Overflow recovery

If an event stream reports overflow, treat incremental state as lost:

1. resubscribe;
2. run sync if appropriate;
3. refresh affected Kysely queries from SQLite;
4. resume incremental updates.
