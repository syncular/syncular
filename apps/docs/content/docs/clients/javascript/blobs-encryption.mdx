# Blobs And Encryption

Blob storage and encryption live in the runtime because they affect local
SQLite state, sync payloads, retry behavior, event streams, and generated
mutation semantics.

## Blob references

Mark blob reference columns in `syncular.app.ts`:

```ts
syncedTable({
  table: 'tasks',
  serverVersion: 'server_version',
  blobColumns: ['image'],
});
```

The generated mutation type treats `image` as a `BlobRef`.

## Store and mutate

```ts
const ref = await syncular.blobs.store(file, {
  mimeType: file.type,
});

await syncular.mutations.tasks.update(taskId, {
  image: ref,
});
```

Queued uploads emit blob events and can be retried by the runtime. Keep file
picker UI, thumbnail generation, and platform file permissions in the host app.

## Retrieve and cache

Use runtime blob APIs to retrieve and cache file bodies. Do not put raw binary
payloads in synced rows.

```ts
const bytes = await syncular.blobs.retrieve(ref);
const objectUrl = URL.createObjectURL(new Blob([bytes], { type: ref.mimeType }));
```

## Encryption

Configure field, CRDT, and blob encryption before writing protected data:

```ts
const key = await syncular.client.encryptionHelper(
  'generateSymmetricKey'
);

syncular.client.setFieldEncryption({
  rules: [
    {
      scope: 'tasks',
      table: 'tasks',
      fields: ['description'],
      rowIdField: 'id',
    },
  ],
  keys: { default: key },
  encryptionKid: 'default',
});
```

Generated mutations stay the public write path. App code should not manually
encrypt individual fields after generation, because Syncular needs the metadata
for sync, conflicts, diagnostics, and recovery.
