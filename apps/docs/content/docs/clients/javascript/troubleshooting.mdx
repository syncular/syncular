# Troubleshooting

Every client error carries a structured envelope (`SyncularClientError`):
`code`, `category`, `retryable`, and a `recommendedAction`. Start with
`client.diagnosticSnapshot()` — it bundles recent diagnostics, sync timings,
outbox/conflict stats, and subscription state. `client.localHealthCheck()`
validates local state and suggests repair actions.

## Sync stalled (writes never reach the server)

Check `lifecycleState()` and the outbox stats (`outbox.pending`,
`outbox.sending`, `outbox.failed`):

- **Phase `offline` / code `sync.offline`** — the network source reports
  offline, or fetches fail. Retryable; the engine resumes when the source
  fires `online`. If you passed a custom `sync.network`, make sure it
  actually emits events.
- **Phase `authRequired` / code `sync.auth_required`** — the token expired
  and no refresh succeeded. Wire `authLifecycle.refreshToken` so 401s trigger
  a single-flight refresh and retry; watch for `auth.refresh_failed`
  diagnostics. `sync.forbidden` (403) is *not* retryable — the actor lacks
  scope access.
- **Outbox `failed` > 0** — a commit was rejected; check
  `conflictSummaries()` (below) and the diagnostic log.
- **Nothing happens after mutations** — `sync.autoSyncAfterMutation` was
  disabled, or no lifecycle/poll loop is running. Call `syncOnce()` to test.

## WASM artifact missing

Open fails with `Syncular WASM runtime artifact is missing (...)`
(`packages/client/src/wasm-runtime.ts`). The WASM glue module could not be
fetched. In this repo, build it first: `bun run javascript-bindings:build:wasm`
(or `build:wasm:dev`). In an app, make sure the
`@syncular/client-javascript-bindings` `dist/wasm` assets are served by your
bundler/host — and that your dev server isn't rewriting the `.js`/`.wasm`
URLs to `index.html`.

## Schema version mismatch

- **`sync.schema_mismatch`** (recommended action `regenerateClient`): the
  generated client no longer matches the server. Run `npx syncular generate`
  and redeploy.
- **`sync.client_schema_unsupported`** (`upgradeClient`): the server dropped
  support for this client's schema version. Users must update the app.
- **Open-time throw `Syncular Rust app schema version mismatch`**: the local
  database was written by a newer generated client than the code now running
  — usually a rollback. Ship the newer build or clear local data.

See [Generated Client → Version handling](/clients/javascript/generated-client#version-handling).

## Conflicts not resolving

`sync.version_conflict` means the server rejected a push because the row
changed since the client read it. The commit stays parked until you act:

```ts
const conflicts = await client.conflictSummaries();
await client.resolveConflict(id, 'keep-server'); // or 'keep-local' | 'dismiss'
await client.retryConflictKeepLocal(id);         // rebase local payload and retry
```

If `conflicts.unresolved` keeps growing, nothing in your app subscribes to
`conflictsChanged` — add UI or an automatic policy. See
[Conflict Resolution](/learn/conflict-resolution).

## Blobs failing

- **`blob.too_large`** — payload exceeds `blobLimits.maxPayloadBytes`
  (default 64 MiB). Raise the limit on client *and* server, or chunk the file.
- **`blob.storage_not_configured`** — the server has no blob store wired;
  see [Server Blobs](/server/blobs).
- **`blob.upload_failed` / uploads stuck in `pending`** — uploads are queued,
  not synchronous. Check `blobs.getUploadQueueStats()`, subscribe to
  `blobUploadFailed`, and force a retry with
  `blobs.processUploadQueue({ retryNow: true })`. Failed uploads are
  retried with backoff; `blob.hash_mismatch` / `blob.size_mismatch` indicate
  the bytes changed between store and upload.

## Last resorts

`sync.integrity_rejected` (hash/chain verification failed) recommends a
forced resync: `client.resetLocalSyncState()` re-bootstraps subscriptions
without dropping local rows; `repairLocalHealth({ action: 'forceRebootstrap' })`
does the same from a health finding. Both are safe — the server remains the
source of truth.
