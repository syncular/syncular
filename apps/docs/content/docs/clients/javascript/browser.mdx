# Browser TypeScript

Browser TypeScript keeps Kysely as the app query builder. SQLite, sync state,
outbox, conflicts, blobs, CRDT persistence, and encryption live in the Rust
WASM Worker.

## Create the database

Generated browser code creates the typed Kysely database and validates runtime
features against the app schema:

```ts
import { createSyncularAppDatabase } from './generated/syncular.generated';

const syncular = await createSyncularAppDatabase({
  config: {
    baseUrl: '/sync',
    actorId: 'user-1',
    clientId: 'device-1',
    projectId: 'project-1',
    fileName: 'app.sqlite',
  },
  getHeaders: async () => ({
    authorization: `Bearer ${await auth.accessToken()}`,
  }),
  authLifecycle: {
    refreshToken: () => auth.refreshAccessToken(),
  },
});
```

The database owns the sync lifecycle: opening it registers subscriptions, runs
the initial sync, and connects realtime (on by default for remote databases),
and `close()` shuts everything down. Without generated code, the same
single-step open is available directly from `@syncular/client`:

```ts
import { createSyncularDatabase } from '@syncular/client';

const syncular = await createSyncularDatabase<AppDb>({
  config: {
    baseUrl: '/sync',
    actorId: 'user-1',
    clientId: 'device-1',
  },
  subscriptions: [
    {
      id: 'tasks:user-1',
      table: 'tasks',
      scopes: { user_id: 'user-1' },
    },
  ],
});
```

Realtime is treated as a wakeup path: websocket reconnects trigger HTTP
catchup sync, and interval polling stays opt-in through `pollIntervalMs`. Pass
`lifecycle: { autoStart: false }` to open the database without starting sync,
then call `syncular.start()` yourself.

Generated clients select packaged browser runtime artifacts from app schema
metadata, so normal browser apps do not import worker or WASM artifact helpers
manually.

## Deployment preflight

Before opening a production or preview build, check that the deployed page can
serve and run Syncular's browser runtime. Run this from the page itself or a
real browser smoke so storage APIs reflect the target browser. The preflight
does not start the Worker or open SQLite; it verifies Worker/WebAssembly
support, HTTPS/localhost secure context for durable storage, OPFS/IndexedDB
availability, persistent storage status, quota, and WASM asset status/content
types. It also reports browser lifecycle and multi-tab primitives:
`BroadcastChannel`, Web Locks, page visibility, `pagehide`, and
`beforeunload`.

```ts
import { getSyncularBrowserDeploymentPreflight } from '@syncular/client';
import { syncularGeneratedRequiredRuntimeFeatures } from './generated/syncular.generated';

const preflight = await getSyncularBrowserDeploymentPreflight({
  requiredRuntimeFeatures: syncularGeneratedRequiredRuntimeFeatures,
  minimumQuotaBytes: 50 * 1024 * 1024,
  requireMultiTabCoordination: true,
  requirePageLifecycleResume: true,
});

if (preflight.status === 'not-ready') {
  console.error(preflight.issues);
}
```

Common issue codes include `browser.runtime_asset_bad_status`,
`browser.runtime_asset_bad_content_type`, `browser.opfs_unavailable`,
`browser.indexeddb_unavailable`, `browser.insecure_context`, and
`browser.storage_persistence_not_granted`. Apps that open the same durable
database from multiple tabs should require multi-tab coordination and use the
reported lifecycle fields to decide whether to allow multiple open tabs, use a
single-open-tab policy, or call `resumeFromBackground()` from host lifecycle
signals after tab suspension.

## Query with Kysely

Reads are normal Kysely queries:

```ts
const tasks = await syncular.db
  .selectFrom('tasks')
  .select(['id', 'title', 'completed'])
  .where('project_id', '=', 'project-1')
  .orderBy('server_version', 'desc')
  .execute();
```

## Mutate through Syncular

App-table writes are rejected from public SQL execution. Use generated
mutations so local rows, outbox, encryption, blobs, CRDT fields, conflicts, and
sync metadata stay coherent:

```ts
await syncular.mutations.tasks.insert({
  title: 'Draft',
  completed: 0,
  user_id: 'user-1',
  project_id: 'project-1',
});

await syncular.mutations.tasks.update(taskId, {
  completed: 1,
});
```

Local mutations schedule a background `syncOnce()` by default after the outbox
commit is written. Bursts are coalesced with a short debounce, and writes that
land during an active sync queue one follow-up sync. Apps that own their own
sync cadence can disable it:

```ts
const syncular = await createSyncularAppDatabase({
  config: {
    baseUrl: '/sync',
    actorId: 'user-1',
    clientId: 'device-1',
  },
  sync: {
    autoSyncAfterMutation: false,
  },
});
```

Use `mutationSyncDebounceMs` to tune the post-mutation sync debounce, and
`rowsChangedDebounceMs` when a UI wants to coalesce changed-row notifications.
Queued blob uploads stay manual by default; set
`autoProcessBlobUploadsAfterStore: true` and tune `blobUploadDebounceMs` when
the browser should process the upload queue after `blobs.store()`.

## Route row deltas

The runtime emits generic `changedRows`; generated helpers turn those into
typed app-table deltas:

```ts
import { syncularChangedRows } from './generated/syncular.generated';

const unsubscribe = syncular.on('rowsChanged', (event) => {
  for (const task of syncularChangedRows.tasks(event)) {
    if (task.isDelete) removeTask(task.rowId);
    if (task.changed.title || task.changed.completed) refreshTask(task.rowId);
    if (task.crdt.title_yjs_state) refreshActiveEditor(task.rowId);
  }
});
```

The same client surface exposes operational events for app chrome:

```ts
syncular.on('outboxChanged', (stats) => {
  setPendingSyncCount(stats.pending + stats.sending);
});

syncular.on('conflictsChanged', (stats) => {
  setConflictCount(stats.unresolved);
});

syncular.on('blobUploadCompleted', ({ ref }) => {
  markBlobUploaded(ref.hash);
});
```

Browser event names use the same Rust-native vocabulary as native event
payloads: `rowsChanged`, `outboxChanged`, `conflictsChanged`,
`presenceChanged`, `blobUploadCompleted`, and `blobUploadFailed`.

## Presence

Realtime presence uses the same websocket connection as sync messages. Join a
scope only after the client has subscribed and completed an initial sync, since
the server authorizes presence by the websocket connection's active scope keys.

A generated `createSyncularAppDatabase` (or `createSyncularDatabase` with
`subscriptions`) has already subscribed and completed the initial sync by the
time it resolves, so presence can be joined right away:

```ts
const unsubscribePresence = syncular.presence.onChange((event) => {
  renderPresence(event.scopeKey, event.presence);
});

syncular.presence.join('user:user-1', {
  editingTaskId: taskId,
});

syncular.presence.updateMetadata('user:user-1', {
  editingTaskId: nextTaskId,
});

const current = syncular.presence.get('user:user-1');

syncular.presence.leave('user:user-1');
unsubscribePresence();
```

Presence scope keys are the unpartitioned sync scope keys, such as
`user:user-1` for a `user:{user_id}` server handler scope. The browser client
normalizes partitioned server messages back to those app-facing keys.

## Rich editors

For rich editor fields, Yjs is the canonical document. ProseMirror JSON, title,
preview, outline, search text, and other UI/search projections should be
materialized from the CRDT field after local writes, remote apply, or
compaction.

```ts
const snapshot = await syncular.client.crdtDocumentSnapshot({
  table: 'tasks',
  rowId: taskId,
  field: 'title',
});

const log = await syncular.client.crdtUpdateLog({
  table: 'tasks',
  rowId: taskId,
  field: 'title',
  limit: 50,
});
```

The snapshot exposes the compact binary Yjs state, state vector, sync mode, and
pending/flushed/acked queue counts. The update log is append-only and tracks
each binary Yjs update through `pending`, `flushed`, and `acked`. Use
`compactCrdtField` to refresh the compact state and
`compactStorage({ olderThanMs, pruneCrdtUpdateLog: true })` to prune old acked
log entries without treating derived app columns as canonical.

## Runtime artifacts

The package ships runtime artifact metadata. Generated clients declare required
features from schema metadata, such as blobs, CRDT/Yjs, or E2EE, and the
browser package selects a compatible WASM artifact. When nothing is specified,
the packaged `full` artifact is used.

The full artifact covers blobs, CRDT/Yjs, and field-level encryption. Schemas
that need none of those can opt into the smaller core artifact with
`runtime: 'core'`.
