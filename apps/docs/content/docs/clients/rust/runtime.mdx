# Runtime

The native runtime is the Rust-owned execution boundary for local storage and
sync. Rust apps can use it directly. JavaScript hosts such as React Native,
Tauri, and Electron use a bridge to this same runtime, but their application
APIs are documented under [JavaScript](/docs/clients/javascript).

## Responsibilities

The runtime owns:

- SQLite connections, migrations, and runtime system tables
- local mutations, outbox commits, conflict commands, and retries
- HTTP sync, persistent realtime websocket wakeups, and auth refresh hooks
- row/field-level event streams for local writes, pulls, conflicts, blobs, and
  CRDT fields
- blob cache/upload/download work
- Yjs/Yrs CRDT persistence, materialization, compaction, and encrypted CRDT
  storage when enabled

App code owns:

- generated table-specific query and mutation code
- product-specific projections, list/detail refresh behavior, and editor
  adapters
- UI lifecycle decisions such as foreground/background transitions

## Worker lifecycle

Keep one hot runtime per app/session. Queue work that can block instead of
running it on a UI thread:

- `enqueueMutationJson`
- `enqueueSyncNow`
- queued CRDT field updates and compaction
- queued blob upload/cache/compaction work

The worker serializes writes, coalesces CRDT bursts, handles retry wakeups, and
emits ordered events. Reads can stay on the Rust SDK through
`SyncularClient::read(...)` or on a JavaScript bridge through the generated
Kysely client.

## Event stream

Start one event subscription per consumer that needs independent delivery.
Subscriptions are fan-out: one slow consumer should not steal events from
another consumer.

```rust
let worker = SyncWorker::start(client);
let events = worker.subscribe_events(256);
let converter = NativeWorkerEventConverter::new();

while let Some(event) = events.next_event() {
    for json in converter.convert_json(event)? {
        route_runtime_event(json)?;
    }
}
```

Every event has an ordered `eventSeq`. Row-changing events carry
`changedRows` with table, row id, operation, changed fields, CRDT fields, and
source metadata. Use those deltas to refresh only the affected projections.

If a bounded event queue overflows, the runtime emits `EventsOverflowed` with
`resyncRequired: true` and closes that subscription. Treat it as lost
incremental state: resubscribe, trigger sync if appropriate, and refresh live
queries from SQLite before trusting new incremental events.

## Realtime

Realtime is a wakeup source, not the only source of truth. Keep the websocket
worker hot when auth headers are available. The runtime owns reconnect/backoff
and feeds wakeups into the sync worker. HTTP catchup remains authoritative for
missed messages, reconnects, and recovery.

## Presence

Presence uses the realtime websocket session. Join a scope after subscriptions
and initial sync are established:

```rust
let mut presence = client.join_presence_handle(
    "user:user-rust",
    Some(serde_json::json!({ "view": "tasks" })),
)?;

presence.update_metadata(serde_json::json!({ "view": "task-detail" }))?;
drop(presence);
```

Joined scopes are remembered and re-sent after realtime reconnect.

## Rich editors

For rich editor fields, Yjs/Yrs state is the canonical document. Syncular owns
durable CRDT update storage, merge/materialization, coalescing, compaction, and
sync. App layers own TipTap/ProseMirror schemas, title/preview derivation,
selection state, undo stacks, and editor bridge messages.

Use generic CRDT field APIs instead of editor-specific storage code:

```rust
use syncular_client::crdt_field::CrdtFieldId;

let field = client.open_crdt_field(CrdtFieldId::new(
    "notes",
    note_id,
    "content_yjs_state",
))?;

client.apply_crdt_field_text(&field, "Draft body")?;
```
