# Client Configuration

`NativeSyncularClient::builder(config)` takes a `NativeClientConfig` and a
small set of builder switches
(`rust/crates/runtime/src/native/facade.rs`). This is the embed-direct native
surface: the same options apply whether your app is pure Rust or a Swift,
Kotlin, JVM, or JS shell embedding the runtime through FFI. The
[JavaScript page](/clients/javascript/client-configuration) documents the
worker-based browser surface, which exposes more knobs; where an option exists
there but not here, this page says so.

```rust
use syncular_client::native::{NativeClientConfig, NativeSyncularClient};

let mut client = NativeSyncularClient::builder(NativeClientConfig {
    db_path: data_dir.join("app-v1.sqlite").to_string_lossy().into_owned(),
    base_url: "https://app.example.com/api/sync".to_string(),
    client_id: device_id,
    actor_id: user_id,
    project_id: Some(project_id),
    app_schema_json: Some(generated_app_schema_json()?),
})
.auth_headers_json(r#"{"authorization":"Bearer <token>"}"#)?
.subscriptions(vec![tasks_subscription(&user_id, Some(&project_id))])
.initial_sync(true)
.shutdown_on_drop(true)
.open()?;
```

## Core identity (`NativeClientConfig`)

| Field | Default | Notes |
| --- | --- | --- |
| `db_path` | — (required) | Absolute path to the local SQLite file. The runtime creates it (plus `-wal`/`-shm` sidecars) on first open. Version the file name (`app-v3.sqlite`) when you want a clean local slate — there is no JS-style `clearOnInit`. |
| `base_url` | — (required) | The sync mount, e.g. `https://host/api/sync`. Also the source of the realtime websocket URL (below). |
| `client_id` | — (required) | Unique per device/replica. Two replicas sharing a `client_id` corrupt each other's cursors. |
| `actor_id` | — (required) | The authenticated user; feeds `actorId`-sourced scopes. |
| `project_id` | `None` | Feeds `projectId`-sourced scopes for multi-tenant apps. |
| `app_schema_json` | `None` | JSON app schema for FFI hosts. Pure-Rust apps usually skip it and pass the generated `AppSchema` via `open_with_options_and_schema` instead — see [Generated Client](/clients/rust/generated-client). |

There is no `mode` field. The JS `'local-sync-compatible'` equivalent is
`SyncularClientConfig::local_sync_compatible(db_path, client_id, actor_id, project_id)`
(`rust/crates/runtime/src/core/client.rs`) passed to
`NativeSyncularClient::open_with_options(..)` — same engine, no push/pull. The
JS `stateId` option (multiple sync states in one file) has no native
counterpart; use one file per state.

## Builder switches

Defaults from `NativeSyncularClientBuilder::new`:

| Option | Default | When to change |
| --- | --- | --- |
| `auto_sync_local_writes(bool)` | `true` | Push after every local write. Disable for bulk importers, then call `trigger_sync()` once at the end. The native side has no `mutationSyncDebounceMs` — coalescing is your loop's job. |
| `realtime(bool)` | `true` | Starts the persistent websocket worker on open. Disable for batch tools or tests without a realtime mount. |
| `auth_headers(..)` / `auth_headers_json(..)` | none | Initial header map attached to every HTTP and websocket request. |
| `subscriptions(..)` / `subscriptions_json(..)` | unset | Explicit `SubscriptionSpec` list. Unset keeps the generated schema's default subscriptions. |
| `initial_sync(bool)` | `false` | Trigger one HTTP sync immediately after open. |
| `initial_websocket_sync(bool)` | `false` | Same, but over the websocket transport; wins over `initial_sync` when both are set. |
| `process_blob_uploads_on_open(bool)` | `false` | Drain the blob upload queue right after open — useful after a crash with pending uploads. |
| `shutdown_on_drop(bool)` | `false` | Stop the sync and realtime workers when the client drops. Otherwise call `close()`/`shutdown()` yourself. |

`open()` blocks while SQLite opens and migrations run. UI hosts should use
`NativeSyncularClient::open_native_async_with_options(config, options)`, which
returns a `NativeClientOpenTask` — poll `is_finished()` and collect the client
with `take_client_timeout(..)` off the UI thread.

## Pull and push tuning

Unlike JS (`config.pull` / `config.push`), the native client has **no
per-client pull/push options**. It runs the shared runtime defaults from
`rust/crates/runtime/src/core/limits.rs`: pulls fetch up to `1000` commits and
`50,000` snapshot rows across `10` pages per round; pushes send `20` commits
per request and adaptively scale to `1000` once more than `100` commits are
due (e.g. after a long offline stretch). Sync failures retry up to 5 times
with exponential backoff (1s doubling to a 30s cap). Read the effective values
at runtime from `native_runtime_manifest().limits` or
`diagnostic_snapshot().limits` instead of hardcoding them.

## Transport

HTTP and websocket behavior is derived, not configured:

- **Timeouts** are fixed at the `SyncTransportTimeouts` defaults
  (`rust/crates/runtime/src/transport/mod.rs`): 10s connect, 30s request, 30s
  response body, 10s websocket open, 10s websocket push-response. There is no
  native `requestTimeoutMs`.
- **Realtime URL** is derived from `base_url`: `http`→`ws`, `https`→`wss`,
  path suffixed with `/realtime`, plus `clientId`, `schemaVersion`, and
  `transportPath=direct` query parameters. The JS `realtime.wsUrl` /
  `getParams` overrides do not exist; auth rides on headers, not URL params.
- **Reconnects** back off from 1s doubling to 30s (floor 250ms) and reset on
  success or whenever you call `set_auth_headers(..)`. Joined presence scopes
  are re-sent automatically after reconnect.

## Auth lifecycle

Native auth is a header map (`SyncAuthHeaders`, a `BTreeMap<String, String>`),
not a callback — there is no `getHeaders` or `authLifecycle`. On a 401 the
worker emits a `NativeEventKind::AuthExpired` event carrying
`auth: { operation, status }`, and the lifecycle phase becomes `authRequired`.
Refresh the token in your app and push it down:

```rust
client.set_auth_headers_json(r#"{"authorization":"Bearer <fresh>"}"#)?;
```

This updates the writer, the sync worker, and the realtime worker in one call
and resets the realtime backoff. `sync.forbidden` (403) is not retryable —
treat it as a permissions bug, not a stale token.

## Lifecycle and diagnostics

- `pause_sync_worker()` / `resume_sync_worker()` /
  `resume_from_background()` — background transitions for mobile hosts;
  resume re-applies subscriptions, headers, and encryption, then triggers a
  sync. See [Runtime](/clients/rust/runtime).
- `subscribe_events(capacity)` — fan-out event stream; the built-in default
  subscription holds 256 events, and overflow emits `EventsOverflowed` with
  `resyncRequired: true`.
- `diagnostic_snapshot()` — runtime manifest, limits, connection state,
  per-subscription bootstrap, the last 100 events, sync timings, and
  outbox/conflict/blob stats. The JS `consoleDiagnostics` streaming option has
  no native equivalent; export `diagnostic_snapshot_json()` or
  `export_local_support_bundle_json()` yourself.
- `local_health_check_json()` / `repair_local_health_json(..)` /
  `reset_local_sync_state_json(..)` — validation and recovery, mirroring the
  JS APIs.

## Worked tuning scenarios

**Mobile shell (iOS/Android lifecycle).** Open async with
`open_native_async_with_options`, set `shutdown_on_drop(true)`, call
`pause_sync_worker()` on background and `resume_from_background()` on
foreground so the first foreground sync is immediate instead of waiting for a
wakeup.

**Bulk importer.** `auto_sync_local_writes(false)` and `realtime(false)`,
enqueue mutations in a loop, then one `trigger_sync()`. Adaptive batching
takes over above 100 due commits, pushing up to 1000 per request.

**Restored-from-backup device with queued blobs.**
`process_blob_uploads_on_open(true)` plus `initial_sync(true)` drains stale
upload state and reconciles cursors before the UI reads anything.
