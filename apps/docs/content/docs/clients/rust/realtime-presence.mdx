# Realtime And Presence

Realtime is a wakeup path. HTTP sync remains the authoritative catchup path for
missed commits, reconnects, auth refresh, and recovery after event overflow.

## Realtime lifecycle

UI apps should keep the runtime hot while the user session is active. The
worker owns websocket reconnect and backoff, then feeds sync wakeups into the
same worker queue as manual sync commands.

```rust
native.start_realtime_worker()?;
native.enqueue_sync_now()?;
```

When realtime emits a sync wakeup, the runtime runs HTTP catchup. App code does
not need to babysit reconnects or call sync directly for every websocket
message.

## Auth refresh

Configure auth headers at the runtime boundary. On auth failure, refresh the
token and reconnect rather than letting hosts loop manually around failed
requests.

For native hosts, generated bindings expose lifecycle options and event-stream
errors so Swift, Kotlin, JVM, React Native, Tauri, and Electron bridges can
surface auth state cleanly.

## Presence

Presence uses the websocket session. Join a scope after subscriptions and the
initial sync are established:

```rust
let mut presence = client.join_presence_handle(
    "tasks:project-rust",
    Some(serde_json::json!({ "view": "tasks" })),
)?;

presence.update_metadata(serde_json::json!({ "view": "task-detail" }))?;
drop(presence);
```

Presence events are emitted through the same event stream as sync events:

```json
{
  "type": "presenceChanged",
  "scopeKey": "tasks:project-rust",
  "presence": [
    {
      "clientId": "device-1",
      "actorId": "user-rust",
      "metadata": { "view": "task-detail" }
    }
  ]
}
```

## Recovery model

Use this rule in app code:

- websocket event: wake sync and refresh from local SQLite after sync
- websocket reconnect: run catchup sync
- event overflow: resubscribe and refresh live queries
- auth failure: refresh token or show a session-expired state

This keeps realtime fast without making it responsible for durable state.
