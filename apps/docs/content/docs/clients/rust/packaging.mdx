# Packaging

This page covers Rust runtime artifact packaging. It is not a host-language API
guide. JavaScript app integrations are documented under
[JavaScript](/docs/clients/javascript); Rust application APIs are documented under
[Querying](/docs/clients/rust/querying), [Mutations](/docs/clients/rust/mutations),
and [Live Updates](/docs/clients/rust/live-updates).

Native packages are generated from the current Rust runtime, headers, and
bridge metadata in one pass. Do not treat checked-in generated binding files or
binary artifacts as release truth.

The current native FFI ABI version is `2`. Generated app clients assert this
against the runtime manifest before opening data, so regenerate app clients
after updating the runtime or native artifacts.

## Local release check

Run from the repo root:

```bash
bun run rust:native:release-check
```

This builds native runtime artifacts, host bridge smoke tests, and release
metadata into `.context/native-packages` by default. Override the output path
with:

```bash
SYNCULAR_NATIVE_PACKAGE_OUT=/path/to/out bun run rust:native:release-check
```

## Targeted commands

```bash
bun run rust:native:package:apple
bun run rust:native:package:android
bun run rust:native:package:java
bun run rust:native:package:java:linux
bun run rust:native:package:java:windows
```

Use targeted commands when validating a package surface locally or debugging CI
packaging failures.

## What packaging should prove

- the Rust runtime builds in release mode
- exported headers and bridge metadata match the current ABI
- event-stream APIs are present
- sync, auth refresh, realtime events, conflicts, blobs, and encrypted fields
  still work through the packaged runtime surface
- artifacts can be consumed by app hosts without relying on generated files in
  this repository
