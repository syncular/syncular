# Querying

Rust reads use Diesel's query builder. Syncular does not generate an ORM or
predefined query methods. Generated code gives you typed table declarations and
row structs; your app writes normal Diesel expressions.

The Syncular client owns the SQLite connection, so app code passes a query
expression to `client.read(...)` and never threads a `db` handle through the UI.

```rust
use diesel::prelude::*;
use generated::{diesel_tables::TaskRow, schema};

let tasks: Vec<TaskRow> = client.read(
    schema::tasks::dsl::tasks
        .filter(schema::tasks::dsl::user_id.eq("user-rust"))
        .order(schema::tasks::dsl::server_version.desc())
        .select(TaskRow::as_select()),
)?;
```

## Select row structs

Generated row structs implement the Diesel select shape for the table:

```rust
let open_tasks: Vec<TaskRow> = client.read(
    schema::tasks::dsl::tasks
        .filter(schema::tasks::dsl::completed.eq(0))
        .select(TaskRow::as_select()),
)?;
```

Use narrower custom selects when a read model does not need the whole row:

```rust
let task_titles: Vec<(String, String)> = client.read(
    schema::tasks::dsl::tasks
        .select((schema::tasks::dsl::id, schema::tasks::dsl::title)),
)?;
```

## Query semantics

Reads are local. They inspect the current SQLite replica and do not talk to the
server. Server catchup happens through `sync_http()`, realtime wakeups, or the
native worker.

This split is intentional:

- reads stay fast, typed, and query-builder based;
- writes stay on the Syncular mutation path;
- sync can run independently from UI query code.

## Raw SQL

Prefer Diesel query-builder reads for app tables. Raw SQL is useful for
diagnostics or specialized read models, but raw app-table writes must not bypass
Syncular. Direct inserts, updates, and deletes skip the outbox, conflicts,
encryption hooks, CRDT metadata, blob queues, and row-change events.

Use [Mutations](/docs/clients/rust/mutations) for app-table writes.
