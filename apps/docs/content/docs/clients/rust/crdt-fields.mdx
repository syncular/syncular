# CRDT Fields

Syncular treats CRDT-backed document fields as runtime primitives, not as
editor-specific APIs. The runtime owns durable Yjs/Yrs update storage, merge,
materialization, compaction, encryption hooks, worker coalescing, and sync.

Apps own editor adapters such as TipTap, ProseMirror schemas, title/preview
derivation, undo stacks, selection state, and WebView bridge messages.

## Configure a CRDT field

Declare CRDT fields in `syncular.app.ts`:

```ts
syncedTable({
  table: 'notes',
  serverVersion: 'server_version',
  crdt: {
    content: yjsText({ stateColumn: 'content_yjs_state' }),
  },
});
```

The generator records the CRDT metadata in `APP_TABLE_METADATA`, and the
runtime installs the CRDT system tables it needs.

## Open and update

Rust app code can use field IDs directly:

```rust
use syncular_client::crdt_field::CrdtFieldId;

let field = client.open_crdt_field(CrdtFieldId::new(
    "notes",
    note_id,
    "content_yjs_state",
))?;

client.apply_crdt_field_text(&field, "Draft body")?;
let materialized = client.materialize_crdt_field(&field)?;
```

Native hosts use the JSON methods exposed by the runtime and generated
bindings:

```json
{
  "table": "notes",
  "rowId": "note-1",
  "field": "content_yjs_state"
}
```

For high-frequency editor updates, enqueue updates through the worker. The
worker coalesces bursts by `(table, row_id, field)` before writing SQLite and
the outbox.

## Compaction

CRDT update logs grow unless they are compacted. Syncular can materialize the
current document, write a checkpoint, and prune redundant updates while keeping
sync safe.

Run compaction from the worker path for UI apps:

```rust
client.enqueue_crdt_field_compaction_json(&serde_json::json!({
    "table": "notes",
    "rowId": note_id,
    "field": "content_yjs_state"
}).to_string())?;
```
