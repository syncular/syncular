# Testing

Use `syncular-testkit` for Rust-first app tests. The goal is the same as the
older JavaScript testkit: avoid mocking Syncular internals while still keeping
tests local, deterministic, and fast.

## Add the testkit

Until crates are published, use path dependencies:

```toml
[dev-dependencies]
syncular-testkit = { path = "../syncular/rust/crates/testkit" }
syncular-runtime = { path = "../syncular/rust/crates/runtime" }
```

## Use generated app schema

Tests should pass your generated app schema into the testkit:

```rust
use syncular_testkit::{
    AppFixtureOptions, AppTestServer, assert_table_has_row,
    open_app_client_with_server,
};

#[test]
fn two_clients_sync_through_stateful_server() {
    let app_schema = my_app::generated::app_schema();
    let server = AppTestServer::new(app_schema);

    let mut writer = open_app_client_with_server(
        app_schema,
        server.clone(),
        AppFixtureOptions {
            client_id: "writer".to_string(),
            ..AppFixtureOptions::default()
        },
    ).unwrap();

    let mut reader = open_app_client_with_server(
        app_schema,
        server,
        AppFixtureOptions {
            client_id: "reader".to_string(),
            ..AppFixtureOptions::default()
        },
    ).unwrap();

    writer.client.apply_local_operation_json(
        &serde_json::json!({
            "table": "notes",
            "row_id": "note-1",
            "op": "upsert",
            "payload": { "title": "From writer", "owner_id": "user-rust" },
            "base_version": 0
        }).to_string(),
        None,
    ).unwrap();

    writer.client.sync_http().unwrap();
    reader.client.sync_http().unwrap();

    assert_table_has_row(&mut reader.client, "notes", "id", "note-1");
}
```

## What the testkit covers

The Rust testkit includes:

- disposable SQLite paths and cleanup
- scripted sync transports and local HTTP sync servers
- a stateful app test server for generated schemas
- blob upload/download test servers
- faulted transports for retry and duplicate delivery cases
- native client fixtures and event waiters
- CRDT field helpers and assertions
- outbox, conflict, blob queue, and cache assertions

Use the testkit for app behavior. Keep app-specific fixtures, assertions, and
domain rows in your app test layer.
