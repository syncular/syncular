# Native Apps

Native apps use the same Rust runtime foundation as browser apps, but access it
through the Rust SDK or generated BoltFFI bindings.

## Rust

Rust is the canonical native SDK surface. Reads use Diesel, and writes use
generated Syncular mutations:

```rust
let mut client = NativeSyncularClient::builder(config)
    .auto_sync_local_writes(true)
    .open()?;

let rows = client.read(
    schema::tasks::dsl::tasks
        .filter(schema::tasks::dsl::user_id.eq("user-rust"))
        .order(schema::tasks::dsl::server_version.desc())
        .select(diesel_tables::TaskRow::as_select()),
)?;

client
    .mutations()
    .tasks()
    .update(syncular::TaskPatch::new(task_id).completed(1))?;
```

The builder starts realtime by default. Use `.realtime(false)` only for a host
policy that cannot keep a websocket open.

## Swift and Kotlin

Swift and Kotlin get generated app helpers over the schema-agnostic native
binding:

- query DSLs for typed read SQL
- generated mutation builders
- generated subscription helpers
- typed CRDT field helpers
- typed row-delta helpers

Swift:

```swift
let rows = try TaskQuery
    .select()
    .filter(TaskQuery.userId.eq("user-rust"))
    .orderBy(TaskQuery.serverVersion.desc())
    .fetch(on: client)

for task in taskChangedRows(in: event) {
    if task.changed.title { refreshTask(task.rowId) }
}
```

Kotlin:

```kotlin
val rows = TaskQuery
    .select()
    .filter(TaskQuery.userId.eq("user-rust"))
    .orderBy(TaskQuery.serverVersion.desc())
    .fetch(client)

for (task in taskChangedRows(event)) {
    if (task.changed.title) refreshTask(task.rowId)
}
```

## UI lifecycle

UI shells should keep one hot native runtime per app/session. Prefer queued
APIs for work that can block:

- `enqueueMutationJson`
- `enqueueSyncNow`
- `enqueueSyncWebsocket`
- queued CRDT field updates and compaction
- queued blob upload/cache/compaction work

Start the native event stream once, read `nextEventJson()` from a background
task, and route updates by `eventSeq`, `commandId`, `changedRows`, and query
ids. The native worker owns backpressure for queued local writes and CRDT
bursts. If the stream emits `EventsOverflowed`, that subscription is no longer
trusted and closes after the overflow event. Discard it, start a new event
stream, trigger sync if appropriate, and force a full live-query/bootstrap
refresh from SQLite before processing incremental events again. The event
includes top-level `droppedCount` and `resyncRequired: true`. During teardown,
close the event stream and then call `shutdown()`.

Treat realtime as the normal native lifecycle. After auth headers are available,
call `start()` and keep the persistent realtime worker hot for the session. The
runtime owns websocket reconnect/backoff and feeds wakeups into the sync worker.
BoltFFI hosts can still call the explicit `startRealtimeWorker()` and
`stopRealtimeWorker()` names; C hosts can use either
`syncular_native_client_start(...)` / `syncular_native_client_stop(...)` or the
longer realtime-worker functions.

Native presence uses the same websocket session. Swift/Kotlin call
`joinPresence(scopeKey:metadataJson:)`, `updatePresenceMetadata(...)`,
`leavePresence(...)`, and `presenceJson(scopeKey:)`; C hosts use the matching
`syncular_native_client_*presence*` functions. Presence updates emit
`PresenceChanged` events with `payload_json.scopeKey` and
`payload_json.presence`.
Joined scopes are remembered and re-sent after realtime reconnect.

Rust, Swift, and Kotlin also expose presence handles for component/screen
lifecycles. Closing or dropping the handle leaves the scope:

```rust
let mut presence = client.join_presence_handle(
    "user:user-rust",
    Some(serde_json::json!({ "view": "tasks" })),
)?;
presence.update_metadata(serde_json::json!({ "view": "task-detail" }))?;
drop(presence);
```

```swift
let presence = try client.joinPresenceHandle(
    scopeKey: "user:user-rust",
    metadataJson: #"{"view":"tasks"}"#
)
try presence.update(metadataJson: #"{"view":"task-detail"}"#)
try presence.close()
```

```kotlin
client.joinPresenceHandle("user:user-rust", """{"view":"tasks"}""").use { presence ->
    presence.update("""{"view":"task-detail"}""")
}
```

App-specific Rust wrappers can subscribe at the worker layer instead of polling
or rebuilding their own event hub:

```rust
let worker = SyncWorker::start(client);
let events = worker.subscribe_events(256);
let converter = NativeWorkerEventConverter::new();

while let Some(event) = events.next_event() {
    for json in converter.convert_json(event)? {
        // Forward the same NativeEvent JSON shape used by Swift/Kotlin/JS hosts.
    }
}
```

Swift:

```swift
try client.start()
for try await eventJson in client.eventJsonStream(capacity: 256) {
        let event = try syncularDecodeNativeEvent(eventJson)
        await viewModel.apply(syncularEvent: event)
}
```

Kotlin:

```kotlin
client.start()
scope.launch(Dispatchers.IO) {
    client.forEachEventJson(256uL) { eventJson ->
        val event = syncularDecodeNativeEvent(eventJson)
        withContext(Dispatchers.Main) {
            viewModel.applySyncularEvent(event)
        }
        true
    }
}
```

Java:

```java
client.start();
client.forEachEventJson(256, eventJson -> {
    SyncularNativeEvent event = syncularDecodeNativeEvent(eventJson);
    viewModel.applySyncularEvent(event);
    return true;
});
```

## Realtime and sync

HTTP sync is the normal path. WebSocket sync is available for native hosts
through `enqueueSyncWebsocket` when the server route supports Syncular's
WebSocket push transport. Realtime notifications remain wake-ups; persisted
row state still comes from sync.

## Rich editors

For rich editor fields, the canonical state is the Yjs document stored by the
Rust client. App columns such as ProseMirror JSON, title, preview, outline, and
search text are derived read models and should be refreshed after CRDT
changed-row events, remote apply, or compaction.

Native bindings expose `crdtDocumentSnapshotJson` and `crdtUpdateLogJson`.
The snapshot includes the compact binary Yjs state, state vector, sync mode,
and pending/flushed/acked queue counts. The update log is append-only and
tracks each binary Yjs update through `pending`, `flushed`, and `acked`.
`compactCrdtFieldJson` refreshes the compact state, and storage compaction can
prune old acked CRDT update-log entries with `pruneCrdtUpdateLog`.

For WebView-hosted editors, keep Rust/Syncular in the native shell as the only
persistence path. The WebView can use
`@syncular/client-rust-crdt-adapters` with
`createSyncularCrdtWebViewHost()` and `createSyncularCrdtJsonTransport()` to
present the editor adapter with an async host, while native/Flutter handles the
`syncular.crdt.host.v1` JSON messages by calling the real native APIs:

- `openCrdtFieldJson`
- `enqueueCrdtFieldYjsUpdateJson`
- `applyCrdtFieldYjsUpdateJson`
- `materializeCrdtFieldJson`
- `crdtDocumentSnapshotJson`
- `crdtUpdateLogJson`
- `snapshotCrdtFieldStateVectorJson`
- `compactCrdtFieldJson`

Forward native `RowsChanged` events back to the WebView with
`type: "syncular.crdt.host.rowsChanged"`. Projection materializers in the
WebView then rebuild derived title, preview, outline, search text, and
ProseMirror JSON from canonical CRDT state after local writes, remote sync, and
compaction. Do not also open a browser-side Syncular client for the same editor
field.

If the host layer has JavaScript available, the adapter package also exports
`createSyncularCrdtHostResponseMessage()` for request handling and
`syncularCrdtRowsChangedMessageFromNativeEventJson()` for forwarding
`nextEventJson()` rows-changed events to the WebView. Swift, Kotlin, and Dart
hosts can mirror those small helpers directly: switch on `method`, call the
matching native JSON API above, then reply with the original request `id`.
