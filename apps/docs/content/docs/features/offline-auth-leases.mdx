# Offline Auth Leases

Offline auth leases let a client queue writes while offline with recorded auth
provenance. They are useful for mobile and desktop apps where users can work
for a while after losing network access.

## Contract

- The server issues a lease for an actor, schema version, TTL, and allowed
  scopes.
- The client stores the lease locally.
- Generated leased mutations attach lease provenance to queued commits.
- The server validates the lease and current business rules when the commit
  eventually pushes.

Auth leases are not permanent authorization. They make offline intent auditable
and bounded, while the server still owns final acceptance.

## Binding examples

The lease contract is binding-neutral. Browser TypeScript, Rust, Swift, Kotlin,
JVM, React Native, and desktop clients all store lease provenance locally and
send it with generated leased mutations.

### Browser TypeScript

```ts
const lease = await syncular.issueAuthLease({
  ttlMs: 60 * 60 * 1000,
  scopes: [
    { table: 'tasks', scopes: { user_id: 'user-1' } },
  ],
});

await syncular.upsertAuthLease(lease);
await syncular.leasedMutations.tasks.update(taskId, {
  title: 'Offline edit',
});
```

React apps can use `useLeasedMutations()`. Rust, Swift, Kotlin, and Tauri
bindings expose the generated leased mutation surface and auth lease JSON
methods.

## Server flow

Enable lease issuing on the server and keep the table handler authoritative.
At push time, the server checks schema version, TTL, scopes, actor, and current
business policy before applying the operation.

## Watch-outs

- Pick TTLs that match real offline behavior, not ideal connectivity.
- Keep scopes narrow enough to avoid broad stale access.
- Show an update-required or re-auth screen when the server cannot honor the
  queued lease.
- Test lease expiration, scope revocation, app upgrades, and undo/redo of
  leased commands.
