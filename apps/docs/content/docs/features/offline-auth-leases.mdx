# Offline Auth Leases

Offline auth leases let a client queue writes while offline with recorded
auth provenance. They are useful for mobile and desktop apps where users can
work for a while after losing network access — and where "this commit was
authorized at the time it was made" needs to be provable when it finally
pushes.

Auth leases are not permanent authorization. They make offline intent
auditable and bounded, while the server still owns final acceptance.

## Lease lifecycle

1. **Issue (online).** The client asks the server for a lease covering
   specific tables and scope values at the current schema version. The server
   intersects the request with the actor's allowed scopes (the same
   `resolveScopes` machinery as subscriptions) and returns a signed token.
2. **Store.** The client persists the lease record locally next to its
   SQLite database.
3. **Queue (offline).** Generated *leased* mutations attach the lease id to
   queued commits, recording which authorization the user was acting under.
4. **Validate (push).** When the commit eventually pushes, the server
   verifies the token signature, expiry, schema version, actor, and that the
   lease actually covers each operation — then still runs current business
   rules in the table handler before applying.

## What a lease contains

The signed payload (`SyncAuthLeasePayload`) carries:

- `leaseId`, `issuer`, `audience`, `actorId`, and an app-defined `subject`;
- `schemaVersion` the lease was issued for;
- `issuedAtMs` / `notBeforeMs` / `expiresAtMs` plus `maxClockSkewMs`;
- `scopes` — per subscription: table, scope values, and the allowed
  `operations`;
- `capabilities` — `allowBlobs`, `allowCrdt`, `allowEncryptedFields`
  (all default to `false`).

The TTL is server-bounded: requests are clamped to `maxTtlMs`, with a
15-minute default when nothing is configured.

## Client flow

```ts
// While online: issue and persist a lease for the user's task scope.
const lease = await syncular.issueAuthLease({
  schemaVersion: 1,
  ttlMs: 60 * 60 * 1000,
  scopes: [
    {
      subscriptionId: 'sub-tasks',
      table: 'tasks',
      values: { user_id: 'user-1' },
      operations: ['upsert', 'delete'],
    },
  ],
});
await syncular.upsertAuthLease(lease);

// While offline: write through the leased mutation surface.
await syncular.leasedMutations.tasks.update(taskId, {
  title: 'Offline edit',
});

// Inspect what is currently held.
const active = await syncular.activeAuthLeases('user-1');
```

`issueAuthLease` posts to the server's `POST /auth-leases/issue` route and
validates the request and response against the generated app schema (tables,
scopes, schema version) before storing anything. React apps can use
`useLeasedMutations()`; Rust and native hosts use the generated leased
mutation surface and the auth-lease JSON methods.

## Server flow

Provide an auth-lease config when creating the sync server — that enables
`POST /auth-leases/issue`. Issuance is built on `issueAuthLease` from
`@syncular/server`:

- you supply `issuer`, `audience`, a key id (`kid`), and a `signer`
  (`createWebCryptoEs256AuthLeaseSigner` wraps a WebCrypto ES256 key);
- requested scopes are resolved through your table handlers'
  `resolveScopes`, so a lease can never grant more than the actor's current
  allowed scopes — if nothing survives the intersection, no lease is issued;
- `defaultTtlMs` / `maxTtlMs` bound how long offline authority lasts.

At push time, `verifyAuthLeaseToken` checks signature, issuer/audience,
expiry (with bounded clock skew), and schema version;
`validateAuthLeaseOperation` / `authLeaseCoversOperation` check that each
queued operation's table, scope values, and operation kind are inside the
lease. The table handler then applies normal authorization and version
checks — a valid lease does not bypass current business policy.

## Watch-outs

- Pick TTLs that match real offline behavior, not ideal connectivity. A field
  team that syncs nightly needs more than the 15-minute default.
- Keep scopes narrow enough to avoid broad stale access; lease the
  subscriptions the user actually works in.
- Capabilities are opt-in: leases cover plain row mutations unless you grant
  `allowBlobs` / `allowCrdt` / `allowEncryptedFields`.
- Show an update-required or re-auth screen when the server cannot honor a
  queued lease (expired, revoked scope, unsupported schema version).
- Test lease expiration, scope revocation between issue and push, app
  upgrades that bump the schema version, and undo/redo of leased commands.
