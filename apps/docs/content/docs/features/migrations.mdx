# Schema Migrations

Syncular's Rust-first migration flow starts from app SQL migrations and a typed
`syncular.app.ts` contract. Codegen turns that into:

- a generated runtime handoff for Rust;
- typed browser/Kysely and Rust/Diesel app clients;
- generated subscription metadata, local table DDL, indexes, codecs, and
  mutation payload types;
- version-aware server helpers for older client schemas.

## What you'll learn

- How client SQLite migrations become generated app clients
- How to keep client and server schemas equal or intentionally different
- How schema versions flow through push and pull
- How to reject unsupported old clients cleanly
- How to build local-only apps without a backend

## Client SQLite migrations

Put client migrations in SQL files. These define the local SQLite shape the
generated clients query and mutate.

```text
your-app/
  migrations/
    0001_initial/
      up.sql
```

```sql
create table notes (
  id text primary key not null,
  title text not null,
  body text not null default '',
  owner_user_id text not null,
  server_version integer not null default 0
);

create index idx_notes_owner_user_id
  on notes (owner_user_id);
```

Do not add Syncular internal `sync_*` tables to app migrations. The Rust
runtime installs and migrates its own sync state tables.

## App contract

Describe sync semantics in `syncular.app.ts`. The contract is development-time
source; generated files are the app-specific client output.

```ts
import { defineSyncularClient, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  typescriptOutputPath: 'src/generated/syncular.browser.ts',
  clientSchemaSupport: {
    minSupported: 1,
  },
  tables: {
    notes: syncedTable({
      table: 'notes',
      subscriptionId: 'sub-notes',
      serverVersion: 'server_version',
      scopes: [
        scope('user_id', {
          column: 'owner_user_id',
          source: 'actorId',
          required: true,
        }),
      ],
    }),
  },
});
```

Generate the Rust handoff and app clients:

```bash
syncular-typegen codegen-config --app ./syncular.app.ts

cargo run --manifest-path rust/Cargo.toml -p syncular-codegen -- \
  --manifest-dir /absolute/path/to/your-app
```

Use the `--check` form in CI.

## Opening the generated client

The generated browser client installs the Syncular system schema, app tables,
generated indexes/read models, codecs, and subscriptions.

```ts
import { createSyncularAppDatabase } from './generated/syncular.browser';

const syncular = await createSyncularAppDatabase({
  config: {
    baseUrl: '/sync',
    actorId: userId,
    clientId,
    fileName: 'app.sqlite',
  },
});
```

Native and Rust clients consume the same generated schema contract. App code
does not hand-create local sync tables.

## Same server and client shape

If the server table has the same shape as the generated client table, use the
default server handler:

```ts
import { createServerHandler } from '@syncular/server';

export const notesHandler = createServerHandler({
  table: 'notes',
  scopes: [{ pattern: 'user:{user_id}', column: 'owner_user_id' }],
  versionColumn: 'server_version',
  resolveScopes: async (ctx) => ({
    user_id: [ctx.actorId],
  }),
});
```

The handler snapshots rows, applies generated mutations, increments
`server_version`, extracts scopes, and emits changes for pull/realtime.

## Different server and client shape

The server database does not have to match the client schema. Keep the client
shape in migrations/app contract, then project server rows into the generated
client row shape in the handler.

```ts
import {
  createSyncularAppServerHandler,
  syncularIsSupportedClientSchemaVersion,
  syncularGeneratedApp,
  syncularUnsupportedClientSchemaResult,
  type NoteMutationPayloadV1,
  type NoteRow,
} from './generated/syncular.server.generated';

export const notesHandler = createSyncularAppServerHandler<ServerDb, Auth>({
  table: syncularGeneratedApp.tables.notes,

  resolveScopes: async (ctx) => ({
    user_id: [ctx.actorId],
  }),

  async snapshot(ctx) {
    const rows = await ctx.db
      .selectFrom('documents')
      .select([
        'id',
        'title',
        'content as body',
        'owner_id as owner_user_id',
        'version as server_version',
      ])
      .where('owner_id', '=', ctx.actorId)
      .execute();

    return {
      rows: rows.map((row) =>
        syncularGeneratedApp.projectClientRowForVersion(
          'notes',
          row as NoteRow,
          ctx.schemaVersion
        )
      ),
      nextCursor: null,
    };
  },

  async applyOperation(ctx, op, opIndex) {
    if (op.op === 'delete') {
      const deleted = await deleteDocument(ctx, op.row_id);
      return {
        result: { opIndex, status: 'applied' },
        emittedChanges: [{
          table: 'notes',
          row_id: deleted.id,
          op: 'delete',
          row_json: null,
          row_version: deleted.version,
          scopes: { user_id: ctx.actorId },
        }],
      };
    }

    const payload = op.payload as NoteMutationPayloadV1;
    const stored = await saveDocument(ctx, op.row_id, payload);
    const clientRow = documentToNoteRow(stored);

    return {
      result: { opIndex, status: 'applied' },
      emittedChanges: [{
        table: 'notes',
        row_id: clientRow.id,
        // Internal emitted-change storage uses the row-write operation marker.
        // Generated app mutations still expose insert/update/delete.
        op: 'upsert',
        row_json: clientRow,
        row_version: clientRow.server_version,
        scopes: { user_id: clientRow.owner_user_id },
      }],
    };
  },
});
```

Server emitted-change records use lower-level protocol operation markers.
Generated client mutation APIs stay app-level: `insert`, `update`, and
`delete`.

## Schema versions

Every push and pull includes the client schema version. The generated client
sets that version from the current app schema, and generated server helpers
expose `ctx.schemaVersion` in `snapshot` and `applyOperation`.

Use schema versions to:

- project current server rows into older generated client row shapes;
- decode mutation payloads from older clients;
- reject versions your server no longer supports.

```ts
async applyOperation(ctx, op, opIndex) {
  if (!syncularIsSupportedClientSchemaVersion(ctx.schemaVersion)) {
    return syncularUnsupportedClientSchemaResult({
      opIndex,
      schemaVersion: ctx.schemaVersion,
    });
  }

  return applyCurrentNoteOperation(ctx, op, opIndex);
}
```

If the product cannot safely support an older schema, fail explicitly so the
app can show a "please update" screen.

## Deployment order

For compatible changes:

1. Deploy the server migration and handler that can read/write old and new
   client schema versions.
2. Release the app update with the new SQLite migration and generated client.
3. After telemetry shows old schema versions are gone, remove old projection or
   payload branches in a deliberate cleanup.

For incompatible changes, bump `clientSchemaSupport.minSupported` and reject
older clients from the server.

## Local-only apps

Apps can use the generated SQLite/Kysely/Rust/native client without a backend by
opening the client in local-sync-compatible mode:

```ts
const syncular = await createSyncularAppDatabase({
  config: {
    mode: 'local-sync-compatible',
    actorId: 'local-user',
    clientId: 'local-device',
    fileName: 'local.sqlite',
  },
  subscriptions: false,
});
```

Use generated mutations even in local-only mode so outbox-compatible metadata,
CRDT fields, blobs, encryption, command history, and future sync remain
coherent.

## Reset and repair

Prefer product-level reset flows over ad hoc table deletion. The runtime can
force selected subscriptions to bootstrap again:

```ts
await syncular.client.forceSubscriptionsBootstrap(['sub-notes']);
await syncular.sync();
```

For user logout or account switch, close the client and open a fresh
actor/client identity. Clear local files only when your product explicitly
wants to discard offline data.
