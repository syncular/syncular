# CRDT Fields

CRDT fields are for parts of a row that need automatic convergence, such as
rich text, editor state, or structured collaborative content. They are not a
replacement for the whole relational data model: durable product records stay
normal synced rows, and only the collaborative field becomes a Yjs document.

## Recommended shape

- Keep durable product records as normal synced rows.
- Mark only the collaborative field as a CRDT/Yjs field in the app contract.
- Treat the Yjs document as canonical for that field.
- Store title, preview, outline, and search text as derived read models.
- Let Syncular own Yjs update persistence, merge/materialization, compaction,
  realtime wake-ups, and sync retries.

## Declaring a CRDT field

CRDT fields are declared once in the generated app contract. The generated
metadata then drives JavaScript, Rust, and native runtime behavior.

```ts
import { syncedTable, yjsText } from '@syncular/typegen';

tasks: syncedTable({
  table: 'tasks',
  subscriptionId: 'sub-tasks',
  serverVersion: 'server_version',
  crdt: {
    title: yjsText({
      stateColumn: 'title_yjs_state',
      containerKey: 'title',
    }),
  },
});
```

`yjsText(...)` is shorthand for a field config with `kind: 'text'`. The full
per-field config (`SyncularCodegenCrdtYjsFieldConfig`) supports:

- `stateColumn` — the column holding the serialized canonical Yjs state.
- `containerKey` — the container inside the Yjs document (defaults to the
  field name).
- `rowIdField` — row id column, defaults to `id`.
- `kind` — `'text'`, `'xml-fragment'`, or `'prosemirror'`.
- `syncMode` — `'server-merge'` (default) or `'encrypted-update-log'`.

## Sync modes

**`server-merge`** — clients send Yjs updates inside the mutation payload
(under a `__yjs` envelope); the server merges them into the canonical state
column with Yjs itself, materializes the plaintext projection, and serves
state-vector-aware deltas on pull. The server can read the content, which
enables server-side search, audit, and projections.

**`encrypted-update-log`** — the server never sees plaintext. Encrypted
updates and checkpoints flow through the `sync_crdt_updates` /
`sync_crdt_checkpoints` system tables and clients converge by decrypting
locally. Use this for end-to-end encrypted collaboration; see
[Encryption](/features/field-encryption) for keys and the server-side system
handlers.

## Server-merge setup

For `server-merge` tables, add the Yjs push plugin from
`@syncular/server-plugin-yjs` so pushed update envelopes are merged before
your table handler applies the row:

```ts
import { createYjsServerPushPlugin } from '@syncular/server-plugin-yjs';

const yjsPlugin = createYjsServerPushPlugin({
  rules: [
    {
      table: 'tasks',
      field: 'title',
      stateColumn: 'title_yjs_state',
      kind: 'text',
    },
  ],
});

// createSyncServer({ sync: { handlers, plugins: [yjsPlugin], ... } })
```

The plugin loads the existing row, applies incoming Yjs updates to the stored
state, strips the envelope, and rewrites the payload so handlers see plain
columns. `createYjsServerModule` exposes the same logic à la carte
(`applyPayload`, `materializeRow`, `transformPullChanges`) for hand-rolled
handlers, and `buildYjsTextUpdate` / `applyYjsTextUpdates` cover server-side
edits such as imports or migrations.

## Client APIs

JavaScript and Rust expose the same low-level document primitives on the
runtime client:

- `openCrdtField` — open/load the document for a row field.
- `applyCrdtFieldYjsUpdate` — apply a local editor update.
- `materializeCrdtField` — refresh the derived plaintext column.
- `crdtDocumentSnapshot` / `crdtUpdateLog` — inspect canonical state and
  pending updates.
- `snapshotCrdtFieldStateVector` — state vector for delta sync.
- `compactCrdtField` — fold acknowledged updates into the snapshot.

Most editor integrations should not call these directly. The
`@syncular/client-crdt-adapters` package binds a Yjs `Y.Doc` to a row field
(`yjs-document-field-adapter`) with queued update flushing
(`enqueueCrdtFieldYjsUpdate`), backpressure events, and restore handling, so
editor bursts never block the UI thread. Pair it with TipTap/ProseMirror or
any Yjs-compatible editor.

## App boundary

Syncular owns the dangerous plumbing: durable CRDT storage shape, Yjs/Yrs
merge, update logs, checkpoints, worker coalescing, backpressure, encryption,
and convergence tests.

The app owns editor-specific behavior: TipTap/ProseMirror schema, extensions,
slash commands, title derivation, preview generation, selection, undo stack,
composition state, scroll position, and WebView bridge UI.

## Watch-outs

- Do not make every row a CRDT document by default.
- Do not persist editor UI state such as selection or scroll in the CRDT
  field.
- Compact old acknowledged update logs (`compactCrdtField`, and
  `pruneEncryptedCrdtSystemRows` server-side for encrypted streams);
  otherwise rich text histories grow without bound.
- Keep Syncular row mutations and editor-internal undo stacks separate — see
  [Undo / Redo](/features/undo-redo).
- For encrypted collaborative fields, use `syncMode: 'encrypted-update-log'`
  instead of hand-encrypting Yjs blobs in app code.
