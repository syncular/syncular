# Blobs and Media

## Use this recipe when

- records need to reference uploaded files, previews, or derived assets
- bootstrap size matters because the catalog is large
- operators need visibility into chunking, storage, and cleanup behavior

## Shape

- structured metadata (filename, owner, status, dimensions) in synced tables
- binary bodies through blob routes backed by S3/R2, the filesystem, or the
  database adapter
- content hashes (`BlobRef`) as the durable reference between rows and files
- snapshot chunking and storage cleanup planned from the start

## App contract

Declare which columns carry blob refs so the runtime tracks uploads and cache
state for them:

```ts
import { defineSyncularClient, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  typescriptOutputPath: 'src/generated/syncular.generated.ts',
  typescriptServerOutputPath: 'src/generated/syncular.server.generated.ts',
  tables: {
    photos: syncedTable({
      table: 'photos',
      subscriptionId: 'sub-photos',
      serverVersion: 'server_version',
      scopes: [scope('user_id', { source: 'actorId', required: true })],
      blobColumns: ['original', 'thumbnail'],
    }),
  },
});
```

## Server: blob routes plus a storage adapter

Mount blob routes next to the sync routes and pick an adapter for the bytes:

```ts
import { createBlobManager } from '@syncular/server';
import { createBlobRoutes } from '@syncular/server/hono';
import { createS3BlobStorageAdapter } from '@syncular/server/s3';

const adapter = createS3BlobStorageAdapter({
  client: s3Client,
  bucket: 'app-media',
  commands: { PutObjectCommand, GetObjectCommand, HeadObjectCommand, DeleteObjectCommand },
  getSignedUrl,
});

const blobManager = createBlobManager({ db, adapter });

const blobRoutes = createBlobRoutes({
  blobManager,
  authenticate: async (c) => {
    const actor = await verifyRequest(c.req.raw);
    return actor ? { actorId: actor.id } : null;
  },
  canAccessBlob: async ({ actorId, hash }) =>
    actorOwnsPhotoWithBlob(db, actorId, hash),
  maxUploadSize: 50 * 1024 * 1024,
});

app.route('/sync', syncRoutes);
app.route('/sync', blobRoutes);
```

`@syncular/server/filesystem` is the single-node equivalent;
the database adapter (`storeBlobInDatabase` / `readBlobFromDatabase`) is the
zero-infrastructure option for small deployments. See
[File / Blob Sync](/features/blobs) for the full route and adapter surface.

## Client: store, reference, upload in the background

```ts
// 1. Store locally — returns a content-hash ref immediately, even offline.
const original = await syncular.blobs.store(file, { mimeType: file.type });
const thumbnail = await syncular.blobs.store(await makeThumbnail(file), {
  mimeType: 'image/webp',
});

// 2. Reference the blobs from a synced row.
await syncular.mutations.photos.insert({
  id: photoId,
  user_id: userId,
  original,
  thumbnail,
});

// 3. The runtime uploads in the background; show queue state in the UI.
syncular.on('blobUploadsChanged', (stats) => updateUploadBadge(stats));

// On other devices: warm thumbnails for the visible gallery page.
await syncular.blobs.preload(visibleRows.map((row) => row.thumbnail));
```

Generate thumbnails and previews in the app layer and store them as their own
blobs — list views then never download originals.

## Large catalogs

- Bootstrap stays metadata-only: blob bodies are downloaded on demand, so a
  100k-photo catalog bootstraps as rows, not gigabytes.
- Snapshot chunking keeps those row bootstraps incremental — verify it with
  realistic data sizes (see [Bootstrap & Snapshots](/server/snapshot-pull)).
- Bound the local cache: `syncular.blobs.pruneCache(maxBytes)` on a product
  cadence, and check `getCacheStats()` before bulk preloads.

## Watch-outs

- Do not put binary payloads in normal row JSON.
- Separate authorization for metadata visibility from authorization to fetch
  blob URLs — `canAccessBlob` enforces the latter.
- Deleting rows does not delete bodies; schedule cleanup for unreferenced
  hashes (`createBlobCleanupScheduler` handles abandoned uploads, product GC
  is yours).
- Test bootstrap behavior on realistic dataset sizes, not only small
  fixtures; test interrupted upload, retry, cache clear, and redownload.
