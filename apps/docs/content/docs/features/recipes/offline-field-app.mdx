# Offline Field App

## Use this recipe when

- users may work offline for hours or days
- local reads and writes are the normal path
- connectivity is intermittent, expensive, or unpredictable

## Recommended shape

- stable partition key such as `account_id`, `site_id`, `region_id`, or `project_id`
- long-lived local SQLite storage with stable `clientId` and identity continuity
- explicit offline session strategy with [Offline Auth Leases](/docs/features/offline-auth-leases)
- conflict resolution UX for the few records that can actually collide

- align prune windows with real offline behavior
- keep scope boundaries stable so clients do not constantly re-bootstrap
- test reconnect churn, duplicate pushes, and stale sessions before release
