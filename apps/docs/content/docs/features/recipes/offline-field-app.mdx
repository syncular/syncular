# Offline Field App

## Use this recipe when

- users may work offline for hours or days
- local reads and writes are the normal path
- connectivity is intermittent, expensive, or unpredictable

## Shape

- stable partition key such as `site_id`, `account_id`, or `region_id` —
  a field worker's device subscribes to its assigned sites
- long-lived local SQLite storage with a stable `clientId` and identity
  continuity across app restarts
- explicit offline session strategy with
  [Offline Auth Leases](/features/offline-auth-leases)
- conflict resolution UX for the few records that can actually collide

## App contract

```ts
import { defineSyncularClient, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  typescriptOutputPath: 'src/generated/syncular.generated.ts',
  typescriptServerOutputPath: 'src/generated/syncular.server.generated.ts',
  tables: {
    inspections: syncedTable({
      table: 'inspections',
      subscriptionId: 'sub-inspections',
      serverVersion: 'server_version',
      // Generated subscriptions take a projectId arg; use it for the site.
      scopes: [scope('site_id', { source: 'projectId', required: true })],
    }),
  },
});
```

Server side, `resolveScopes` returns the sites the worker is assigned to —
same membership pattern as
[Multi-Tenant SaaS](/features/recipes/multi-tenant-saas), keyed on an
assignment table instead of project membership.

## Client: durable identity, lease before going dark

```ts
const database = await createSyncularAppDatabase({
  config: {
    baseUrl: 'https://field.example.com/sync',
    actorId: session.userId,
    clientId: stableDeviceId,        // persisted, never regenerated
    fileName: 'field-app-v1.sqlite', // long-lived local database
  },
  getHeaders: async () => ({ authorization: `Bearer ${session.token}` }),
  subscriptions: assignedSiteIds.map((siteId) =>
    inspectionSubscription({ actorId: session.userId, projectId: siteId })
  ),
});

// While online (e.g. at shift start): take a lease for the day's sites.
const lease = await database.client.issueAuthLease({
  schemaVersion: 1,
  ttlMs: 12 * 60 * 60 * 1000, // matches a shift, server clamps to maxTtlMs
  scopes: assignedSiteIds.map((siteId) => ({
    subscriptionId: 'sub-inspections',
    table: 'inspections',
    values: { site_id: siteId },
    operations: ['upsert'],
  })),
});
await database.client.upsertAuthLease(lease);

// While offline: write through the leased surface.
await database.leasedMutations.inspections.insert({ ...inspection });
```

The runtime queues offline writes in the outbox and retries on its own
schedule once connectivity returns — do not add polling loops. Surface
`useOutboxStats()` ("3 pending") and connection state in the app chrome so
workers trust the queue.

## Recovery paths

- **Conflicts**: with per-site, per-worker rows most writes cannot collide;
  for shared records, build the
  [conflict flow](/features/conflict-resolution) and test duplicate pushes.
- **Expired leases**: a lease that expires before push is rejected — show a
  re-auth screen and retry, do not drop the queued work.
- **Schema upgrades in the field**: version-aware handlers reject
  too-old clients explicitly; the app shows "please update" instead of
  corrupting data (see [Schema Migrations](/features/migrations)).
- **Resuming**: call `resumeFromBackground()` when the OS wakes the app so
  catch-up sync runs without waiting for user action.

## Watch-outs

- Align server prune windows with real offline behavior: pruning commits
  while devices are dark for two weeks forces re-bootstraps.
- Keep scope boundaries stable so clients do not constantly re-bootstrap;
  reassigning a worker should change subscriptions, not rewrite rows.
- Pick lease TTLs from the actual offline pattern (shift, route, trip), not
  ideal connectivity.
- Test reconnect churn, duplicate pushes, and stale sessions before release —
  `withFaults` from `@syncular/testkit` scripts exactly these failures.
