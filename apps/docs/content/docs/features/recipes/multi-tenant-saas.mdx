# Multi-Tenant SaaS

## Use this recipe when

- your product has clear tenant or organization boundaries
- users collaborate inside projects, workspaces, or accounts
- operators need to debug sync issues without raw database access

## Shape

- base scope boundary: `org_id` (the eviction/deletion boundary)
- collaboration scope: `project_id` — what users actually subscribe to
- private data scope: `user_id` for drafts, preferences, or personal queues
- backend: Postgres plus server handlers with explicit membership lookups
- client: local SQLite plus one subscription per active project

## App contract

Declare the project scope in `syncular.app.ts` so generated subscriptions
take a `projectId`:

```ts
import { defineSyncularClient, scope, syncedTable } from '@syncular/typegen';

export const app = defineSyncularClient({
  typescriptOutputPath: 'src/generated/syncular.generated.ts',
  typescriptServerOutputPath: 'src/generated/syncular.server.generated.ts',
  tables: {
    tasks: syncedTable({
      table: 'tasks',
      subscriptionId: 'sub-tasks',
      serverVersion: 'server_version',
      scopes: [scope('project_id', { source: 'projectId', required: true })],
    }),
    drafts: syncedTable({
      table: 'drafts',
      subscriptionId: 'sub-drafts',
      serverVersion: 'server_version',
      scopes: [scope('user_id', { source: 'actorId', required: true })],
    }),
  },
});
```

Run `npx syncular generate` after editing migrations or the contract.

## Server: membership-based authorization

Authorization lives in `resolveScopes` — a lease, subscription, or push can
never reach projects the actor is not a member of:

```ts
import { createServerHandler } from '@syncular/server';
import { createSyncServer } from '@syncular/server/hono';
import { syncularGeneratedCodecs } from './generated/syncular.generated';

const tasksHandler = createServerHandler<ServerDb, AppDb, 'tasks', Auth>({
  table: 'tasks',
  scopes: ['project:{project_id}'],
  codecs: syncularGeneratedCodecs,
  resolveScopes: async (ctx) => {
    const memberships = await ctx.db
      .selectFrom('project_members')
      .select('project_id')
      .where('user_id', '=', ctx.actorId)
      .execute();
    return { project_id: memberships.map((m) => m.project_id) };
  },
});

const { syncRoutes, consoleRoutes } = createSyncServer<ServerDb, Auth>({
  db,
  dialect,
  sync: {
    handlers: [tasksHandler, draftsHandler],
    authenticate: async (request) => {
      const session = await verifySession(request);
      return session ? { actorId: session.userId, orgId: session.orgId } : null;
    },
  },
  console: { token: process.env.CONSOLE_TOKEN! },
});
```

Store `org_id` as a column on every tenant-owned table even when it is not a
subscription scope — it is the routing key you will need for tenant deletion
and data eviction.

## Client: subscribe per active project

```ts
import { createSyncularAppDatabase, taskSubscription, draftSubscription }
  from './generated/syncular.generated';

const database = await createSyncularAppDatabase({
  config: {
    baseUrl: 'https://app.example.com/sync',
    actorId: session.userId,
    clientId: deviceId,
    fileName: 'app.sqlite',
    storage: 'indexedDb',
  },
  getHeaders: async () => ({ authorization: `Bearer ${session.token}` }),
  subscriptions: [
    ...activeProjectIds.map((projectId) =>
      taskSubscription({ actorId: session.userId, projectId })
    ),
    draftSubscription({ actorId: session.userId }),
  ],
});
```

When the user opens or leaves projects, call `setSubscriptions` with the new
list; revoked memberships surface as revoked subscriptions and the runtime
clears the local rows.

## Operator tooling

Mount the console routes and keep
[Audit & History](/features/audit-and-history) endpoints available to product
admins, so most "why is this tenant out of sync" questions never require raw
database access. See [Console](/operate/console).

## Watch-outs

- Do not make row-level scopes your primary partitioning strategy — subscribe
  to projects, not documents (see
  [Data Modeling](/features/data-modeling)).
- Test membership changes and revocation paths early: removing a member must
  revoke their subscription and clear local data.
- Make tenant deletion and data eviction part of your schema design, not an
  afterthought — that is what the `org_id` column is for.
