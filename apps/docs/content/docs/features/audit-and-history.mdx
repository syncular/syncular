# Audit & History

Use this guide for product-facing history views: recent sync history,
conflict context, and long-lived app-owned audit trails.

If you are investigating incidents or operating a live deployment, use the
Console guides in [Operate](/operate/console) instead.

## Use this page when

- you want users or admins inside your product to inspect recent sync history
- you need a durable business or compliance audit trail that outlives sync
  retention
- you want a clear boundary between product-facing history and operator-only
  forensics

## Choose the right history surface

| Need | Use |
|---|---|
| Product-facing recent sync history for authorized users | This guide plus the `/sync/audit/*` routes |
| Undo/redo of local commands | [Undo / Redo](/features/undo-redo) — local, per device, prunable |
| Long-lived business or compliance audit records | Your own `audit_log` table |
| Incident forensics, request payloads, client lag investigation | [Console](/operate/console) |

## Recommended building blocks

1. Runtime row events: `rowsChanged` with `commitId`, `commitSeq`, changed
   fields, and source metadata
2. Outbox and conflict counters for UX: `outboxChanged`, `conflictsChanged`,
   `useOutboxStats()`, and `useConflictStats()`
3. Commit attribution in apply handlers: `commitSeq`, `actorId`, `createdAt`
4. Scoped audit API: the `/sync/audit/*` routes below

## Client UX patterns

The following examples use the browser TypeScript/React surface because they
show product UI behavior. Rust, React Native, and desktop hosts receive the
same row/outbox/conflict event metadata through their generated clients or
native event streams.

### Immediate push feedback

```ts
const unsubscribeRows = syncular.on('rowsChanged', (event) => {
  for (const row of event.changedRows) {
    if (event.source === 'localWrite') {
      addRecentLocalChange({
        table: row.table,
        rowId: row.rowId,
        operation: row.operation,
        commitId: row.commitId,
        commitSeq: row.commitSeq,
      });
    }
  }
});

const unsubscribeOutbox = syncular.on('outboxChanged', (stats) => {
  if (stats.failed > 0) {
    showToast(`${stats.failed} local commit(s) need review`);
  }
});
```

### Conflict notifications

```tsx
import { createSyncularReact } from '@syncular/react';

const { useConflictStats } = createSyncularReact<AppDb>();

function ConflictNotifications() {
  const stats = useConflictStats();
  useEffect(() => {
    if (!stats?.unresolved) return;
    showToast(`${stats.unresolved} conflict(s) need review`);
  }, [stats?.unresolved]);
  return null;
}
```

## Sync audit API

The server exposes auth-scoped audit routes next to the sync routes. They are
partition/auth scoped and intended for product admin UI, not console-only
operator workflows:

- `GET /sync/audit/commits` — recent commits visible to the authenticated
  actor
- `GET /sync/audit/commits/{commitSeq}` — one commit with its changes
- `GET /sync/audit/rows/{table}/{rowId}` — row-level history within the
  actor's allowed scopes
- `GET /sync/audit/debug/export` — a size-bounded, redacted support bundle
  for the authenticated actor

Payloads are redacted by default: change entries are summarized to a change
kind (`app_row`, `delete`, `blob_reference`, encrypted envelope kinds), field
names, scope fields, and detected sensitive fields — row values are omitted.
Build history UI on the metadata; read current values through your normal
scoped queries.

## App-owned row audit recipe

Sync history is optimized for sync and may be pruned or compacted. If you
need long-lived row-level audit records, keep an app-owned `audit_log` table
and write it from your table handlers' `applyOperation`, where actor, commit,
and before/after rows are all in hand.

Suggested schema:

```sql
create table audit_log (
  id bigserial primary key,
  table_name text not null,
  row_id text not null,
  op text not null, -- insert | update | delete
  actor_id text,
  commit_seq bigint,
  changed_at timestamptz not null default now(),
  before_hash text,
  after_hash text,
  before_snapshot jsonb,
  after_snapshot jsonb
);

create index audit_log_table_row_changed_at_idx
  on audit_log (table_name, row_id, changed_at desc);

create index audit_log_actor_changed_at_idx
  on audit_log (actor_id, changed_at desc);
```

Because the audit write happens inside the same transaction as the applied
operation, the trail cannot drift from the data.
