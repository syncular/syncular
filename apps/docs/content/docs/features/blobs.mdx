# File / Blob Sync

Blob sync is for files, images, media, and other large binary payloads. Keep
metadata in synced tables and store binary bodies through Syncular blob routes
and storage adapters. Blob bodies never travel inside row commits, so large
files cannot inflate push/pull payloads.

## Recommended shape

- Put file metadata, ownership, status, and references in normal synced rows.
- Store the binary body through blob routes backed by object storage, the
  filesystem, or the database adapter.
- Use content hashes (`sha256:` refs) as durable references from rows to
  blobs; declare those columns as `blobColumns` in the app contract so the
  runtime knows which fields carry refs.
- Let the runtime queue uploads/downloads and emit progress or failure events.
- Keep image resizing, thumbnail generation, and file picker behavior in the
  app layer.

## Client API

Clients store the blob locally first, write a generated mutation that
references it, then let the runtime upload and retry in the background:

```ts
// Store locally; returns a content-hash BlobRef immediately.
const ref = await syncular.blobs.store(file, { mimeType: file.type });

// Reference it from a synced row.
await syncular.mutations.tasks.update(taskId, { image: ref });

// Later, anywhere: fetch bytes (local cache first, then download).
const bytes = await syncular.blobs.retrieve(ref);
```

The full `syncular.blobs` surface:

- `store(data, options?)` ‚Äî accepts `Blob`, `File`, or `Uint8Array`; hashes,
  caches locally, and enqueues the upload.
- `retrieve(ref)` / `isLocal(hash)` / `preload(refs)` ‚Äî read paths and cache
  warming.
- `processUploadQueue(options?)` ‚Äî explicit queue drain (the runtime also
  drains automatically; `blobUploadDebounceMs` tunes the debounce).
- `getUploadQueueStats()` / `getCacheStats()` ‚Äî counters for UI.
- `pruneCache(maxBytes?)` / `clearCache()` ‚Äî local cache lifecycle.

Upload progress and failures are part of the same runtime event model as
rows, outbox, conflicts, and presence: subscribe to `blobUploadsChanged`,
`blobUploadCompleted`, and `blobUploadFailed` instead of polling storage.

### Limits

The browser client rejects oversized payloads before they reach storage:
`blobLimits.maxPayloadBytes` defaults to 64 MB and a violation throws a
`SyncularClientError` plus emits a blob diagnostic. Server routes enforce
their own `maxUploadSize` (default 100 MB). Size both to your product, and
keep them consistent across client and server.

## Server routes

Mount blob routes next to sync routes and authorize them with the same actor
boundary. `createBlobRoutes` from `@syncular/server/hono` provides:

- `POST /blobs/upload` ‚Äî initiate an upload, returns a presigned URL (or
  reports the blob already exists ‚Äî content addressing dedupes).
- `POST /blobs/:hash/complete` ‚Äî finalize after the body is uploaded.
- `GET /blobs/:hash/url` ‚Äî presigned download URL.
- `PUT /blobs/:hash/upload` / `GET /blobs/:hash/download` ‚Äî direct
  token-authorized body transfer, used by the database storage adapter.

```ts
import { createBlobManager } from '@syncular/server';
import { createBlobRoutes } from '@syncular/server/hono';

const blobRoutes = createBlobRoutes({
  blobManager,
  authenticate: async (c) => {
    const actor = await verifyRequest(c.req.raw);
    return actor ? { actorId: actor.id } : null;
  },
  canAccessBlob: async ({ actorId, hash, partitionId }) => {
    // Enforce your tenant/ownership model for download URLs.
    return canActorSeeBlob(actorId, hash, partitionId);
  },
  maxUploadSize: 100 * 1024 * 1024,
});

app.route('/api/sync', blobRoutes);
```

`canAccessBlob` is the authorization boundary for bodies ‚Äî row visibility
(metadata) and the right to fetch the blob URL are separate decisions.

### Partitions, scopes, and shared data

Blob storage is partitioned the same way as the server routes. A `BlobRef`
contains the content hash, size, and MIME type; it does not carry an authority
grant by itself. The route can mint a download URL only when the current actor
and partition can see a synced row that references that hash.

For scoped apps, prefer a scoped reference row over a global hash lookup:

- Put global/base catalog rows in a global partition only when every consumer
  fetches them through that same global partition.
- If a campaign/project needs a package, file, or base asset, create a scoped
  package/file row in that campaign/project that references the blob hash.
- If multiple scopes intentionally share the same bodies, share the object
  storage bytes but keep separate scoped metadata rows or an explicit shared
  partition policy.
- Do not rely on "the hash exists somewhere" as access. A blob body can exist
  while `GET /blobs/:hash/url` correctly returns `blob.forbidden` because no
  row in the current scope grants access to it.

`createScopedBlobAccessDecisionChecker` from `@syncular/server` returns the
same scoped access decision that the route exposes in error details:

```ts
import {
  createBlobManager,
  createScopedBlobAccessDecisionChecker,
} from '@syncular/server';
import { createBlobRoutes } from '@syncular/server/hono';

const canAccessBlob = createScopedBlobAccessDecisionChecker({
  db,
  handlers: [photosHandler],
  references: [{ table: 'photos', blobColumns: ['original', 'thumbnail'] }],
});

const blobRoutes = createBlobRoutes({
  blobManager,
  authenticate,
  canAccessBlob,
});
```

Blob route failures include stable `details` fields for tests and diagnostics:

| Failure | Code | Details to assert |
| --- | --- | --- |
| No scoped row references the hash | `blob.forbidden` | `failureKind: "blob_access_denied"`, `accessReason: "missing_reference"`, `accessStage: "reference"` |
| A row references the hash but the actor's resolved scope cannot read it | `blob.forbidden` | `accessReason: "scope_denied"`, `accessStage: "scope"`, `referenceTable`, `referenceColumn` |
| The upload/blob metadata row is missing in this partition | `blob.not_found` | `failureKind: "blob_record_missing"` or `"upload_record_missing"` |
| Storage cannot mint a signed URL | `blob.signing_failed` | `failureKind: "signed_url_failed"`, `storageAdapter` |
| A direct download token points at missing bytes | `blob.not_found` | `failureKind: "storage_object_missing"`, `storageAdapter` |
| The direct upload/download token is invalid or expired | `blob.invalid_token` | `failureKind: "invalid_token"`, `tokenAction` |

## Storage adapters

The blob manager (`createBlobManager`) tracks upload state and delegates the
bytes to a storage adapter:

- **S3 / R2** ‚Äî `createS3BlobStorageAdapter` from
  `@syncular/server/s3`; pass an `S3Client`, bucket, the AWS SDK
  command constructors, and `getSignedUrl`. SHA-256 checksum validation is on
  by default. Works with Cloudflare R2 (see [Cloudflare](/server/cloudflare)).
- **Filesystem** ‚Äî `createFilesystemBlobStorageAdapter` from
  `@syncular/server/filesystem`; content-addressed files under
  `basePath`, with a `tokenSigner` authorizing the direct upload/download
  routes. Good for single-node and self-hosted deployments.
- **Database** ‚Äî `storeBlobInDatabase` / `readBlobFromDatabase` from
  `@syncular/server`; bodies live in the `sync_blobs` table. Simplest to
  operate, best for small deployments or tests.

`createBlobCleanupScheduler` runs background cleanup of abandoned uploads;
`BLOB_CLEANUP_TUNING_PRESETS` provides starting points.

## Encryption

With blob encryption configured on the client, bodies are encrypted before
upload and decrypted after download; routes and adapters are unchanged. See
[Encryption](/features/field-encryption#blob-encryption).

## Watch-outs

- Do not put raw binary data in synced row JSON.
- Keep metadata visibility separate from authorization to fetch blob bodies ‚Äî
  `canAccessBlob` must enforce the latter.
- Test interrupted upload, retry, cache clear, and redownload paths.
- Size snapshot and blob storage independently; they use related storage
  primitives but have different lifecycle pressure.
- Deleting rows does not delete blob bodies; plan garbage collection for
  unreferenced hashes as a product decision.
