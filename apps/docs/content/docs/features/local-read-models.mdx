# Local Read Models

Local read models are generated SQLite projections for views that are
expensive to compute repeatedly from synced rows. Use them for dashboards,
counts, materialized summaries, and app-local indexes. They are declared in
the app contract, installed by codegen, and maintained by the runtime — app
code only reads them.

## Recommended shape

- Keep canonical data in synced app tables.
- Declare local-only tables or local read models in the app contract.
- Let codegen install setup SQL and rebuild SQL.
- Let the runtime refresh projections after row-delta events and rebuild them
  when schema versions change.
- Use read models for measured hotspots, not as the default for every query.

## Declaring read models

```ts
import { countByReadModel, defineSyncularClient } from '@syncular/typegen';

export const app = defineSyncularClient({
  // Plain local tables that never sync (preferences, drafts, caches).
  localOnlyTables: ['local_preferences'],
  localReadModels: [
    countByReadModel({
      name: 'taskCountsByUserCompletion',
      sourceTable: 'tasks',
      outputTable: 'syncular_task_counts',
      dimensions: ['user_id', 'completed'],
      countColumn: 'task_count', // defaults to 'row_count'
    }),
  ],
  tables,
});
```

`countByReadModel` is the currently supported kind: it materializes
`SELECT dimensions..., COUNT(*)` grouped by the listed dimension columns of
`sourceTable` into `outputTable`. The generated client installs the output
table and keeps it consistent as synced rows change, so a dashboard badge
reads one indexed row instead of re-aggregating the task table on every
render:

```ts
const counts = await syncular.db
  .selectFrom('syncular_task_counts')
  .selectAll()
  .where('user_id', '=', userId)
  .execute();
```

`localOnlyTables` is the related escape hatch for state that should live in
the same SQLite database but never sync. Both come from your normal SQL
migrations; the contract just tells codegen which tables are local.

## When to use one

Use a local read model when a benchmark or trace shows repeated aggregation,
search, or join work dominating user-visible latency. For normal list/detail
screens, a typed query builder over the local SQLite replica is usually
enough — SQLite is fast and the replica is local.

Signals that a read model is warranted:

- the same `COUNT`/`GROUP BY` runs on every navigation or row change;
- a denormalized summary is read by many components but derived from one
  table;
- live-query re-renders are dominated by recomputing an aggregate.

## Watch-outs

- Keep read models derived and rebuildable; they are projections, not data.
- Include schema-version rebuild behavior in tests — codegen emits rebuild
  SQL precisely so a new app version can re-derive projections from synced
  rows.
- Do not optimize for one full-partition bootstrap if the product contract is
  scoped access per user/project/team.
- Measure before and after; keep read models that materially reduce real app
  latency and delete the rest.
