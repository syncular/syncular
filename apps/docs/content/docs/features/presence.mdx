# Presence

Presence is ephemeral online state over the realtime connection. Use it for
viewers, editors, cursors, typing indicators, and lightweight collaboration
signals that should not become durable row history.

## Recommended shape

- Join presence after the client has subscribed and completed an initial sync.
- Use app-facing scope keys such as `user:user-1` or `project:project-1`.
- Keep metadata small and UI-specific.
- Leave presence when a screen, editor, or document closes.
- Treat presence as best-effort; durable state still belongs in synced rows.

## Binding examples

Presence is the same runtime concept across bindings. The exact host API
differs: browser and React expose TypeScript helpers, while native hosts expose
presence through generated Swift/Kotlin/Java wrappers and the native lifecycle
event stream.

### Browser TypeScript

```ts
await syncular.setSubscriptions([
  taskSubscription({ actorId: 'user-1' }),
]);
await syncular.sync();
await syncular.start();

const unsubscribe = syncular.presence.onChange((event) => {
  renderPresence(event.scopeKey, event.presence);
});

syncular.presence.join('user:user-1', {
  editingTaskId: taskId,
});

syncular.presence.updateMetadata('user:user-1', {
  editingTaskId: nextTaskId,
});

syncular.presence.leave('user:user-1');
unsubscribe();
```

### Native lifecycle

Native bindings expose presence handles and events through the generated
Swift/Kotlin/Java clients. Tie the handle to a screen or editor lifecycle so
closing or dropping it leaves presence automatically. Use
[Native Lifecycle](/docs/client/native#ui-lifecycle) for the host-specific
shape.

## Watch-outs

- Do not use presence as authorization. Server scopes authorize the realtime
  session before presence is accepted.
- Do not store long-lived user state in presence metadata.
- Keep cursor/selection detail in the editor layer unless other clients need it.
- Rejoin after realtime reconnect is runtime-owned; app code should react to
  presence events, not rebuild websocket loops.
