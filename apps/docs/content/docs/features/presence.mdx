# Realtime & Presence

This is the canonical page for Syncular's realtime behavior: what the
websocket connection actually delivers, how clients recover when it drops,
and how presence rides on top of it. Host-specific setup lives in
[JavaScript Realtime & Presence](/clients/javascript/realtime-presence) and
[Rust Realtime & Presence](/clients/rust/realtime-presence); server route
setup lives in [Realtime WebSocket](/server/realtime-websocket).

## What realtime is (and is not)

Realtime is an acceleration layer over the durable HTTP sync protocol. The
websocket delivers either binary sync-pack deltas or explicit "pull required"
wake-ups for the scopes a client is subscribed to. The commit log and the
combined `POST /` pull remain the source of truth: a client that never
connects a websocket still converges, just on its own schedule.

Consequences worth designing around:

- Losing the websocket loses freshness, not data.
- Anything delivered over realtime can also be recovered by an HTTP pull.
- Presence is the exception — it is ephemeral and never enters the commit log.

## WebSocket lifecycle

1. **Connect and authenticate.** The connection is authorized with the same
   actor boundary as sync routes; the server resolves the actor's effective
   scope keys and registers the connection against them.
2. **Hello.** The server sends a `hello` event carrying the session id, the
   connection's scope count, the latest cursor, and `requiresSync`. If the
   client's cursor is behind, the runtime schedules a reconnect catch-up pull
   instead of trusting the socket to replay history.
3. **Deltas and wake-ups.** While connected, changed scopes produce either
   binary sync-pack payloads (applied directly) or a sync message with a
   reason such as `server-wakeup`, `payload-too-large`, or `resync-required`
   that triggers an HTTP pull.
4. **Heartbeat.** The server emits heartbeats on an interval
   (`heartbeatIntervalMs` in the websocket route options) so dead connections
   are detected on both sides.
5. **Reconnect.** On close or error the runtime schedules reconnects with
   exponential backoff plus jitter (`initialReconnectDelayMs`,
   `maxReconnectDelayMs`, `reconnectBackoffFactor`, `reconnectJitterRatio`)
   and emits `realtime.reconnect_scheduled` diagnostics. App code should
   subscribe to connection state, not rebuild websocket loops.

Scope changes mid-session are handled server-side: when an actor's effective
scopes change, the server updates the connection's scope keys, so revocation
applies to live connections too.

## Fallback and polling

The managed lifecycle treats realtime as optional:

```ts
const lifecycle = new SyncularClientLifecycle(database.client, {
  realtime: {
    params: { token },
    initialReconnectDelayMs: 500,
    maxReconnectDelayMs: 5_000,
  },
  // Disable interval polling when realtime is on; or set a number (ms)
  // to poll in environments where websockets are blocked.
  pollIntervalMs: false,
});
```

- With realtime connected, scheduled polling is unnecessary — leave
  `pollIntervalMs: false` and let wake-ups drive pulls.
- Where websockets are unavailable (restrictive proxies, some embedded
  webviews), set `pollIntervalMs` to a sane interval; sync semantics are
  identical, only latency changes.
- `reconnect-catchup` pulls after every reconnect close the gap for changes
  that happened while disconnected.

For multi-instance servers and edge topologies, wake-up fan-out crosses
instances via the realtime broadcaster; see
[Realtime WebSocket](/server/realtime-websocket) and
[Edge Relay](/server/edge-relay).

## Presence

Presence is ephemeral online state over the realtime connection. Use it for
viewers, editors, cursors, typing indicators, and lightweight collaboration
signals that should not become durable row history.

### Recommended shape

- Join presence after the client has subscribed and completed an initial sync.
- Use app-facing scope keys such as `user:user-1` or `project:project-1`.
- Keep metadata small and UI-specific.
- Leave presence when a screen, editor, or document closes.
- Treat presence as best-effort; durable state still belongs in synced rows.

### Browser TypeScript

```ts
await syncular.setSubscriptions([taskSubscription({ actorId: 'user-1' })]);
await syncular.sync();
await syncular.start();

const unsubscribe = syncular.presence.onChange((event) => {
  renderPresence(event.scopeKey, event.presence);
});

syncular.presence.join('project:project-1', {
  editingTaskId: taskId,
});

syncular.presence.updateMetadata('project:project-1', {
  editingTaskId: nextTaskId,
});

syncular.presence.leave('project:project-1');
unsubscribe();
```

Each presence entry the server tracks per scope key carries `clientId`,
`actorId`, `joinedAt`, and the app-provided `metadata`. Presence sends are
skipped (with a `realtime.presence_not_connected` diagnostic) while the
socket is down; rejoin after reconnect is runtime-owned.

### Rust runtime

Rust apps expose presence handles and events through the native runtime. Tie
the handle to a screen or editor lifecycle so closing or dropping it leaves
presence automatically. Use [Runtime](/clients/rust/runtime#presence) for the
host-specific shape.

## Watch-outs

- Do not use presence as authorization. Server scopes authorize the realtime
  session before presence is accepted.
- Do not store long-lived user state in presence metadata.
- Keep cursor/selection detail in the editor layer unless other clients need
  it.
- Treat `resync-required` wake-ups as a forced-bootstrap boundary, not a
  normal pull — see [Error Handling](/features/error-handling).
- Test reconnect churn: backoff, catch-up pulls, and presence rejoin should
  all be observable through diagnostics, not guessed at.
