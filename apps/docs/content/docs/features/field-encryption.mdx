# Encryption

This is the canonical page for Syncular's three encryption surfaces: field
encryption, encrypted CRDT streams, and blob encryption. The Rust runtime owns
encryption and decryption on the client, so generated mutations, outbox rows,
blob uploads, CRDT updates, and retries stay coherent across JavaScript and
Rust hosts — and the server only ever sees ciphertext for protected values.

## What Syncular encrypts

| Surface | Declared in | Runtime config | Server storage |
|---|---|---|---|
| Row fields | `encryptedFields` in the app contract | `setFieldEncryption` | opaque values in normal table rows |
| CRDT updates | `syncMode: 'encrypted-update-log'` on a CRDT field | `setEncryptedCrdt` | `sync_crdt_updates` / `sync_crdt_checkpoints` system tables |
| Blob bodies | blob columns + encryption keys | `setBlobEncryption` | ciphertext bodies in blob storage |

All three use authenticated encryption with key ids (`kid`), so payloads can
be rotated to new keys without rewriting history.

## Declaring encrypted fields

Encrypted fields are part of the generated app contract. The TypeScript
contract is the authored source; codegen emits the encryption metadata that
the Rust runtime, JavaScript clients, and generated app code consume.

```ts
import { encryptedField, scope, syncedTable } from '@syncular/typegen';

tasks: syncedTable({
  table: 'tasks',
  subscriptionId: 'sub-tasks',
  serverVersion: 'server_version',
  scopes: [scope('user_id', { source: 'actorId', required: true })],
  encryptedFields: [
    // scope groups fields that share one key; rowIdField defaults to `id`.
    encryptedField('description', { scope: 'tasks' }),
  ],
});
```

## Configuring keys at runtime

Configure keys before applying encrypted mutations or processing encrypted
pulls. The browser/worker client takes structured configs; native hosts call
the generated JSON equivalents (`setFieldEncryptionJson`,
`setEncryptedCrdtJson`, `setBlobEncryptionJson`, `encryptionHelperJson`).

```ts
const key = await syncular.client.encryptionHelper('generateSymmetricKey', {});

await syncular.client.setFieldEncryption({
  rules: [
    {
      scope: 'tasks',
      table: 'tasks',
      fields: ['description'],
      rowIdField: 'id',
    },
  ],
  keys: { default: key },
  encryptionKid: 'default',
  // 'throw' (default) or 'keepCiphertext' for recovery UI.
  decryptionErrorMode: 'keepCiphertext',
});
```

The config shape (`SyncularFieldEncryptionConfig`):

- `rules` — which table/fields each key scope covers.
- `keys` — a map of key id → key material (`string` or `Uint8Array`).
- `encryptionKid` — which key new writes use; older `kid`s stay available for
  decryption, which is how rotation works.
- `decryptionErrorMode` — whether missing-key rows throw or surface ciphertext
  so the app can show a "locked" state.

Blob and CRDT encryption use the same `keys` + `encryptionKid` pattern:

```ts
await syncular.client.setBlobEncryption({
  keys: { default: key },
  encryptionKid: 'default',
});

await syncular.client.setEncryptedCrdt({
  keys: { default: key },
  encryptionKid: 'default',
});
```

## Key management patterns

The runtime exposes `encryptionHelper(method, args)` so apps never hand-roll
crypto. Available methods cover the common key lifecycles:

- **Generate and back up**: `generateSymmetricKey`, then `keyToMnemonic` /
  `mnemonicToKey` for a human-recoverable backup phrase.
- **Passphrase-derived keys**: `derivePassphraseKeyArgon2id` (preferred) or
  `deriveScopedPassphraseKeyPbkdf2` when the key must come from a user secret.
- **Sharing with other users/devices**: `generateKeypair`,
  `wrapKeyForRecipient`, and `unwrapKey` for asymmetric key handoff;
  `keyToShareUrl`, `publicKeyToShareUrl`, and `parseShareUrl` for link-based
  flows.

Store and rotate keys outside Syncular's synced database — a synced row that
contains its own decryption key defeats the model. Typical homes are the
platform keystore, a passphrase-derived key, or a wrapped key delivered by
your auth backend.

To rotate: add the new key under a new `kid`, switch `encryptionKid` to it,
and keep the old key in `keys` until old payloads are rewritten or expired.

## Encrypted CRDT streams

For collaborative fields that must stay end-to-end encrypted, declare the
CRDT field with `syncMode: 'encrypted-update-log'` in the app contract. The
server then never materializes the Yjs document; it stores opaque update and
checkpoint rows in the `sync_crdt_updates` and `sync_crdt_checkpoints` system
tables and fans them out by scope.

The server side is wired with `createEncryptedCrdtSystemHandlers` from
`@syncular/server`:

```ts
import { createEncryptedCrdtSystemHandlers } from '@syncular/server';

const crdtHandlers = createEncryptedCrdtSystemHandlers({
  scopePatterns: ['project:{project_id}'],
  resolveScopes: async (ctx) => ({
    project_id: await memberProjects(ctx.db, ctx.actorId),
  }),
  // Optional per-row authorization for updates and checkpoints.
  authorizeUpdate: ({ row }) => row.app_table === 'documents',
});
```

The handlers authorize and store ciphertext (`key_id` + `ciphertext` per
update), and `pruneEncryptedCrdtSystemRows` compacts covered updates once
checkpoints exist. Clients converge by exchanging encrypted updates and
decrypting locally with `setEncryptedCrdt` keys. See
[CRDT Fields](/features/crdt-fields) for the unencrypted `server-merge` mode
and the shared client APIs.

## Blob encryption

With `setBlobEncryption` configured, `syncular.blobs.store(...)` encrypts the
body before upload and `retrieve(...)` decrypts after download; the blob hash
references the stored ciphertext. Server blob routes and storage adapters are
unchanged — they already treat bodies as opaque bytes. Authorization to fetch
a blob URL stays a server concern either way; see
[File / Blob Sync](/features/blobs).

## Server boundary

The server should treat encrypted values as opaque payloads. Keep auth, scope
checks, conflict/version checks, audit metadata, and row lifecycle on the
server — none of those need plaintext. Do not make table handlers parse,
validate, or index encrypted field contents; anything searchable must be
either unencrypted by design or derived locally on the client after
decryption.

## Watch-outs

- Store and rotate keys outside Syncular's synced database.
- Decide upfront whether decryption failures throw
  (`decryptionErrorMode: 'throw'`) or keep ciphertext visible to recovery UI.
- Test key rotation, new-device key sharing, offline writes, conflict
  handling, and encrypted CRDT convergence.
- Encrypted fields cannot drive server-side filtering, scope extraction, or
  read models — keep scope columns and query keys unencrypted.
