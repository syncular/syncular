# Testing Strategy

This page is the what-to-verify-before-shipping guide. It walks the four test
layers bottom-up, each with a runnable example built on
[`@syncular/testkit`](/clients/javascript/testing). Rust/native users get the
same layers from the Rust testkit â€” see
[Rust testkit pointers](#rust-and-native-clients) below.

The principle behind all four layers: **run real pieces, mock only your own
externals**. The testkit spins up real sync routes, real SQLite/Postgres
storage, and real protocol payloads in-process, so there is no reason to mock
Syncular system tables, outbox semantics, cursor movement, or event streams.

```bash
bun add -d @syncular/testkit
```

## Layer 1 â€” handler tests

**What it proves:** your server table handlers apply writes, enforce scope
authorization, and reject what they should â€” before any client code exists.

`createHttpServerFixture` boots real `@syncular/server/hono` routes against an
in-memory database (`serverDialect: 'sqlite'` for SQLite semantics, `'pglite'`
for Postgres semantics). The protocol builders construct the same combined
`POST /` request a real client sends:

```ts
import { describe, expect, it } from 'bun:test';
import {
  createHttpServerFixture,
  createProjectScopedTasksHandler,
  createSyncCombinedRequest,
  createSyncSubscription,
  createSyncUpsertOperation,
  ensureProjectScopedTasksTable,
  postSyncCombinedRequest,
} from '@syncular/testkit';

describe('tasks handler', () => {
  it('applies writes and revokes foreign-scope subscriptions', async () => {
    const server = await createHttpServerFixture({
      serverDialect: 'sqlite',
      // Swap in your own DDL + handler here:
      createTables: ensureProjectScopedTasksTable,
      handlers: [createProjectScopedTasksHandler()],
      authenticate: async (c) => {
        const actorId = c.req.header('x-actor-id');
        return actorId ? { actorId } : null;
      },
    });

    try {
      // Push a row and pull it back in one round-trip.
      const { json } = await postSyncCombinedRequest({
        fetch,
        url: `${server.baseUrl}/sync`,
        actorId: 'user-a',
        body: createSyncCombinedRequest({
          clientId: 'client-a',
          push: {
            commits: [
              {
                clientCommitId: 'commit-1',
                schemaVersion: 1,
                operations: [
                  createSyncUpsertOperation({
                    table: 'tasks',
                    rowId: 'task-1',
                    payload: { title: 'Ship it', completed: 0, project_id: 'p0' },
                  }),
                ],
              },
            ],
          },
          pull: {
            schemaVersion: 1,
            limitCommits: 50,
            subscriptions: [
              createSyncSubscription({
                id: 'tasks-p0',
                table: 'tasks',
                scopes: { user_id: 'user-a', project_id: 'p0' },
              }),
            ],
          },
        }),
      });

      expect(json.push?.commits[0]?.status).toBe('applied');
      expect(json.pull?.subscriptions[0]?.status).toBe('active');

      // The row really landed in the server database.
      const row = await server.db
        .selectFrom('tasks')
        .select(['id', 'user_id'])
        .where('id', '=', 'task-1')
        .executeTakeFirstOrThrow();
      expect(row.user_id).toBe('user-a');

      // Another actor requesting user-a's scope gets revoked, not data.
      const foreign = await postSyncCombinedRequest({
        fetch,
        url: `${server.baseUrl}/sync`,
        actorId: 'user-b',
        body: createSyncCombinedRequest({
          clientId: 'client-b',
          pull: {
            schemaVersion: 1,
            limitCommits: 50,
            subscriptions: [
              createSyncSubscription({
                id: 'steal-tasks',
                table: 'tasks',
                scopes: { user_id: 'user-a', project_id: 'p0' },
              }),
            ],
          },
        }),
      });
      expect(foreign.json.pull?.subscriptions[0]?.status).toBe('revoked');
    } finally {
      await server.destroy();
    }
  });
});
```

Write one of these per synced table: the happy-path apply, the scope-revocation
case, and any handler-specific rejection (`status: 'error'` with your error
code). See [Table Handlers](/server/table-handlers) for the result shapes your
`applyOperation` should return.

For campaign/project-style apps, use
`createProjectScopedTestActor(...)`, `createProjectMembership(...)`, and
`createProjectScopedTasksHandler({ projectsByActor })` from the testkit so one
actor has an allowed project and another project is explicitly denied. The same
fixture then proves allowed writes, `sync.forbidden` writes, and revoked
foreign subscriptions without inventing app-specific auth boilerplate.

## Layer 2 â€” two-client convergence

**What it proves:** a write on one device shows up on another, through the real
client runtime (WASM SQLite, outbox, cursor tracking) and the real HTTP routes.

Use your generated app client against the same fixture. The pattern below
mirrors the framework's own conformance tests:

```ts
import {
  createSyncularAppDatabase,
  taskSubscription,
} from './src/generated/syncular.generated';

async function openClient(baseUrl: string, actorId: string, clientId: string) {
  return createSyncularAppDatabase({
    config: {
      baseUrl: `${baseUrl}/sync`,
      actorId,
      clientId,
      storage: 'memory', // real SQLite, no persistence between tests
      clearOnInit: true,
    },
    getHeaders: () => ({ 'x-actor-id': actorId }),
    runtime: 'core',
    subscriptions: [taskSubscription({ actorId })],
  });
}

const writer = await openClient(server.baseUrl, 'user-a', 'device-a');
const reader = await openClient(server.baseUrl, 'user-a', 'device-b');

try {
  await writer.mutations.tasks.insert({
    id: 'task-1',
    title: 'Converge me',
    user_id: 'user-a',
  });

  await writer.client.syncPush();
  await reader.client.syncPull();

  const rows = await reader.db
    .selectFrom('tasks')
    .select(['id', 'title'])
    .execute();
  expect(rows).toEqual([{ id: 'task-1', title: 'Converge me' }]);
} finally {
  await writer.close();
  await reader.close();
}
```

Pick the runtime artifact that matches your schema's features: `'core'` is the
smallest; CRDT fields or blobs need an artifact built with those features (the
client fails loudly at open time if the artifact is missing a required
feature). For hand-rolled (non-generated) schemas, `createSyncularDatabase`
from `@syncular/client` takes the same `config` plus explicit
`appTables`/`tableConfig`.

Write a convergence test for every table your product actually depends on â€”
this is the layer that catches scope-mapping mistakes where data syncs to the
server but never reaches the second device.

## Layer 3 â€” offline queue and reconnect

**What it proves:** mutations made while offline are queued durably and replay
when connectivity returns, and the outbox drains to zero.

Mutations are local-first: they commit to local SQLite and the outbox
immediately, with no server round-trip. So the offline half of this test is
just "mutate without syncing", and the reconnect half is "now push":

```ts
const client = await openClient(server.baseUrl, 'user-a', 'device-offline');

try {
  // "Offline": mutate locally, never push.
  await client.mutations.tasks.insert({
    id: 'task-offline',
    title: 'Written underground',
    user_id: 'user-a',
  });

  const before = await client.client.diagnosticSnapshot();
  expect(before.outboxStats?.pending).toBeGreaterThan(0);

  // "Reconnect": replay the outbox.
  await client.client.syncPush();

  const after = await client.client.diagnosticSnapshot();
  expect(after.outboxStats?.pending).toBe(0);

  // The write reached the server.
  const row = await server.db
    .selectFrom('tasks')
    .select('id')
    .where('id', '=', 'task-offline')
    .executeTakeFirst();
  expect(row).toBeDefined();
} finally {
  await client.close();
}
```

For harsher network conditions at the protocol level â€” pushes that fail mid
sequence, latency, scripted fault plans â€” wrap a transport with `withFaults`
or use `createMockTransport` and `withRecording` from the testkit. Push
delivery is idempotent per `clientCommitId`, so a commit that fails after the
server applied it must not double-apply on retry; that is exactly the property
fault plans are for.

## Layer 4 â€” conflict resolution

**What it proves:** concurrent edits to the same row are detected (not silently
overwritten), and your product's resolution path works.

Conflict detection runs on `base_version`: a push whose `baseVersion` no longer
matches the server's version column comes back with `status: 'conflict'`, the
current `server_version`, and the winning `server_row`:

```ts
import {
  createSyncCombinedRequest,
  createSyncUpsertOperation,
  postSyncCombinedRequest,
} from '@syncular/testkit';

// task-1 exists at server_version 1. A stale writer pushes against version 1
// after someone else already bumped the row to version 2.
const { json } = await postSyncCombinedRequest({
  fetch,
  url: `${server.baseUrl}/sync`,
  actorId: 'user-a',
  body: createSyncCombinedRequest({
    clientId: 'stale-client',
    push: {
      commits: [
        {
          clientCommitId: 'stale-commit',
          schemaVersion: 1,
          operations: [
            createSyncUpsertOperation({
              table: 'tasks',
              rowId: 'task-1',
              payload: { title: 'Stale edit', completed: 0, project_id: 'p0' },
              baseVersion: 1, // server is already at 2
            }),
          ],
        },
      ],
    },
  }),
});

const commit = json.push?.commits[0];
expect(commit?.status).toBe('rejected');
const result = commit?.results[0];
expect(result?.status).toBe('conflict');
if (result?.status === 'conflict') {
  expect(result.server_version).toBe(2);
  // result.server_row carries the winning row for client-side resolution.
}
```

A rejected commit rolls back atomically â€” no partial application. On the
client side, generated mutations send `base_version` automatically and surface
conflicts through the conflict store (`useConflictStats` in React, conflict
summaries + resolution commands on the client). Test the resolution flow your
UI actually exposes: keep-local retry, accept-server, or field merge. The
mechanism is documented in
[Conflict Resolution](/learn/conflict-resolution) and the server-side batch
semantics in [Push & Conflicts](/server/apply-push).

## Minimum suite before production

The smallest suite worth shipping behind:

- [ ] **L1** handler tests for every synced table: apply, scope revocation,
      handler-specific rejections.
- [ ] **L1** an authorization test proving actor B cannot read or write actor
      A's scopes.
- [ ] **L1** for project/campaign apps: explicit membership with one allowed
      scope and one denied scope that returns `sync.forbidden` or
      `status: "revoked"`.
- [ ] **L2** two-client convergence for every table your product depends on.
- [ ] **L3** offline queue + replay for every mutation users can make offline,
      asserting the outbox drains.
- [ ] **L4** a conflict test for every user-visible resolution path.
- [ ] Realtime recovery if your UI depends on WebSocket wake-ups: kill the
      socket, assert HTTP pull still converges (helpers in
      `openRealtimeWs`/`waitForWsJsonMessage` from the testkit).
- [ ] Blob, CRDT, and encryption round-trips for each capability you enabled.
- [ ] Diagnostic marker assertions for lifecycle-sensitive failures:
      `sync.scope_revoked`, `realtime.hello`, `realtime.sync_wakeup`,
      `realtime.pull_required`, `blob.forbidden`, `blob.signing_failed`, and
      `sync.local_visibility_timeout`.
- [ ] A schema-version test for the oldest deployed app version you still
      support (old `schemaVersion` pull against the current server).

Run the suite against `serverDialect: 'pglite'` at least once in CI if you
deploy on Postgres â€” SQLite and Postgres fixtures share the same API, so this
is usually a one-line parameterization.

## What not to mock

Do not mock Syncular system tables, outbox semantics, cursor movement, CRDT
storage, blob queues, or event streams in tests that claim to verify sync
behavior. The fixtures above are fast enough (in-process, in-memory) that
mocking buys nothing and hides exactly the bugs these layers exist to catch.
Mock only your app's own external services and UI shell.

## Rust and native clients

The Rust testkit (`rust/crates/testkit`, crate `syncular-testkit`) covers the
same layers with real Diesel SQLite stores, generated app schemas, and real
transports: app fixtures (`app`, `app_server`), HTTP harnesses (`http`),
deterministic clocks/IDs (`deterministic`), CRDT and file-asset helpers, and
shared conformance scenarios (`conformance`).

```bash
cargo test --manifest-path rust/Cargo.toml -p syncular-testkit
```

Start at [Rust Testing](/clients/rust/testing) for the fixture-by-fixture
guide. Native (Swift/Kotlin) app shells are exercised by the conformance
gates' native lanes â€” see [Conformance](/testing/conformance).
