# Testing

Testing is split by language/runtime so app tests do not accidentally mock the
wrong layer.

  <Card title="JavaScript Testing" href="/docs/javascript/testing" description="@syncular/testkit for TypeScript, server, browser, HTTP, and runtime tests" />
  <Card title="Rust Testkit" href="/docs/rust/testing" description="syncular-testkit for generated Rust/native clients and real SQLite state" />
  <Card title="Conformance" href="/docs/testing/conformance" description="Shared scenarios that should hold across JavaScript, Rust, and host runtimes" />
  <Card title="Testing Strategy" href="/docs/testing/strategy" description="What to verify before production and how to layer the suites" />

## Use the right surface

- Use `@syncular/testkit` when the test opens TypeScript clients, server
  handlers, HTTP routes, websocket helpers, or JavaScript runtime fixtures.
- Use `syncular-testkit` when the test opens generated Rust/native clients,
  validates native worker events, or needs Rust-side app-server fixtures.
- Use conformance scenarios when behavior must match across language bindings.

The goal is the same in both languages: real local SQLite behavior, real sync
semantics, deterministic fixtures, and app-owned domain assertions.
