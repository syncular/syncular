wrong layer.

## Use the right surface

| Test need | Use |
|---|---|
| TypeScript client, server handler, HTTP route, browser runtime, or websocket helper tests | [`@syncular/testkit`](/clients/javascript/testing) |
| Generated Rust client, native worker, real SQLite fixture, event stream, CRDT/blob queue tests | [`syncular-testkit`](/clients/rust/testing) |
| Behavior that must match JavaScript, Rust, and native hosts | [Conformance](/testing/conformance) |
| Server-only handler logic, auth, scopes, and schema-version projection | [Server](/server) plus the JavaScript testkit server fixtures |
| Product-specific rows, fixtures, and assertions | App-owned test helpers layered above the Syncular testkit |

The goal is the same in both languages: real local SQLite behavior, real sync
semantics, deterministic fixtures, and app-owned domain assertions.

## Minimum suite before shipping

The layered walkthrough — handler tests, two-client convergence,
offline/reconnect, conflicts — plus the full pre-production checklist lives in
[Testing Strategy](/testing/strategy).

## What not to mock

Do not mock Syncular system tables, outbox semantics, cursor movement, CRDT
storage, blob queues, or event streams in product tests that claim to verify
sync behavior. Use real fixtures, then mock only your app's external services
or UI shell.
