# Conformance

Syncular's runtimes (Rust native, browser WASM, the generated JavaScript and
Rust clients, and the native Swift/Kotlin shells) are held to a shared behavior
contract by a set of **conformance gates** that run in this repository's CI.
This page explains what those gates actually execute and — more usefully —
what that means you do *not* have to test yourself.

## What the gates run

The gates live in `rust/scripts/run-conformance-gates.sh` and are exposed as
repo scripts. They run in lanes, smallest first:

| Lane | Command | What it executes |
|---|---|---|
| Fast | `bun run rust:conformance:fast` | `syncular-testkit` crate tests, the Rust runtime's protocol/blob/CRDT contract tests, the generated Rust todo-app example, and the browser generated-app contract test |
| Browser/Hono | `bun run rust:conformance` | Fast lane plus browser WASM tests against real `@syncular/server/hono` routes: auth, sync, realtime, and blob conformance |
| Native | `bun run rust:conformance:native` | Swift/Kotlin/JVM native smoke suites against the generated app |
| App shell | `bash rust/scripts/run-conformance-gates.sh --native-app-shell` | Real iOS and Android app-shell lifecycle smokes (needs simulator/emulator toolchains) |

All lanes exercise the **same shared todo-app fixture**: one generated app
contract, pushed through every runtime and binding. The full gate-selection
policy (which lane must pass for which kind of change) is documented in
`rust/docs/QUALITY_GATES.md`.

## The behavior contract

Concretely, the gates prove these scenarios hold identically across
JavaScript, Rust, and native hosts:

- bootstrap from server state into a fresh local database (inline rows,
  snapshot chunks, and scoped artifacts)
- a local mutation writes the row, an outbox commit, and row/field events
- push then pull converges two clients
- offline mutations replay after reconnect; push is idempotent per
  `clientCommitId`
- a stale `base_version` push produces a conflict with the winning server row,
  and resolution commands work
- realtime WebSocket messages wake sync, while HTTP catch-up stays
  authoritative
- blob upload/download/cache behavior survives retries
- CRDT field updates converge after duplicate and reordered delivery
- encrypted field/CRDT persistence never leaks plaintext to the wrong layer
- old-schema-version clients receive correctly projected rows from a newer
  server

## What this means for your tests

Because the framework re-proves the contract above on every change:

- **You don't need to re-test sync mechanics.** Outbox replay ordering, cursor
  movement, chunked bootstrap decoding, CRDT merge math, wire-format
  encode/decode — these are covered. App tests that re-assert them add run
  time, not confidence.
- **You do need to test your app contract.** The gates run against the todo
  fixture, not your schema. Your handlers' scope rules, your conflict
  resolution UX, your table-specific rejections, and your product flows are
  yours to cover — that is the [Testing Strategy](/testing/strategy) layer
  guide.
- **Cross-language consistency is inherited.** If your app ships a generated
  client on both JavaScript and Rust from the same `syncular.app.ts`, the
  runtimes underneath are the ones the gates already hold to the same
  behavior. You only need one set of app-level tests per behavior, not one per
  language — pick the language where the behavior lives.

## Running the gates yourself

Useful when you vendor or patch the framework, or want a pre-upgrade check:

```bash
# Fastest meaningful signal (Rust testkit + runtime contract + generated apps)
bun run rust:conformance:fast

# Production-shaped browser/Hono sync proof
bun run rust:conformance

# Native binding proof (needs local toolchains)
bun run rust:conformance:native
```

For your own app suites, follow the same philosophy the gates use: real
storage, real protocol payloads, real event streams — no app-wide mocks. The
JavaScript entry point is [`@syncular/testkit`](/clients/javascript/testing),
the Rust entry point is [`syncular-testkit`](/clients/rust/testing).
