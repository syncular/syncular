# API Reference

The Syncular server exposes two API families:

- `sync` endpoints for client synchronization, blobs, and product-facing audit history
- `console` endpoints for operator dashboards, forensics, and maintenance workflows

All paths below are shown as they appear in the OpenAPI docs (i.e. mounted at
`/sync` and `/console/*`). In your app they live wherever you mount the
routes â€” the docs examples assume `/api/sync` and `/api/console`.

## Semantics that apply across endpoints

### One combined sync call

There are no separate push/pull endpoints. [`POST /sync`](/reference/api/postSync)
carries an optional `push` and an optional `pull` section in one JSON body;
push is processed first, then pull, so a client sees its own writes in the
same round-trip.

### Binary responses (SSP1)

`POST /sync` accepts JSON but always responds with a binary sync pack
(content type `application/vnd.syncular.sync-pack.v1`). Generated clients
decode this transparently; if you are probing with curl, expect bytes
starting with the `SSP1` magic, not JSON. Errors are returned as plain JSON.
See [Sync Protocol](/reference/protocol) for the encoding.

### Push idempotency

Every pushed commit carries a client-generated `clientCommitId`. Replaying a
commit (after a timeout or crash) is safe: the server detects the id and
returns the cached result with commit status `cached` instead of applying it
twice. Commit statuses are `applied`, `cached`, or `rejected`; rejection
rolls back the whole commit atomically.

### Cursor model

Each pull subscription tracks a `cursor` â€” the last server commit sequence
the client has applied. The server returns commits after the cursor plus a
`nextCursor`. A subscription with no usable cursor (or after revocation)
goes through bootstrap: snapshot pages with inline rows, chunk refs, or
artifact refs, resumable via the opaque `bootstrapState` the client echoes
back unchanged.

### Snapshot chunk downloads require scopes

`GET /sync/snapshot-chunks/{chunkId}` and
`GET /sync/snapshot-artifacts/{artifactId}` re-authorize on every request:
pass the subscription's scope values as a JSON-encoded `scopes` query
parameter (or the `x-syncular-snapshot-scopes` header). Without them the
server returns `400 sync.invalid_request`; with scopes that fail
authorization it returns `403 sync.forbidden`.

### Error shape

All error responses use one JSON envelope with a stable `code`, `category`,
`retryable`, and `recommendedAction` â€” see [Error Codes](/reference/errors)
for the full catalog:

```json
{
  "error": "sync.auth_required",
  "code": "sync.auth_required",
  "message": "Authentication is required.",
  "category": "auth-required",
  "retryable": true,
  "recommendedAction": "refreshAuth"
}
```

### Authentication

Syncular does not prescribe an authentication mechanism. The server calls
your `authenticate()` function with the raw request and uses whatever you
decide (Bearer tokens, cookies, session headers). Console routes use a
dedicated bearer token (`console.token` / `SYNC_CONSOLE_TOKEN`).

## Endpoints by feature

### Sync

| Method | Path | Description |
|--------|------|-------------|
| `POST` | [`/sync`](/reference/api/postSync) | Combined sync (`push` and/or `pull` in one request) |
| `GET` | `/sync/realtime` | WebSocket realtime connection for verified deltas and pull-required wake-ups (not part of OpenAPI) |
| `GET` | [`/sync/snapshot-chunks/{chunkId}`](/reference/api/getSyncSnapshotChunksByChunkId) | Download a gzip `binary-table-v1` bootstrap chunk (requires `scopes`) |
| `GET` | [`/sync/snapshot-artifacts/{artifactId}`](/reference/api/getSyncSnapshotArtifactsByArtifactId) | Download a verified scoped bootstrap snapshot artifact (requires `scopes`) |

### Audit (product-facing history)

| Method | Path | Description |
|--------|------|-------------|
| `GET` | [`/sync/audit/commits`](/reference/api/getSyncAuditCommits) | List commit audit history (auth/partition scoped, cursor-paginated) |
| `GET` | [`/sync/audit/commits/{commitSeq}`](/reference/api/getSyncAuditCommitsByCommitSeq) | One commit with operation-level changes (payloads redacted by default) |
| `GET` | [`/sync/audit/rows/{table}/{rowId}`](/reference/api/getSyncAuditRowsByTableByRowId) | Scoped row audit history |
| `GET` | [`/sync/audit/debug-export`](/reference/api/getSyncAuditDebugExport) | Export a redacted sync debug bundle |

### Blobs

| Method | Path | Description |
|--------|------|-------------|
| `POST` | [`/sync/blobs/upload`](/reference/api/postSyncBlobsUpload) | Initiate a content-addressed blob upload |
| `POST` | [`/sync/blobs/{hash}/complete`](/reference/api/postSyncBlobsByHashComplete) | Mark a blob upload complete (hash/size verified) |
| `GET` | [`/sync/blobs/{hash}/url`](/reference/api/getSyncBlobsByHashUrl) | Get a signed download URL for a blob |

### Console â€” read model and forensics

Stats ([`/console/stats`](/reference/api/getConsoleStats),
[latency](/reference/api/getConsoleStatsLatency),
[timeseries](/reference/api/getConsoleStatsTimeseries)), commit browsing
([`/console/commits`](/reference/api/getConsoleCommits)), client inventory
([`/console/clients`](/reference/api/getConsoleClients)) and Rust client
diagnostics ([`/console/client-diagnostics*`](/reference/api/getConsoleClientDiagnostics)),
including browser preview
([`POST /console/client-diagnostics/browser-preview-failure`](/reference/api/postConsoleClientDiagnosticsBrowserPreviewFailure))
and Cloudflare runtime
([`POST /console/client-diagnostics/cloudflare-runtime-failure`](/reference/api/postConsoleClientDiagnosticsCloudflareRuntimeFailure))
failure artifact ingestion,
handlers ([`/console/handlers`](/reference/api/getConsoleHandlers)), the
activity [timeline](/reference/api/getConsoleTimeline), row
[history](/reference/api/getConsoleRowHistoryByTableByRowId) and
[visibility investigation](/reference/api/getConsoleRowInvestigationByTableByRowId),
and a redacted [debug export](/reference/api/getConsoleDebugExport).

### Console â€” events and storage

Request events ([`/console/events*`](/reference/api/getConsoleEvents),
including stored request/response [payload snapshots](/reference/api/getConsoleEventsByIdPayload)
when `requestPayloadSnapshots` is enabled) and optional object-storage
browsing ([`/console/storage*`](/reference/api/getConsoleStorage), requires
`console.blobBucket`).

### Console â€” operator auth

API key lifecycle: [list/create](/reference/api/getConsoleApiKeys),
[rotate](/reference/api/postConsoleApiKeysByIdRotate),
[staged rotation](/reference/api/postConsoleApiKeysByIdRotateStage), and
[bulk revoke](/reference/api/postConsoleApiKeysBulkRevoke).

### Console â€” maintenance

[Operations log](/reference/api/getConsoleOperations), production
[ops readiness](/reference/api/getConsoleOpsReadiness),
[readiness trends](/reference/api/getConsoleOpsReadinessTrends), and ingestion
([`POST /console/ops/readiness`](/reference/api/postConsoleOpsReadiness)),
[notify-data-change](/reference/api/postConsoleNotifyDataChange) (wake
clients after out-of-band writes),
[prune preview](/reference/api/postConsolePrunePreview) /
[prune](/reference/api/postConsolePrune), and
[compact](/reference/api/postConsoleCompact).

## OpenAPI specification

The per-endpoint pages in this section are generated from the OpenAPI spec at
`packages/server/openapi.json` (regenerate with
`bun --cwd apps/docs generate:openapi`; do not edit the generated pages by
hand). To serve the spec from your own app:

```ts
import { createOpenAPIHandler } from '@syncular/server/hono';

app.get('/openapi.json', createOpenAPIHandler(app, { title: 'Syncular API' }));
```

For a paired Scalar reference page at `/spec`:

```ts
import { createOpenAPIDocsRoutes } from '@syncular/server/hono';

app.route('/', createOpenAPIDocsRoutes(app, { title: 'Syncular API' }));
```
