# Server Overview

This page describes the server-side surfaces that make up a Syncular deployment.

Use [Server Setup](/docs/build/server-setup) for first-time implementation. Use this section when you need to know which package, route, or extension point owns a piece of server behavior.

## Package map

| Package | Responsibility |
|---|---|
| `@syncular/server` | Core sync engine, schema helpers, handler factory, pruning/compaction primitives, blob manager |
| `@syncular/server-hono` | Hono route factories, realtime wiring, console routes, OpenAPI handler |
| `@syncular/server-dialect-*` | Dialect-specific schema and SQL behavior |

## Server lifecycle

1. Your app authenticates a request and produces an actor context.
2. Table handlers resolve allowed scopes for that actor.
3. Sync routes read snapshots or apply operations through handlers.
4. The server records accepted changes in the sync tables.
5. Optional realtime infrastructure wakes subscribed clients so they pull again.

This split is intentional:

- guides in [Build](/docs/build) explain how to wire the server
- pages in [Reference](/docs/reference/server) document the server API surface and extension points

## Sync Tables

Running migrations creates these tables:

| Table | Purpose |
|-------|---------|
| `sync_commits` | Commit log with idempotency keys |
| `sync_changes` | Individual changes within commits (with JSONB scopes) |
| `sync_client_cursors` | Per-client cursor tracking |
| `sync_snapshot_chunks` | Large bootstrap chunk storage |

## Mounted surfaces

These routes are relative to where you mount them.

| Surface | Typical mount | Purpose |
|---|---|---|
| Sync route | `/api/sync` | Combined push/pull entrypoint |
| Snapshot chunks | `/api/sync/snapshot-chunks/:chunkId` | Download paged bootstrap chunks |
| Realtime | `/api/sync/realtime` | WebSocket wake-up channel |
| Blob routes | `/api/sync/blobs/*` | Binary upload/download flow |
| Console routes | `/api/console/*` | Operator dashboards, forensics, and maintenance |

## Core extension points

### Table handlers

Table handlers define what syncs, who can access it, and how rows are read or applied.

- Use [Table Handlers](/docs/build/table-handlers) for patterns and examples.
- Use [Table Handlers Reference](/docs/reference/server/table-handlers-reference) for factory options and interfaces.

### Subscription registry

Subscriptions describe the requested data shape, and the registry resolves them into the effective scopes that can actually be read.

See [Subscription Registry](/docs/reference/server/subscription-registry).

### Dialects

Dialects encapsulate database-specific behavior for schema setup and sync operations.

See [Dialects](/docs/reference/server/dialects).

### Realtime bridge

Realtime is optional. It does not transport row data; it wakes clients so they perform another HTTP pull.

See [Realtime Bridge](/docs/reference/server/realtime-bridge) and the [Realtime guide](/docs/build/realtime).

### Blob manager

Blob support is separate from core row sync. The server tracks blob metadata and issues authenticated upload/download flows through dedicated routes.

See [Blob Storage](/docs/build/blob-storage).
