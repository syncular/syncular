# Sync Protocol (SSP1)

This page orients implementers of bindings, proxies, and load tooling. It is
**not a byte-level specification** — the normative encoder/decoder is
`packages/core/src/sync-packs.ts` (TypeScript) and
`rust/crates/protocol/src/binary_sync_pack.rs` (Rust). A dependency-free
reference reader that documents exactly what is parseable without
decompression lives in `tests/load/lib/ssp1.js`.

## Where it appears

`POST /` (the combined sync route) accepts a **JSON request** and always
returns a **binary sync pack** with content type
`application/vnd.syncular.sync-pack.v1`. Errors are plain JSON with the
[error envelope](/reference/errors). Realtime websocket deltas use the same
pack encoding for payloads up to `maxSyncPackBytes`; larger updates degrade
to a pull-required wakeup.

## Envelope

`binary-sync-pack-v1` removes the repeated JSON field-name overhead of
combined sync responses: protocol metadata is encoded positionally,
little-endian throughout. The envelope itself is **not compressed** — only
row bodies inside it are.

```text
"SSP1" magic (4 bytes)
u16 wire version        (currently 14; other versions are rejected)
u16 flags               (must be 0)
bool ok
optional i32 requiredSchemaVersion
optional i32 latestSchemaVersion
optional push section
optional pull section
```

Strings are length-prefixed UTF-8; lists are count-prefixed; optionals are
presence-byte prefixed. Per-pack table and scope dictionaries deduplicate
repeated names.

The wire version is strict: decoders reject any version other than the
current one (14). There is no multi-version negotiation — client and server
ship in lockstep, and schema-version gating happens above the wire layer via
`requiredSchemaVersion` / `latestSchemaVersion`.

## Push section

Mirrors `SyncPushBatchResponse`: `ok` plus a list of per-commit records —
`ok`, `clientCommitId` (the idempotency key the client chose), a status byte
(`1 = applied`, `2 = cached`, `3 = rejected`), an optional `commitSeq`, and
per-operation results including conflict payloads (`server_version`,
`server_row`) and error codes.

## Pull section

A list of per-subscription records:

- `id`, `status` (`active`/`revoked`), `scopes`, `bootstrap` flag
- `bootstrapState` — opaque JSON that round-trips into the next request body
  unchanged (partial-bootstrap resume)
- `nextCursor` and subscription-level integrity roots (since v13 integrity
  lives here, not on every commit)
- `commits[]` with full per-change metadata: table, `row_id`, op,
  `row_version`, scopes
- `snapshots[]`: inline JSON rows, chunk refs, page flags,
  `bootstrapStateAfter` checkpoints, a manifest for chunk
  ordering/integrity, and (since v14) scoped artifact refs

## Row groups (SBT1)

Incremental change bodies for the same table can be grouped into generated
binary table payloads ("SBT1" magic) instead of inline JSON, when the server
has a generated encoder for that table and the commit carries at least 8
rows. These frames are gzip-backed and length-prefixed, so a reader that
doesn't want to decompress can skip them: the affected changes keep
`row_json: null` plus a row-ref marker, while their `row_id`/op/scopes
metadata stays fully readable. Decoding row contents requires the generated
column metadata (`snapshotBinaryColumns`) shipped with the client.

## Snapshot chunks and artifacts

Large bootstraps are delivered as **refs**, not inline rows:

- **Chunks** — `GET /snapshot-chunks/:chunkId`, gzip-compressed
  `binary-table-v1` row pages. The request must carry the subscription's
  scope values (`?scopes=` JSON or the `x-syncular-snapshot-scopes` header);
  the server re-authorizes them and returns 400 without them. Refs carry
  `id`, `byteLength`, and `sha256` for verification.
- **Artifacts** — `GET /snapshot-artifacts/:artifactId`, gzip-compressed
  scoped SQLite artifact bodies for direct-apply bootstraps.

## What load tooling can and cannot read

The k6 reader (`tests/load/lib/ssp1.js`) parses the entire envelope — push
results, subscription metadata, commit/change metadata, chunk refs and
manifests — without zlib. It cannot read row bodies inside gzip row groups
or external chunks; row-id-level convergence tracking is unaffected.
