Here's what nobody tells you about offline-first: the reads are easy. Stick data in SQLite, query it locally, ship it. Done. The hard part — the part that will eat your weekends — is writes. What happens when two devices edit the same row while one is on a plane? What happens when a push fails halfway through? What happens when your user closes the app mid-sync and reopens it three days later?

This isn't a new obsession. Back in 2019 I built [debe](https://github.com/bkniffler/debe) — a reactive, offline-first datastore with CRDT-based sync, multi-master replication, adapters for SQLite, Postgres, and in-memory stores. It never reached production-ready, but I learned where the real complexity hides. Seven years later, the landscape has exploded with options, but the core problems haven't changed.

I spent the better part of a year evaluating the current generation of sync tools. I built prototypes with each, hit limits, wrote glue code, and eventually realized the glue code was the actual product I needed. So I built it.

This post walks through what I tried, what I learned along the way, and how those lessons led to [Syncular](https://syncular.dev). Even if you never use Syncular, the patterns and tradeoffs here apply to anyone building sync.

## Why offline writes are the real problem

Most sync frameworks start with the read path: how do we get server data to the client fast? That's a well-understood problem. Stream shapes from Postgres, cache queries locally, use reactive subscriptions. There are good tools for all of this.

But the moment you need durable offline writes, things get ugly fast:

- **Durability across restarts.** The user typed something, closed the app, and opened it a week later. Is it still there?
- **Ordering and idempotency.** The push failed, so we retry. Does the server apply it twice? Does the order change?
- **Conflict detection.** Two users edited the same task offline. Who wins? Can we even detect this happened?
- **Authorization that holds offline.** The user was removed from a project while offline. Their pending writes to that project land on the server. Now what?
- **Debugging.** Something went wrong. A row is missing on device B. Where do you even start looking?

These aren't edge cases. If you're building for construction sites, trains, or anywhere with spotty signal, they're your Tuesday.

How hard is this? Hard enough that the Rocicorp team — whose entire expertise is sync — [deliberately disabled offline writes in Zero](https://news.ycombinator.com/item?id=43397640). Aaron Boodman explained on HN that general-purpose conflict resolution for long-term offline writes is an "irreducible complexity," that the "ideal offline-first write/write UI has never been built," and that they wanted to teach developers to handle conflicts correctly before enabling it. When the people who built Replicache say "we're not ready to ship this yet," the problem is real.

Notion reached a similar conclusion from the product side. When they [shipped offline mode](https://www.notion.com/blog/how-we-made-notion-available-offline) — their #1 feature request for years — they restricted offline edits to pages that are fully available locally, because "conflict resolution becomes less predictable" with missing data. They cap auto-download at 50 pages per database view. Even Notion treats offline writes as a scoping problem, not just a queue-and-send.

### The IndexedDB floor

There's a structural reason this is so hard on the web. Every browser-based offline-first tool eventually hits the same floor: [IndexedDB](https://developer.mozilla.org/en-US/docs/Web/API/IndexedDB_API). It's fundamentally an object store, not a relational database. OPFS is changing this picture, but adoption is still limited.

Every relational abstraction on top of IndexedDB must emulate relational guarantees over a key-value store, and that emulation comes with tradeoffs. PowerSync stores WASM SQLite binary chunks in IndexedDB. WatermelonDB uses an in-memory LokiJS database that flushes to IndexedDB. Triplit and InstantDB use triples stored in IndexedDB. Each approach trades off differently, but they all share the same bottleneck.

The [Marco team's offline-first landscape post](https://marcoapp.io/blog/offline-first-landscape) documents this painfully well. They're building a data-heavy email client — hundreds of MB, hundreds of thousands of rows. At that scale, every one of these abstractions started to crack. Their conclusion: Replicache worked because it's "just a KV store" — a thin layer on IndexedDB with raw get/set performance, no relational pretensions.

The overhead can be staggering — a [LokiJS issue](https://github.com/techfort/LokiJS/issues/620) reports 3.5MB of entities ballooning to 2,944MB in Chrome's IndexedDB usage.

[Notion's migration to WASM SQLite](https://www.notion.com/blog/how-we-sped-up-notion-in-the-browser-with-wasm-sqlite) showed what's possible on the other side: 20% faster page navigation (up to 33% in some regions), 50% faster initial loads. But OPFS "doesn't come with graceful handling of concurrency" — Notion had to build a SharedWorker architecture where a single "active tab" handles all writes with Web Locks. And early practitioner reports suggest OPFS-based WASM SQLite may have [higher corruption rates](https://news.ycombinator.com/item?id=45066070) than native SQLite.

The point for write durability: your offline outbox is only as reliable as your persistence substrate. If IndexedDB silently bloats or OPFS corrupts under concurrent writes, "durable offline queue" becomes a support nightmare.

## What I tried (and what each one taught me)

### PowerSync

PowerSync is probably the most mature option out there. Multi-database support (Postgres, MongoDB, MySQL), strong mobile SDKs (Flutter, React Native, Swift, Kotlin), and a real team with enterprise customers.

But the developer experience was rough. Sync rules are defined in YAML — powerful for flat access patterns, but painful for hierarchical permissions. "Show me tasks in projects I'm a member of" requires expressing a join relationship in a declarative config language that doesn't support joins. Self-hosting requires Postgres _and_ a separate MongoDB cluster _and_ a lot of configuration. The Marco team (building an offline-first email client) reported 45-second client initialization times and 100ms+ query latency on web.

**The general lesson:** Declarative auth configuration (YAML, JSON rules) works for flat access control — "user X can read table Y." But real applications have hierarchical scoping: users belong to organizations, organizations have projects, projects have tasks. Expressing that in a declarative config language means either flattening your hierarchy (denormalization) or building escape hatches. This isn't a PowerSync-specific problem — Firebase security rules hit the same wall. At a certain complexity, permissions need to be code, not config.

### Zero

Zero, from the Replicache team, is the most exciting of the bunch. Reactive queries that run against a local SQLite cache, [Incremental View Maintenance](https://en.wikipedia.org/wiki/Materialized_view#Incremental_view_maintenance) to update results efficiently, and a demo that is genuinely _fast_. The DX is exceptional.

But Zero's model is different from traditional sync — it's closer to a reactive query cache than a replication framework. It sits between your client and Postgres as an additional service, keeping the active working set in memory with a recommended ~40MB limit for the initial sync to avoid browser memory pressure. Authorization and permissions are your responsibility — and the team is candid about how hard that is. Their own docs note that "syncing a static block of data is easy; syncing just the slice the user wants and has access to is much more difficult." The mental model is different — reactive queries decide what data lives on a device, which means debugging "why is this row missing?" requires understanding the query planner's decisions rather than inspecting an explicit subscription list.

**The general lesson:** There's a real architectural tension between reactive query replication and explicit subscription-based replication. Reactive queries are magical when they work — you write a query, and the system figures out what to sync. But they trade debuggability for convenience. When something goes wrong, you have to reason about cache coherence and invalidation chains. Explicit subscriptions are more work to set up, but the state is always inspectable: "this device asked for X, was authorized for Y, received Z."

### Electric SQL (+ TanStack DB)

Electric SQL is seriously impressive. It consumes the Postgres logical replication stream and turns it into HTTP-streamable "Shape Logs" — append-only logs of database operations scoped to a query shape. It's CDN-native by design and can support millions of concurrent clients with minimal load on the sync service and zero load on Postgres.

But Electric is read-path only. It doesn't handle mutations, conflict resolution, or offline writes. Those stay in your application. You write the outbox, you handle the retries, you build the conflict detection. TanStack DB adds an optimistic mutation layer on the client, but the server-side write path — validation, authorization, conflict resolution — is entirely your problem.

**The general lesson:** Read-path sync and write-path sync are fundamentally different problems, and it's valid to solve them separately. Electric proves you can build an excellent, scalable read-path with a clean architecture. But if you need offline writes, choosing a read-only sync layer means you're committing to building the hard part yourself.

### Replicache

Replicache was the closest to what I needed. Server-authoritative, push/pull over HTTP, speculative local execution with rebase — like `git rebase`, where local mutations replay on top of confirmed server state. The architecture is elegant, proven, and [well-documented](https://doc.replicache.dev/concepts/how-it-works).

But Replicache is a KV store on the client. No relational queries without building your own indexing layer. And mutations are implemented twice — once on the client (speculative) and once on the server (authoritative). The Marco team loved the raw performance, but they had to bolt on [Orama](https://orama.com/) for indexing and search. Replicache itself is now in maintenance mode, with the team focusing on Zero.

**The general lesson:** The "two mutations" problem is a real architectural cost. When you implement each mutation on the client for optimistic UI _and_ on the server for authority, you maintain two codebases that must stay in sync. Divergence is subtle and hard to test. The alternative — single-definition mutations that run on the server, with the client holding optimistic state until confirmation — trades some perceived latency for correctness. Whether that tradeoff is right depends on your use case.

### LiveStore

[LiveStore](https://livestore.dev/), from Johannes Schickling (creator of Prisma) and born out of the [Riffle](https://riffle.systems/) research project at MIT CSAIL, is the tool I came closest to adopting. Architecturally, it's the most similar to what I ended up building: event-sourced, local SQLite as the primary database, Git-inspired push/pull sync. The reactive query layer is excellent — it replaces Redux/MobX-style state management with reactive SQLite, and it's backed by real research (peer-reviewed at [UIST 2023](https://dl.acm.org/doi/10.1145/3586183.3606801)).

But LiveStore is primarily a client-side state management framework that happens to sync, not a sync protocol. In practice, that means several things I needed weren't there yet: authorization and partial sync are [documented as TODO](https://docs.livestore.dev/reference/syncing/). Conflict resolution beyond last-write-wins is [not yet implemented](https://docs.livestore.dev/reference/syncing/). And critically, there's no event log compaction — the log grows without bound, and every client must replay the full history. A compacted version doesn't exist yet. The docs are [upfront about this](https://docs.livestore.dev/evaluation/state-of-the-project/): it "doesn't scale for unbounded amounts of data."

For apps with bounded event histories (a personal todo app, a small team tool), this is fine. For multi-tenant SaaS with years of data, growing teams, and hierarchical permissions, you need the server to control what each client sees, prune old history, and compact the log. That's the gap that pushed me to build my own.

**The general lesson:** Event sourcing gives you a beautiful audit trail and time-travel debugging, but an ever-growing event log creates a scaling ceiling. At some point you need compaction (condensing history into snapshots) and pruning (discarding events clients no longer need). If those aren't built into the sync protocol, you'll build them yourself — or hit the ceiling.

## Patterns that apply to any sync system

Across all of this, a few patterns kept showing up.

### The doorbell pattern

Replicache introduced me to what Paulus Esterhazy calls the ["shoulder tap" pattern](https://gist.github.com/pesterhazy/3e039677f2e314cb77ffe3497ebca07b): WebSockets send wake-up signals only, never data. All actual data flows through HTTP.

Why this matters: HTTP requests are cacheable, retryable, and debuggable with standard tools. You can stick a CDN in front of them. You can replay them. If one fails, you just retry — the server is stateless between requests. WebSocket connections, by contrast, are stateful, fragile (dropped connections, reconnection logic, ordering guarantees), and much harder to debug.

Replicache calls this "poke." The WebSocket is a doorbell — it says "something changed, come pull." The pull itself is a plain HTTP request with a cursor.

### Append-only logs beat reactive invalidation (for debugging)

There are two broad approaches to keeping clients in sync. Reactive invalidation (used by Zero, Triplit, and others) pushes updates through a dependency graph — when data changes, dependent queries are re-evaluated and results are pushed to clients. Append-only commit logs (used by Replicache, and adapted by Syncular) are simpler: every change gets a monotonically increasing sequence number, and clients say "give me everything after position N."

Both work. But when something goes wrong — and something _always_ goes wrong — their debugging stories are very different.

With a commit log, you read the log. You see every change, in order, with metadata. It's `git log` for your data. You can answer "what happened to row X" by scanning entries. With reactive invalidation, you reconstruct the dependency graph and reason about which invalidation triggered which re-evaluation. That's the difference between reading a receipt and reconstructing a conversation.

Linear's sync engine illustrates both the power and the cost of the log-based approach. When they [recovered from their January 2024 incident](https://linear.app/now/linear-incident-on-jan-24th-2024), they replayed their sync action log to restore state. It worked — but it surfaced 4,136 sync packets with "unresolvable conflicts" across workspaces (about 0.44 per workspace). The log made recovery possible and the conflicts visible. With a pure invalidation model, those conflicts might have silently corrupted state instead of being explicitly flagged.

This isn't a novel insight — it's why Kafka, event sourcing, and `git` itself use append-only logs. The pattern is battle-tested at every scale.

### CRDTs vs. server authority: pick your tradeoff

The local-first ecosystem splits into two camps. [CRDTs](https://en.wikipedia.org/wiki/Conflict-free_replicated_data_type) (Automerge, Yjs) guarantee conflict-free merging — every client converges to the same state without coordination. Server-authoritative systems (Replicache, Zero, PowerSync, Syncular) treat the server as the source of truth and handle conflicts explicitly.

CRDTs are the right choice for real-time collaborative editing — documents, whiteboards, shared cursors. They're designed for that. But for structured data (tasks, invoices, user records), CRDTs add complexity without clear benefit. You rarely want "last writer wins" on an invoice line item; you want to detect the conflict and decide.

[Figma](https://www.figma.com/blog/how-figmas-multiplayer-technology-works/) proved that server authority scales. Their multiplayer system serves millions of users with a server-ordered model — no CRDTs, no OT. The server decides ordering, clients apply it. Figma explains the reasoning clearly: CRDTs are designed for decentralized systems and carry "unavoidable performance and memory overhead." When you have a central server, you can remove that overhead and build "faster and leaner."

The costs of CRDTs for structured data are real and quantified. [Cinapse moved away from Automerge](https://www.powersync.com/blog/why-cinapse-moved-away-from-crdts-for-sync) after hitting WebAssembly's 4GB memory limit on long-running film production schedules — CRDT history files grew with every edit. They were spending roughly $1,000/month on Automerge sync servers as a band-aid before migrating to server-authoritative database replication. Per-character change tracking, designed for text editing, was unnecessary overhead for structured scheduling data. After switching, they reported an 89% reduction in support requests.

### The denormalization tax

Every sync framework needs to answer one question: "which data goes to which client?" The way they force you to answer it shapes your entire data architecture.

**Denormalize to fit the sync model.** PowerSync organizes data into ["sync buckets"](https://docs.powersync.com/sync/rules/organize-data-into-buckets) — each bucket is a unit of replication defined by YAML sync rules. To make this work, your Postgres data needs to carry the information that determines bucket membership on each row. If a task is visible because the user is a member of a project, and project membership lives in a join table, you're stuck: either denormalize `project_id` onto every task row and flatten the membership lookup into a YAML rule, or restructure your schema so the bucket boundary aligns with your authorization boundary.

The Marco team hit this directly — PowerSync ["required us to completely denormalize our Postgres tables, as relations are not properly supported across the sync buckets."](https://marcoapp.io/blog/offline-first-landscape) The [Facilitron migration to PowerSync](https://www.powersync.com/blog/how-facilitron-migrated-from-atlas-device-sync-to-powersync) describes the same pattern: client mutations written to SQLite plus an upload queue, with developers implementing `uploadData()` to apply those mutations against their backend API.

Triplit takes it further: your relational data is decomposed into [triples](https://www.triplit.dev/) `(entity_id, attribute, value)`. That's not denormalization — it's a different data model. Joins become graph traversals. Aggregates become client-side reductions. And as the Marco team found, [5MB of JSON can balloon to 1GB of SQLite](https://marcoapp.io/blog/offline-first-landscape) under the triples representation.

**Define shapes over your existing schema.** [Electric](https://electric-sql.com/docs/reference/architecture) avoids denormalization entirely. Shapes are queries against your Postgres tables as-is — `SELECT * FROM tasks WHERE user_id = ?`. Your schema stays clean. But the shape definition IS the authorization layer, and it's limited to what a WHERE clause can express. "Tasks in projects the user is a member of" requires your proxy layer to resolve the membership, compute the project IDs, and inject them into shape parameters. The complexity doesn't disappear — it moves to your API proxy. And shapes only cover reads.

The core tension: denormalization couples your data model to your auth model. When your permission model changes — add a role, add an organization tier, change visibility rules — you're not just updating auth code. You're restructuring sync buckets, YAML rules, or shape definitions. Teams migrating from [Realm to PowerSync](https://www.powersync.com/blog/interview-migrating-from-atlas-device-sync-and-realm-to-powersync) describe schema changes as "walking on eggshells" — runtime errors from small changes, error-prone database file swaps when transitioning between offline and sync modes. That coupling is invisible in getting-started guides and dominant in real projects.

**If your sync engine dictates your database schema, you are paying the denormalization tax.**

## Applying these patterns: how Syncular works

Those patterns aren't theoretical — they're the design constraints I used when building [Syncular](https://syncular.dev). Here's how they translate into a concrete architecture.

### Local SQLite as the primary database

Not a cache. Not a projection. The client reads and writes against local SQLite directly. Sub-millisecond queries, full SQL, works completely offline.

```typescript
const { data: tasks } = useSyncQuery(
  ({ selectFrom }) =>
    selectFrom('tasks')
      .selectAll()
      .where('completed', '=', 0)
      .orderBy('created_at', 'desc')
);
```

That's [Kysely](https://kysely.dev/) building a typed SQL query against the local database. Joins, aggregates, subqueries — whatever SQL supports. No custom query language, no KV store abstractions.

### Explicit subscriptions over reactive replication

Instead of a query planner deciding what data lives on the device, you declare it:

```typescript
<SyncProvider
  sync={sync}
  identity={{ actorId: userId }}
>
```

The server intersects what you asked for with what you're allowed to see. The result is deterministic and inspectable — no guessing about what's on the device.

### Commit log with scoped authorization (no denormalization required)

This is where the denormalization tax disappears. The server keeps its schema normalized and relational. The client can have its own schema — optimized for local queries, potentially flattened or pre-computed. Table handlers connect the two and extract scopes from the data at sync time using code, not config.

Every change flows through an append-only commit log. Every entry carries scope values computed from the row data:

```
Commit Log
─────────────────────────────────────────────────────────────────
  seq 1  │  device-A  │  user:u1, project:p1  │  insert task
  seq 2  │  device-B  │  user:u2, project:p1  │  insert task
  seq 3  │  device-A  │  user:u1, project:p1  │  update task → done
─────────────────────────────────────────────────────────────────
         ▲ Device B cursor: 2 → needs seq 3
```

Authorization isn't a middleware layer or a YAML config. It's embedded in the log. Every change is tagged, filterable, and auditable. When a client pulls, it says "give me commits after my cursor, for my scopes." The server computes the intersection. The protocol is designed so that filtering happens before data leaves the server — scope mismatches are caught at the sync layer, not in application code.

```typescript
const tasksHandler = createServerHandler({
  table: 'tasks',
  scopes: ['user:{user_id}', 'project:{project_id}'],
  resolveScopes: async (ctx) => {
    const memberships = await ctx.db
      .selectFrom('project_members')
      .select('project_id')
      .where('user_id', '=', ctx.actorId)
      .execute();
    return {
      user_id: [ctx.actorId],
      project_id: memberships.map((m) => m.project_id),
    };
  },
});
```

That's code, not config. Hierarchical permissions, dynamic lookups, any logic you need. When your permission model changes — say you add "organization admin can see all projects" — you update `resolveScopes` to include `org_id`. Your Postgres schema doesn't change. Your client schema doesn't change. The commit log format doesn't change.

### The doorbell + durable outbox

Writes go to local SQLite and a durable outbox in the same transaction. The outbox survives crashes. When connectivity returns, pushes are idempotent — keyed by `clientId + clientCommitId`, safe to retry indefinitely. Other clients get a WebSocket wake-up and pull over HTTP.

All data flows through HTTP. The WebSocket is just a doorbell.

### Compaction and pruning

An append-only commit log is great for debugging and auditability — until it grows to millions of entries. Syncular handles this with two mechanisms: **pruning** removes commits that all active clients have already consumed (no client needs them anymore). **Compaction** condenses sequences of operations on the same row into a single snapshot entry, preserving the final state while discarding intermediate history. New clients bootstrap from a compact snapshot rather than replaying years of history.

This is the kind of infrastructure you don't think about until your commit log hits a few GB and client bootstrap takes minutes instead of seconds. Building it into the protocol from the start was a direct lesson from evaluating tools that defer it.

### Conflicts and revocation: closing the loop

I opened this post with two questions: what happens when two devices edit the same row offline, and what happens when a user is removed from a project while offline? Here's how Syncular handles both.

**Conflicts.** Every push can carry a `baseVersion` — the server version the client last saw for that row. When the server receives the push, it compares `baseVersion` against the current version. If they match, the write applies cleanly. If they don't, the server returns a conflict response with the current server row:

```typescript
// Device A pushes with the version it last saw
await m.tasks.update('task-1', { title: 'Buy groceries' }, { baseVersion: 5 });

// Server is now at version 7 (Device B wrote in between)
// → conflict response: { status: 'conflict', serverVersion: 7, serverRow: { ... } }
```

The conflict is surfaced to the application — not silently merged, not auto-resolved with last-write-wins. You decide: accept theirs, keep mine, merge specific fields. The conflict and its resolution are recorded in the commit log, so you can always audit what happened.

**Revocation.** When a user is removed from a project, the next pull reflects the scope change. `resolveScopes` now returns a set that no longer includes that project — so the server stops sending commits for it. On the client, data outside the current scopes is pruned. Any pending outbox writes that target scopes the user no longer has are rejected by the server on push, with a clear error the client can handle (surface to the user, discard, or queue for review).

This doesn't make revocation trivial — it's inherently hard in any offline system. But the scope model makes it explicit and auditable rather than implicit and surprising.

## The landscape in early 2026

If you're evaluating sync tools right now, here's where I see things:

There are 14+ solutions and no clear winner — the problem just looks different depending on what you're building.

**Two architectural camps.** CRDTs (Automerge, Yjs) for collaborative document editing. Server-authoritative systems (Zero, Electric, PowerSync, Syncular, Replicache) for structured data with permissions.

**SQLite-on-client is where things are headed.** LiveStore, Notion's WASM SQLite migration, and the broader libSQL/Turso ecosystem all point the same direction: give the client a real relational database, not a KV store with a relational wrapper.

**Vendor lock-in is a real, demonstrated risk.** MongoDB [announced the deprecation of Atlas Device Sync](https://www.mongodb.com/docs/atlas/app-services/sync/device-sync-deprecation/) (formerly Realm Sync) in September 2024, with final shutdown on September 30, 2025. Teams with production apps had a year to migrate — with [no clear first-party alternative](https://github.com/realm/realm-swift/discussions/8680) and a lot of frustration. If your sync layer is a hosted service with proprietary internals, you're one corporate decision away from a rewrite.

**What developers actually want** (based on the [Zero launch HN thread](https://news.ycombinator.com/item?id=42453431) and similar discussions): speed, bring-your-own-database, partial sync with permissions, custom server-side logic, SQL familiarity, open source, and low barrier to entry. No single tool delivers all of these today. [Ink & Switch's seven ideals](https://www.inkandswitch.com/local-first/) — fast, multi-device, offline, collaboration, longevity, privacy, user control — remain the north star, but the ecosystem is still early.

**The decentralized camp is solving a different problem.** Projects like [Jazz](https://jazz.tools/), [Evolu](https://www.evolu.dev/), and [Anytype/AnySync](https://anytype.io/) optimize for data sovereignty — no central server, end-to-end encryption, cryptographic identity and permissions. If your primary requirement is that no authority controls user data, these projects are doing important work. The tradeoff is that without a central server, authorization must be enforced via cryptography rather than server-side validation, conflict resolution must be automatic (typically CRDTs), and revocation is fundamentally harder — you can't un-share a key a client already has.

Syncular assumes a server exists and makes it the authority. Different problem, different architecture.

Martin Kleppmann [makes a compelling argument](https://martin.kleppmann.com/2024/05/30/local-first-conference.html) that the _developer_ benefits of local-first may matter as much as the user benefits: no dedicated backend team, no on-call for sync infrastructure, no network error handling in every component. That's the promise. We're not there yet.

## What others optimize for (and why that's valid)

Different tools optimize for different things. The right choice depends on your priorities.

**Zero** optimizes for developer experience and reactive queries. If you want the lowest-friction path to a fast, reactive UI with local-first semantics, Zero is probably the best option right now.

**Electric SQL** optimizes for read-path scale. If you have a stable write API and need to stream Postgres changes to millions of clients through a CDN, Electric is hard to beat.

**PowerSync** optimizes for platform coverage. If you need Flutter and MongoDB and professional support, PowerSync is the only serious option.

**Syncular** optimizes for write-path correctness, debuggability, and explicit control. It runs on any database [Kysely](https://kysely.dev/) supports — Postgres, SQLite, Neon, LibSQL, and more. You define table handlers and scope resolution, and you can hook into existing tables with typegen. The commit log is always readable, the scopes are always auditable, and the outbox is always durable.

## What Syncular doesn't do

- **Not a CRDT engine.** If you're building Google Docs-style real-time collaborative text editing, use Yjs or Automerge. Syncular is for structured data sync, not document collaboration.
- **Alpha.** APIs, wire formats, and storage layouts will change. Don't use in production unless you're comfortable pinning versions and migrating manually.
- **Self-hosted only.** No managed service. That's a feature for some teams and a cost for others.

## Who this is for

If you've been burned by opaque sync layers — spent hours wondering why a row is missing on one device, or duct-taped an outbox onto a read-only framework — Syncular is the tool I wished existed. Full SQL on the client, authorization baked into the commit log, TypeScript end-to-end, offline writes that survive app crashes and three days without signal.

If you want the lowest-friction reactive UI, try [Zero](https://zero.rocicorp.dev). Read-path streaming at CDN scale, try [Electric](https://electric-sql.com). Flutter and MongoDB, try [PowerSync](https://powersync.com). But if you've tried those and kept writing glue code for the parts they don't cover — [that's what Syncular is](https://syncular.dev).
